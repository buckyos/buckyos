# BuckyOS Reset and Unbind

## 1. 文档目标

本文用于明确 BuckyOS 中与“重置（Reset）”“重激活（Reactivate）”“重绑定（Rebind）”相关的产品语义、触发入口与系统行为边界。

这里讨论的不是单一安装/卸载动作，而是设备身份、用户数据、系统状态与域名/Zone 绑定关系发生变化时，产品应如何定义与处理。

本文重点覆盖两类场景：

1. **设备侧重置 / 重激活**：一台已经激活过的设备，如何回退到“可再次激活”的状态。
2. **账号/域名侧强制改绑**：用户通过自己的账号、合约或 DID/DNS 配置，将域名/默认 Zone 从旧设备或旧 Zone 转移到新设备或新 Zone。

---

## 2. 核心结论

### 2.1 BuckyOS 只定义两种正式重置模式

BuckyOS 设备重置分为两类：

- **Mode1：完全重置（Full Reset）**
  - 删除系统数据。
  - 删除用户数据。
  - 删除设备原有激活状态与绑定状态。
  - 结果等价于“恢复到未激活、未使用、可重新交付”的状态。

- **Mode2：保留用户数据的系统重置（System Reset with User Data Preserved）**
  - 保留用户的 Home/Data 区数据。
  - 删除系统数据、设备身份相关状态、绑定相关状态、系统配置缓存等。
  - 设备随后重新进入激活流程。
  - 本质上是“保留用户数据的重装 + 重激活”。

### 2.2 “仅替换 NodeIdentity 文件”不是支持路径

从技术上看，设备重绑定常常表现为更换 `NodeIdentity`。但从产品和系统一致性的角度，**不支持通过简单替换 NodeIdentity 文件来完成重绑定**。

原因是：

- 系统中通常存在大量与当前设备身份、当前 Zone、当前 OOD 角色绑定的配置与缓存。
- 仅替换身份文件，极易造成系统状态不一致、服务配置残留、集群连接异常、数据索引错误等问题。
- 尤其对 OOD 来说，旧身份对应的系统状态在更换绑定后通常已经不再有效。

因此，BuckyOS 的正式产品逻辑是：

- **需要连同用户数据一起清空时，走 Mode1。**
- **需要保留用户数据但重建系统绑定关系时，走 Mode2。**

### 2.3 Mode1 必须要求物理控制权限

Mode1 会造成不可逆的数据销毁，因此它的触发必须以**设备物理控制权**为前提。

默认原则：

- **不提供一个普通系统面板入口，让用户通过“点一下 + 输密码”远程触发 Mode1。**
- 对硬件产品，Mode1 更适合通过**硬件按键组合、恢复模式、物理控制台**等方式触发。
- 对桌面版/安装包版，Mode1 更适合放在**安装器/重装器**中，以显式、高风险提示的方式提供。

## 3. 术语与范围

本文沿用 BuckyOS 现有术语：

- **OOD**：BuckyOS 中承担核心在线职责的设备角色。
- **Zone**：设备集合与信任/路由边界。
- **NodeIdentity**：节点身份文件。
- **DID Document / DNS 绑定信息**：用户域名、默认 Zone、默认 OOD 等对外可解析的权威绑定信息。
- **Headless**：某台旧设备或旧 Zone 仍在运行，但已经不再是域名/默认入口指向的当前权威对象，因而“还活着，但外部请求不再到达”。

本文不深入规定底层实现细节，而是定义：

- 产品对外语义。
- 允许与不允许的重置/重绑方式。
- 启动时的检测与处置原则。
- 安装器、硬件入口、账号侧改绑入口需要提供的用户体验与安全约束。

---

## 4. 设备侧重置 / 重激活

### 4.1 场景一：二手出售 / 设备转交

这是最典型、最明确的设备重置需求。

典型诉求：

- 用户拥有一台已经激活过的 BuckyOS 硬件设备。
- 用户打算出售、转交或回收该设备。
- 用户希望该设备回到“像新设备一样、未激活、无历史数据”的状态。

### 产品定义

该场景必须使用 **Mode1**。

### 结果要求

执行后：

- 设备不再保留原用户身份。
- 设备不再保留原绑定关系。
- 设备不再保留用户数据。
- 设备重新进入首启/未激活状态。

这可以视为 BuckyOS 的“恢复出厂设置”语义。

---

### 4.2 场景二：绑定错误 / 需要改绑到新的 Zone 或新身份

另一类常见需求是：

- 设备已经被激活，但激活时绑定错了对象。
- 或者用户后续希望把设备改绑到另一个 Zone / 另一套身份体系。
- 用户希望尽量保留自己的用户数据。

### 产品定义

该场景默认应使用 **Mode2**。

### 原因

对这类场景而言：

- 用户数据本身仍有价值，应尽量保留。
- 但系统数据、设备绑定、身份关联配置往往已经失效，不能直接沿用。
- 单纯改 `NodeIdentity` 风险很高，不应作为产品支持路径。

### 结果要求

执行后：

- 用户的 Home/Data 分区保留。
- 系统分区、系统配置、身份配置、Zone/OOD 绑定相关状态被清理。
- 设备重新进入激活流程。
- 新一轮激活完成后，由新绑定关系重建系统状态。

可以将 Mode2 理解为：

> “保留用户数据，但把系统当作重新安装一次。”

---

## 5. 两种重置模式的边界

| 项目 | Mode1：完全重置 | Mode2：保留用户数据的系统重置 |
|---|---|---|
| 用户数据 | 删除 | 保留 |
| 系统数据 | 删除 | 删除 |
| 设备激活状态 | 清除 | 清除 |
| Node / Zone / OOD 绑定状态 | 清除 | 清除 |
| 适用场景 | 二手出售、设备转交、彻底恢复出厂 | 绑定错误、改绑、保留数据的重装 |
| 安全要求 | 必须强调物理控制权 | 仍建议要求本地/物理控制 |
| 后续动作 | 重新激活 | 重新激活 |

### 5.1 系统数据的建议范围

Mode2 中应清理的“系统数据”至少包括：

- NodeIdentity 及其派生配置。
- 与当前 Zone/OOD 绑定相关的本地状态。
- 与当前设备身份强绑定的服务配置。
- 集群成员关系缓存、路由缓存、控制面缓存。
- 依赖旧身份生成的本地凭据、会话、索引与派生元数据。

### 5.2 用户数据的建议范围

Mode2 中保留的“用户数据”原则上包括：

- 用户 Home 目录。
- 明确位于 Data 分区中的个人数据。
- 与用户内容直接相关、且不依赖旧系统身份才能解释的数据。

是否保留某一类数据，原则上以“**它是否依赖旧系统身份/旧绑定关系**”为判断标准。

## 6. 触发入口设计

### 6.1 硬件产品

对于硬件设备，特别是涉及 Mode1 的情况，重置必须依赖**物理控制台权限**。

产品原则：

- 不应把高破坏性的重置放进日常系统设置页面中，作为普通功能暴露。
- 更适合通过：
  - 硬件组合键；
  - 开机恢复菜单；
  - 本地控制台恢复工具；
  - 专用恢复介质。

这样做的核心目的是降低误触发、被盗号远程抹机、或单纯因为口令泄露而造成不可逆损失的风险。

### 6.2 Desktop / 安装包版

桌面版更适合把重置能力放在**安装器/重装器**中，而不是运行时系统面板中。

建议安装器提供以下三种选项：

1. **覆盖安装（默认）**
   - 尽量保留现有系统与数据。
   - 不等价于重置。

2. **重装并清空所有数据（Mode1）**
   - 删除用户数据。
   - 删除系统数据。
   - 设备回到未激活状态。

3. **重装并保留用户数据（Mode2）**
   - 保留用户数据。
   - 删除系统数据与绑定状态。
   - 安装后重新进入激活流程。

从实现思路上，这两种重置都可以理解为：

> “先按相应规则卸载/清理，再重新安装，然后重新激活。”

## 7. 高级安全重置：可信计算、全盘加密与远程擦除

BuckyOS 曾考虑过一类更高级、但短期内不建议面向 C 端主推的能力：

- 可信计算区 / TPM / 安全硬件。
- 基于可信计算区的全盘加密。
- 远程擦除数据。
- 依赖外置密钥设备（如安全 UKey/智能卡）才能完成引导。

这些能力确实可以衍生出更强的重置与防护模型，但目前应作为**未来企业版/小企业版的增强能力**，而不应作为当前 C 端产品默认能力。

### 7.1 远程擦除

在手机场景中，远程擦除很常见，因为设备容易遗失，而且通常有成熟的账号体系与多设备授权模型。

但对家庭场景下的个人服务器或家中固定设备来说：

- “找回设备 / 远程抹除”并不是高频刚需。
- 误触发的代价非常大。
- 与用户对设备“长期在家中运行”的预期不完全一致。

因此，**短期不建议将远程擦除作为核心产品能力**。

### 7.2 外置启动密钥 + PIN 的企业级方案

另一类更强的安全模型是：

- 设备磁盘被全盘加密。
- 引导时必须插入额外的硬件密钥。
- 该密钥还需 PIN 码解锁。
- 只有启动阶段获得正确引导密钥，系统才能读取盘上主密钥并继续启动。

它的好处是：

- 设备一旦掉电，再次启动必须通过人工授权。
- 即便设备被物理搬走，攻击者也无法在不知道外置密钥与 PIN 的情况下读取数据。
- 非常适合对重启、搬迁、审计有严格要求的企业环境。

但它的代价也很明显：

- 日常使用门槛高。
- 故障恢复复杂。
- 一旦用户管理不好密钥，极易演化为“安全性很好，但用户把自己的数据彻底弄丢”。

因此，结论是：

- **短期不做 C 端默认能力。**
- **未来可作为面向小企业/企业场景的高级安全选项。**

## 8. 账号/域名侧强制改绑

除了单设备视角的“重激活”，BuckyOS 还存在另一类“重绑定”：

> 用户通过自己掌握的账号、合约、DNS 私钥或 DID Document 控制权，把原本指向 Zone A / OOD A 的权威绑定信息，改为指向 Zone B / OOD B。

这类操作的关键点在于：

- 域名/合约资源本质上属于用户，而不是属于某台设备。
- 用户只要掌握自己的账号与私钥，就有权修改绑定关系。
- **这个改绑动作不需要旧 Zone / 旧 OOD 的同意。**

这是一种“用户发起的强制改绑（Force Unbind ）”。

> Rebind = Unbind + Active

### 8.1 典型场景

### 场景 A：用户把默认 Zone 从旧 Zone 改到新 Zone

例如：

- 用户原来有 Zone A。
- 现在用户把自己域名/DID 中记录的默认 Zone 改成 Zone B。
- 那么 Zone A 就失去对外权威入口。

### 场景 B：用户仍保留原 Zone，但更换默认 OOD 或其公私钥 / DID Document

例如：

- 用户仍使用同一 Zone。
- 但默认 OOD 已被替换，或其 DID Document 中的权威身份已变化。
- 旧 OOD 不再是当前解析结果对应的设备。

---

## 9. Headless 状态

当用户完成账号/域名侧的强制改绑后，旧设备或旧 Zone 可能进入一种“**Headless**”状态。

#### 9.1 Headless 的定义

Headless 指：

- 旧设备仍然在运行。
- 本地数据通常还在。
- 但其域名入口、默认路由或权威解析结果已经不再指向它。
- 因此它可能“还活着”，但外部请求已经不会再到达。

可以把它类比为：

> 一个网站进程还在跑，但 DNS 已经改走了，所以网站并不是“宕机”，只是“对外入口不再指向它”。

#### 9.2 这类改绑不应等价于数据删除

用户在账号层面改绑域名/Zone 时，系统不应把它解释成“需要自动清空旧设备数据”。

这类操作的正确语义应当是：

- 旧设备失去当前权威入口。
- 旧设备可能无法继续承担原角色。
- **但数据不会因此自动消失。**

---

## 10. OOD 启动时的绑定检查与处理

OOD 在启动时，应主动检查当前权威绑定信息（例如 DNS / DID Document / Zone 配置），并与本地预期进行比对。

### 10.1 可恢复变化

如果 OOD 发现：

- 自己仍然在当前 Zone 中；
- 只是 Zone 成员数量、OOD 列表、集群拓扑发生变化；
- 或某些连接范围、共识范围、同步范围需要调整；

则这属于**可恢复变化**。

### 处理原则

- 系统应尝试完成拓扑重计算与配置收敛。
- 允许更新需要连接的新成员或缩减旧成员。
- 允许重新建立服务依赖关系。
- 启动流程可以继续，但要有明确的成员变化日志与审计信息。

### 10.2 不可恢复变化

如果 OOD 发现：

- 当前权威解析结果中，自己已经不再属于目标 Zone；
- 或自己已不再是当前默认 OOD / 当前权威设备；
- 且本地无法通过普通启动修复这一事实；

则这属于**不可恢复变化**。

这是 Headless 的典型触发条件。

---

## 11. Headless 的推荐产品行为

对于“启动时发现自己已不在当前权威绑定中”的情况，本文建议 BuckyOS 采用以下默认策略：

#### 11.1 默认进入 Headless 模式，而不是直接拒绝启动

推荐默认行为：

- 系统**继续启动**。
- 但进入 **Headless Mode**。
- 明确停止把自己当作当前权威入口对外宣告。
- 保留本地数据。
- 向本地用户显式告警。

#### 11.2 原因

这样设计有几个好处：

- 与用户直觉更一致：改绑是“入口切走了”，不是“旧机器必须立刻死掉”。
- 有利于数据迁移、排查、导出、人工善后。
- 更接近已有互联网产品经验：DNS 改绑后旧服务进程通常仍可继续运行，只是流量不再到达。

#### 11.3 Headless 模式下的限制

Headless 模式下，系统应至少满足以下行为要求：

1. **不尝试自动夺回权威绑定。**
2. **不把自己继续当作当前默认 OOD 对外发布。**
3. **保留本地数据访问能力。**
4. **允许用户执行导出、备份、Mode1、Mode2 或重新激活。**
5. **对依赖“当前权威身份”的服务进行降级或停用。**
6. **向本地控制台/安装器/管理页面输出明确告警。**

建议提示文案：

> 当前设备已不再是该域名/Zone 的权威绑定对象。设备已进入 Headless 模式。数据未被删除，但该设备可能无法继续作为当前入口提供服务。你可以选择导出数据、重新激活、执行系统重置或完全重置。

#### 11.4 企业场景的可选更严格策略

在未来企业版中，可以考虑增加策略开关：

- 默认继续启动（C 端默认）。
- 发现 Headless 后停止关键服务。
- 发现 Headless 后拒绝完成完整启动，等待人工授权。

但这不应作为当前通用产品默认行为。

## 12. BuckyOS App侧的用户确认与风险提示

无论是 Mode1/Mode2，还是账号/域名侧 Force Rebind，入口设计都必须让用户非常明确地知道自己在做什么。

### 12.1 Mode1 的提示要求

至少要明确提示：

- 将删除所有用户数据。
- 设备将恢复到未激活状态。
- 此操作不可撤销。
- 建议仅在二手出售、设备转交、彻底恢复时使用。

### 12.2 Mode2 的提示要求

至少要明确提示：

- 将保留用户数据。
- 但系统配置、绑定状态、设备身份状态会被清空。
- 操作完成后需要重新激活。
- 某些依赖旧系统身份的缓存/应用状态将失效。

### 12.3 Force Rebind 的提示要求

用户通过 BMS/账号/合约入口修改域名、默认 Zone、默认 OOD、DID Document 时，应明确提示：

- 此操作会让旧 Zone / 旧 OOD 失去当前权威入口。
- 旧设备上的数据不会自动删除。
- 旧设备可能进入 Headless 状态。
- 改绑完成后，旧设备可能无法再通过原域名访问。

建议提示文案：

> 你正在将当前域名/默认入口从旧设备或旧 Zone 切换到新的绑定对象。切换完成后，旧设备上的数据不会自动删除，但旧设备可能进入 Headless 状态，并不再接收来自当前域名的请求。

## 13. 推荐的统一状态机

#### 13.1 设备侧状态

```text
未激活
  -> 激活中
  -> 已激活
      -> Mode2 -> 未激活（保留用户数据）
      -> Mode1 -> 未激活（清空所有数据）
```

#### 13.2 域名/Zone 侧状态

```text
域名 -> Zone A / OOD A
  -> 用户执行 Force Unbind
  -> 域名 -> Null
  -> Zone A / OOD A 进入 Headless 语义（仍可能继续运行）
```

#### 13.3 OOD 启动检查结果

```text
启动
  -> 查询当前权威绑定
      -> 仍在当前 Zone 中：正常启动 / 重算拓扑
      -> 已不在当前 Zone 中：进入 Headless 模式
```

---

## 14. 不做什么

当前阶段，BuckyOS 不将以下能力作为默认产品能力：

1. **运行时通过普通系统面板远程触发 Mode1。**
2. **把“替换 NodeIdentity 文件”当作正式支持的重绑定流程。**
3. **面向 C 端默认启用远程擦除。**
4. **面向 C 端默认启用高门槛的外置启动密钥 + PIN 全盘加密方案。**
5. **在账号侧 Force Unbind 后自动抹除旧设备数据。**

---

## 15. 最终产品决策摘要

### 必须做

- 支持 **Mode1** 与 **Mode2** 两种正式重置模式。
- 明确规定重置后设备重新进入激活流程。
- 禁止把“仅替换 NodeIdentity”作为正式产品路径。
- 为硬件产品提供基于物理控制权的重置入口。
- 为 Desktop/安装包版提供安装器中的重装/重置选项。
- OOD 启动时必须检查当前权威绑定信息。
- 当发现自己已不再属于当前权威绑定时，默认进入 **Headless 模式**。
- 在BuckyOS App做 Force Unbind 时，必须向用户提示其对旧设备的影响。

### 当前不做

- 不做面向 C 端的远程擦除。
- 不做面向 C 端默认开启的高强度全盘加密引导体系。
- 不做运行时普通面板中的一键高破坏性重置。

### 后续可扩展

- 企业版 Headless 策略开关。
- 企业版可信计算 / 审计增强。
- 企业版外置启动密钥、冷启动授权、设备搬迁审计流程。

## 16. 一句话总结

BuckyOS 的“重置”本质上是**让设备重新回到可激活状态**；“重绑定”本质上是**改变设备或域名当前所指向的权威身份与入口**。前者在产品上收敛为 **Mode1 / Mode2** 两种设备重置，后者在产品上体现为**用户可通过账号/合约进行 Force Unbind，而旧设备默认进入 Headless 语义而非被自动抹除**。
