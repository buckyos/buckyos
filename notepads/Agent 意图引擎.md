# Agent 意图引擎：一种基于 LLM 幻觉的 Tool 调用新范式

## 问题背景

大语言模型（LLM）在 Agent 场景中如何高效地调用工具（Tool），是当前工程实践中的核心痛点。现有方案都在"如何让 LLM 知道有哪些工具可用"这个问题上做文章，但每种方案都有明显的代价。

## 现有四种流派及其局限

### 1. 全塞流（Schema Stuffing）

把所有 tool 的 JSON Schema 一次性塞进 system prompt。

简单直接，但 tool 数量一多，prompt 急剧膨胀，LLM 的 attention 被大量工具定义稀释，真正的任务上下文反而被挤压。对于只有三五个工具的简单场景够用，但无法扩展到几十甚至上百个工具的真实系统。

### 2. 选择流（Select-then-Call）

每次调用至少走两轮 LLM：第一轮只载入一个 `list_tools` 函数，LLM 从中选择需要的 tool；第二轮把选中 tool 的 schema 注入 prompt，LLM 再发起真正的调用。

解决了 prompt 膨胀问题，但每次 tool call 的延迟翻倍，且选择本身也可能出错——LLM 需要仅凭工具名称和简短描述来做判断，信息量有限。

### 3. 极简命令流（Minimal Primitives）

只提供 `bash`、`read_file`、`write_file`、`edit_file` 四个原语。将文件系统视为一个巨大的状态机，LLM 凭借训练中积累的 bash 知识自行组合命令完成任务。

这是 Claude Code 等产品验证过的路线，实际效果很好。它的成功揭示了一个关键事实：LLM 在 bash 命令上的"内隐知识"极其丰富，`exec bash` 这个单一入口背后激活的能力远超显式注入几十个 tool schema。从注意力效率的角度看，这种模式更优。

### 4. 万物皆 Python（Interpreter 流）

告诉 LLM 不要调用函数，直接写 Python 脚本来完成一切任务。

LLM 的代码能力确实强大，但代码生成不是一步完成的——它需要 plan → code → compile → test → debug 的完整循环。对于"查天气"这样的简单意图，让 LLM 把 attention 花在构造 Python 代码上是一种浪费。我们更希望 LLM 专注于理解任务本身，而不是被实现细节拖住。

## 核心洞察：幻觉即意图

四种流派的共同假设是：**LLM 必须事先知道工具的存在，才能调用它。**

但观察 LLM 的实际行为，会发现一个有趣的现象——它经常会"调用"一个根本不存在的函数，或者在 bash 中运行一个当前环境里没有的命令。过去，我们一律把这当作**错误**来处理：系统抛出异常，返回 `command not found`，LLM 收到报错后尝试换一种方式。

但换个角度看：**LLM 调用一个不存在的函数，本质上是在表达一个意图。**

它不需要知道 `get_weather` 这个命令是否真的存在——它只是根据上下文判断"现在需要获取天气信息"，然后用最自然的方式表达了这个意图。这个"幻觉"不是 bug，而是一个**高质量的意图信号**。

## 意图引擎：第五种流派

### 设计思路

基于上述洞察，我们可以构建一种全新的 tool 调用架构——**意图引擎（Intent Engine）**。

核心理念：**不给 LLM 注入任何 tool schema，让它自由表达意图；系统负责捕获意图并按需实现。**

### 运行机制

整体流程分为两个并行的循环：

**主循环（LLM Session）**

```
用户输入 → LLM 推理 → 在 bash 中表达意图（如 `get_weather cupertino`）
  → 命令存在？→ 执行，返回结果，继续对话
  → 命令不存在？→ 捕获意图，session 进入等待 → 意图实现后恢复执行
```

**意图实现循环（Intent Realization Agent）**

```
监听意图队列 → 读取意图上下文（命令名 + 参数 + 对话上下文）
  → 查找系统中已有的工具和 API
  → 生成一个薄 wrapper 脚本（通常只是参数转发）
  → 部署到 session 的 bin 目录
  → 通知主循环恢复
```

### 关键设计：Bash 作为统一意图层

LLM 表达意图的方式不是 JSON function call，而是**直接在 bash 中执行命令**。这是最自然的意图表达形式——命令名就是意图，参数就是上下文。

当意图引擎 Agent 收到一个未实现的意图时，它看到的输入是：

> **LLM 希望完成 `get_weather cupertino` 的调用**
> 
> 上下文：用户询问 Cupertino 今天的天气，LLM 决定获取天气数据

意图引擎 Agent 的工作极其聚焦：

1. **检索系统已有工具**——比如系统中已经存在一个完整的天气 API SDK
2. **生成薄 wrapper 脚本**——一个 Python 脚本，接收 bash 参数，转发给已有工具，格式化输出
3. **部署到 session 的 `bin/` 目录**——使其成为一个可执行命令

```python
#!/usr/bin/env python3
# ~/.session/bin/get_weather
# 自动生成的意图实现 - 对 weather_api 的参数转发
import sys
from tools.weather_api import fetch_weather

location = sys.argv[1] if len(sys.argv) > 1 else "unknown"
result = fetch_weather(location=location, unit="celsius")
print(f"🌤 {result['location']}: {result['temp']}°C, {result['condition']}")
```

下次 LLM 再调用 `get_weather cupertino`，bash 直接找到这个脚本并执行——零延迟，零额外推理。

### 生命周期与自然衰减

意图实现的生命周期与 session 绑定：

- **创建**：LLM 首次"幻觉"出一个命令 → 意图引擎实现它 → 写入 session 的 `bin/` 目录
- **复用**：同一 session 内后续调用直接命中，无需重新实现
- **衰减**：session 结束或历史上下文滚动后，`bin/` 目录中的命令自然消失
- **重建**：新 session 中 LLM 再次产生相同意图 → 意图引擎重新实现

这不是缓存失效——这是**按需实例化**的自然节奏。每次重建还可能产生更好的实现，因为系统中可用的工具可能已经更新。

## 与现有流派的对比

| 维度 | 全塞流 | 选择流 | 极简命令流 | Interpreter 流 | 意图引擎 |
|------|-------|-------|-----------|---------------|---------|
| Prompt 开销 | 高（所有 schema） | 中（两轮调用） | 低（4个原语） | 低 | **零**（无 schema 注入） |
| LLM Attention 效率 | 低 | 中 | 高 | 中（写代码占用） | **最高**（专注任务） |
| 工具扩展性 | 差 | 中 | 依赖 bash 生态 | 好（Python 生态） | **无限**（按需生成） |
| 首次调用延迟 | 低 | 高（两轮） | 低 | 高（写+跑代码） | 高（等待意图实现） |
| 重复调用延迟 | 低 | 高 | 低 | 高 | **低**（命中 bin 缓存） |
| 实现复杂度 | 低 | 中 | 低 | 中 | 高（需要意图引擎） |

## 核心优势

**零注入成本**：主 LLM 的 context window 里不需要塞任何 tool schema，全部 attention 集中在用户任务上。

**完全利用 LLM 内隐知识**：LLM 在训练中见过海量的命令行工具用法，它"幻觉"出的命令名和参数结构往往是合理的、符合直觉的。我们不是在对抗幻觉，而是在**利用**幻觉。

**无限工具空间**：不存在"注册多少工具"的问题。LLM 可以表达任何意图，系统按需实现。工具空间的边界等于意图引擎的实现能力。

**自然的 PATH 语义**：session 级别的 `bin/` 目录天然具备作用域隔离。不同 session 可以有不同的工具集，工具的生命周期与 session 一致，无需额外的管理机制。

## 待解决的问题

**意图解析的鲁棒性**：同一个意图可能被表达为 `get_weather cupertino`、`weather cupertino`、`check_forecast --city cupertino` 等不同形态。意图引擎需要一定的归一化能力，或者接受每次生成略有不同的 wrapper。

**首次调用的时间窗口**：意图实现需要时间（coding agent 生成 + 测试脚本），session 的等待时长有上限。需要在用户体验和实现质量之间找到平衡。

**安全边界**：LLM 驱动的 tool 自动生成意味着近乎无限的能力边界，需要 sandbox 机制确保生成的脚本不会执行危险操作。

**意图去重与缓存策略**：跨 session 的意图实现是否值得持久化？如何判断两个不同表述的意图在语义上等价？

## 总结

意图引擎的哲学是：**不要试图告诉 LLM 它能做什么，让它告诉你它想做什么。**

过去我们把 LLM 的幻觉视为需要修正的错误。意图引擎将其重新定义为系统的输入信号——一个高质量的、基于上下文推理得出的意图声明。围绕这个信号构建实现层，就得到了一个无需 tool schema、无需选择流程、理论上可以调用无限工具的 Agent 架构。

这可能是 tool calling 的下一个演化方向：从"给 LLM 一个工具箱"，到"让 LLM 自己描述它需要什么工具，系统现场造出来"。
