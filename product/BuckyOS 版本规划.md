# BuckyOS 版本规划

> 面向市场、销售、商务合作伙伴的版本说明文档。
> 用于回答："BuckyOS 现在能买到什么？""Desktop / 标准版 / 企业版到底差在哪？""我应该向哪类用户/客户推荐哪一个？"

---

## 1. 一句话定位

| 版本 | 定位 | 形态 | 价格/商业模式 | 当前状态 |
| --- | --- | --- | --- | --- |
| **BuckyOS Desktop** | 普通用户尝试 BuckyOS 的低成本通道 | Mac / Windows 桌面应用 | 免费 | **已发布**，是现阶段唯一对外可获取的版本 |
| **BuckyOS**（标准版） | 面向家庭与个人用户的开箱即用方案 | 由硬件厂商预装出厂的设备 | 含在硬件价格中 | 硬件研发中，未发布 |
| **BuckyOS Enterprise**（企业版） | 面向组织的私有部署方案，服务多用户 Zone | 由系统集成商部署，可上客户现有服务器或新服务器 | 通过认证体系（系统集成 / 系统支持 / 硬件）变现 | 规划中 |

**命名约定：** 不带任何后缀的"BuckyOS"特指**标准版**；"Desktop"和"Enterprise"是在标准版基础上**改变了交付方式与使用约束**的发行版本，不是另一个产品。

**重要前提：** BuckyOS 走**开源兼容**的路线。任何硬件厂商都可以在自己的硬件上预装 BuckyOS 出厂，也允许任何系统集成商在客户现有服务器上部署 BuckyOS。BuckyOS 不绑定任何一家硬件厂商。

## 2. 首要区别：单点 vs 集群

这是 Desktop 与其他两个版本之间最根本、也最不可调和的差异。

- **BuckyOS Desktop 永远是单点的**。它不会、也不计划在未来支持把多台 Desktop 组成集群。Desktop 用户拿到的始终是一个跑在自己 PC 上的、单节点的 BuckyOS Zone。
- **标准版与企业版从设计上就是集群化的**。可以从 1 个 OOD 起步，逐步扩展，**上限均为 7 个 OOD**（遵循 2n+1，n≤3 的拓扑——这是 raft 协议的硬约束，不是产品策略）。

为什么这条线划得这么死？因为集群能力会渗透到调度器、存储池、Zone 网关、服务高可用等几乎每一层内核，需要专门的硬件保障与运维约束才能做对。Desktop 的目标是"普通用户在自己的笔记本上零门槛跑起来"，把集群能力塞进去既不现实也无意义——一台笔记本掉电，再多副本也没用。

集群可用性与可靠性是 BuckyOS 的核心价值之一，**真正要把数据托付给 BuckyOS 的用户与组织，最终都应当迁移到标准版或企业版**。

---

## 3. 标准版 vs 企业版：内核相同，交付不同

这是市场伙伴最容易混淆的一点，先讲清楚：

**标准版与企业版的内核完全相同，技术能力没有差异**——节点上限同为 7，集群拓扑、调度器、存储、服务高可用等所有内核机制全部一致。差别**只在交付链路与生态角色**：

| 维度 | 标准版 | 企业版 |
| --- | --- | --- |
| 交付方 | **硬件厂商**（出厂预装） | **系统集成商**（上门部署） |
| 硬件来源 | 厂商指定的整机 | 客户现有服务器或新服务器，由集成商决定 |
| 起步规模 | 1 OOD 即可 | 默认 3 OOD 起步（保证高可用） |
| 节点上限 | 7（raft 上限） | 7（raft 上限） |
| 典型 Zone 用户数 | 个人或家庭，少量用户 | 一个 Zone 内多个普通用户 |
| 运维支持 | 用户自服务，厂商支持 | **鼓励雇佣兼职系统维护人员**长期支持 |
| 目标客群 | 家庭、个人 | 组织（公司、团队、机构） |

> 因为内核一致，所以"用户/权限管理"等场景能力在两个版本上都存在。企业版强调多用户，是**使用场景**层面的强调，不是技术能力层面的差异。

## 4. 形态与运行环境

### BuckyOS Desktop

- 安装在用户已有的 **Mac 或 Windows** 桌面系统上，作为一款桌面应用。
- 不提供 Linux 桌面版。仓库里能看到的 Linux 构建是面向**无桌面服务器**的，可视为标准版/企业版的技术预览，不属于 Desktop 产品线。
- 用户的笔记本即是 OOD，关机即下线。Zone 的可用性等于这台机器的可用性。

### BuckyOS 标准版

- 由**硬件厂商**预装出厂，开箱即用。
- 厂商负责硬件选型、整机调试、出厂前的系统初始化，用户买回家通电就能用。
- 任何硬件厂商都可以加入 BuckyOS 生态，在自家硬件上预装 BuckyOS 出厂销售。

### BuckyOS 企业版

- 由**系统集成商**为客户部署。集成商根据客户需求决定使用客户现有服务器，还是采购新硬件。
- 默认 3 OOD 起步，保证企业级高可用。
- 客户被鼓励**雇佣兼职系统维护人员**做日常支持——这是企业版正常的运维形态，不是缺陷。
- BuckyOS 公司本身不直接面向企业客户做交付；交付与一线支持都由生态里的系统集成商承担。

## 5. 能力与商业模式对比

| 维度 | Desktop | 标准版 | 企业版 |
| --- | --- | --- | --- |
| 节点数 | 永远 1 节点 | 1 → 7 | 3 → 7（推荐 3 起步） |
| 集群管理（Zone Manager 修改接口） | 不提供 | 完整 | 完整 |
| 服务高可用 / 自动迁移 | 不适用 | 支持 | 支持 |
| 数据冗余 | 用户自管 | 系统级 + Backup Suite | 系统级 + Backup Suite |
| 多用户支持 | 内核一致，但场景上以单人为主 | 内核一致，少量家庭用户 | 内核一致，**典型场景多用户** |
| 运行宿主 | 自有 Mac/Windows | 厂商预装的 BuckyOS 整机 | 客户现有或新购服务器 |
| 安装与初始化 | 用户自助下载安装 | 出厂即装好 | 由系统集成商部署 |
| 长期运维 | 用户自助 | 用户自助 + 厂商支持 | 鼓励兼职维护人员 |
| 价格 | 免费 | 含在硬件价格中 | 由系统集成商按项目报价 |
| BuckyOS 公司的商业关注点 | — | SN 相关服务 | **认证体系**（系统集成 / 系统支持 / 硬件认证） |
| 商业服务依赖 | 仅 SN；自有公网 IP + 自有域名时 0 依赖 | 同左 | **更关注私有部署**，对外服务依赖最低 |
| 升级路径 | 通过数据导入导出迁移到标准版/企业版 | 通过数据导入导出迁移到企业版 | — |

> **关于 SN 与 BuckyOS 公司的收入结构**：
> - Desktop 与标准版的商业逻辑都聚焦在 SN（远程接入与中转）相关的服务。用户如果有公网 IP + 自己的域名，可以做到 100% 不依赖任何 BuckyOS 商业服务。
> - 企业版客户更关心私有部署，几乎不依赖 SN。BuckyOS 公司在企业版上的主要收入来自**生态认证**：系统集成商资质认证、系统支持人员认证、特定服务器机型的兼容认证。

## 6. Desktop 上的功能差异详表

> 这一节回答"Desktop 因为永远单点，到底有哪些功能不可用、哪些功能退化、哪些功能完全相同"。
> **重要前提：BuckyOS 的内核在三个版本上是同一份代码**，差异不是来自"砍功能"，而是来自"单节点环境下某些机制天然没有作用对象"。

### 6.1 Desktop 上完全不可用的功能

| 功能 | 为什么不可用 |
| --- | --- |
| **集群管理（Zone Manager 修改接口）** | 没有可管理的拓扑；状态展示仍保留，永远显示"Desktop（单节点）" |
| **添加新节点** | 永远单点 |
| **服务高可用 / 自动故障迁移** | 没有迁移目标 |
| **调度器自动迁移（Scheduler Failover）** | 同上 |
| **跨节点存储池扩容** | DCFS 多节点 chunk storage 在单节点上无对象 |
| **Finder Server（多 OOD 发现服务）** | 内核明确仅在 `multi_ood` 时启动 |

### 6.2 Desktop 上退化、但仍可用的能力

这些能力在内核层面保留，但因为只有一个节点，行为会"自然降级"。**应用层和用户感知一般是无感的**，但开发者文档需要明确退化语义。

| 能力 | 在 Desktop 上的退化行为 |
| --- | --- |
| **Klog（基于 raft 的分布式日志）** | 退化为本地日志；raft 共识层为 no-op；语义上仍可调用，但不再提供"跨节点一致性"保证 |
| **系统级数据冗余** | 系统不再提供副本冗余；可靠性 100% 依赖外部备份策略（Backup Suite 或用户手动） |
| **存储池** | 仅本机磁盘组成池，容量与吞吐受单机限制 |
| **OPTask** | 仍可下发任务，但所有任务最终都跑在唯一节点上 |
| **Service HA** | 服务仍可重启，但不会"换节点" |

### 6.3 Desktop 上完全保留、与标准版/企业版一致的能力

> 这是销售与开发者最容易误解的一类——千万不要把它们当成"Desktop 砍掉的功能"。

- **多用户**（内核一致；Desktop 也能开多用户账号，只是典型场景是 1 人）
- **SN 远程接入**（Desktop 用户出门时通过 SN 回家访问自己的笔记本，**这是 Desktop 的核心使用场景**）
- **ZoneGateway 对外服务暴露**（公网 IP + 自有域名同样适用）
- **dApp 与 OpenDAN 应用生态**（同一套 App 在三个版本上都能跑）
- **Agent Runtime / Jarvis / Mia / Nexus**（笔记本算力通常足够，体验甚至更顺）
- **Computer Center Service**（LLM Router 等）
- **Message Bus / InBox**
- **system-config / 内核身份与密钥体系**

## 7. 对 App 开发者的指引

### 7.1 默认结论：不要做版本区分

**绝大多数 dApp / Agent / Workflow 在三个版本上是同一份代码、同一套 API、同一个安装包。** BuckyOS 的内核抽象（system-config、DCFS、klog、调度器、Service Loader）正是为屏蔽集群细节而设计的。

App 开发者**不应该**：
- 在代码里探测"当前是不是 Desktop"，并据此分支行为
- 写两个版本的安装包
- 假设"Desktop 上没有 SN，所以远程访问不了"——SN 在 Desktop 上是核心能力

### 7.2 Manifest 是否需要声明部署需求

### 7.4 少数需要开发者主动考虑的场景

下列场景虽然技术上可以装在 Desktop 上，但体验/可靠性不达预期，开发者**应当在 App 描述里明确推荐运行环境**（不是技术阻断）：

- **依赖"始终在线"的对外服务**：例如对外内容站、Agent 邮箱接收、Webhook 入口。Desktop 跟着笔记本作息，体验不佳。
- **作为重要数据最终归宿的服务**：例如家庭照片归档、长期文档库。这类服务应当推荐到标准版/企业版，且无论哪个版本都强烈推荐配套 Backup Suite。
- **依赖大容量本地存储的服务**：受限于笔记本磁盘容量。

## 8. 谁该用哪一个

**推荐 Desktop 给：**

- 第一次听说 BuckyOS、想低成本试用的用户
- 开发者、爱好者、想在自己电脑上跑一跑 dApp / Agent 看看效果的人
- 数据量不大、对可用性不敏感、能接受"机器关了就访问不了"的个人场景
- 还没准备好买硬件、但希望先建立账号与基础数据的潜在标准版用户

**推荐标准版给（产品发布后）：**

- 真正想把家庭数据托管在 BuckyOS 上的用户
- 对 7×24 可用性有要求的家庭场景（家庭相册、智能家居中枢、共享存储）
- 想要开箱即用、不希望自己折腾安装与运维的普通家庭用户

**推荐企业版给：**

- 多人协作、希望共用一个 Zone 的公司或团队
- 出于合规/数据主权要求必须做私有部署的组织
- 有能力雇佣（或外包）兼职运维人员的客户
- 希望由本地系统集成商提供长期支持的客户

## 9. 三个版本之间的迁移

| 起点 | 终点 | 方式 |
| --- | --- | --- |
| Desktop | 标准版 / 企业版 | **数据导入导出**（不是把 Desktop 节点加入集群） |
| 标准版 | 企业版 | **数据导入导出** |
| 同版本扩容 | 加节点 | 标准版/企业版内部支持 1→3→5→7 的逐步扩容 |

Desktop 的"永远单点"是一个不可跨越的边界。标准版与企业版虽然内核一致，但因为是两条不同的交付链路（厂商预装 vs 集成商部署），通常也通过数据导入导出来切换，而不是原地"升级许可"。

迁移工具会随标准版/企业版一起发布，目前还在规划中。

---

## 10. 关于 Desktop 的长期承诺

**BuckyOS Desktop 不是一个临时产物，会作为正式产品长期存在**。

它承担两个角色：

1. **入门通道**：让普通用户以零成本、零硬件门槛接触 BuckyOS。
2. **开发与轻量场景**：满足开发者与轻度用户对单点形态的真实需求。

哪怕标准版与企业版全面铺开后，Desktop 仍会持续维护、与其他版本同步迭代——只是它在能力边界上永远不会跨越"单点"这条线。

---

## 11. 给市场与销售伙伴的话术建议

- **不要把 Desktop 描述成"标准版的 Beta"或"过渡形态"**。它是有明确边界的正式版本，会长期存在。
- **不要承诺 Desktop 将来会支持集群**。这条路径在产品规划上是关闭的。
- **强调 BuckyOS 的开源兼容生态**：标准版不是某一家硬件厂商独占，企业版不是某一家集成商独占，谁都可以加入这个生态。
- **回答"标准版和企业版有什么技术差异"**：内核一样，差别在交付方式与目标客群——标准版是硬件厂商出厂预装的家用整机，企业版是系统集成商按项目部署的组织级方案。
- **回答"为什么不能把我现有的 PC 加进集群"**：消费级 PC 的可用性、网络环境、硬件一致性都达不到集群可靠性的前提；企业要上集群应当用企业版（由集成商部署到合适的服务器上）。
- **回答"为什么节点最多 7 个"**：这是底层 raft 协议的硬约束（2n+1，n≤3），不是产品策略，标准版和企业版都一样。
- **企业版的销售对象是系统集成商，不是终端组织**。市场端的工作重点是发展和认证集成商生态，让集成商去服务最终客户。
