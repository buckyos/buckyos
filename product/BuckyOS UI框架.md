## 一、整体设计原则：移动端优先（Mobile First）

BuckyOS 的系统 UI 设计遵循移动端优先的原则。这是所有 UI 讨论的大前提。

核心逻辑：我们希望 PI（Personal Intelligence）能够取代 PC。从这个角度看，今天拥有 PC 的普通用户已经非常少了。如果我们想做一个广泛的 C 端产品，不应该假设用户是"先有 PC 再有 PI"的。用户的典型场景是：目前只有手机，购买了 PI 之后，通过手机就可以完成对 PI 的使用。这是一种交互方式比较轻量的体验。

目前的桌面版本也已经按照移动端的 UI 结构来做了。

## 二、BuckyOS App 的基础架构

### 2.1 身份管理（钱包视图）

BuckyOS App 是系统在移动端的入口。它解决的第一个基本问题是身份管理，跟传统的 Web3 钱包基本一致。

### 2.2 社交视图（网络拓扑角度）

有了身份之后的第二层是社交。但这里的"社交"是从网络拓扑的角度来理解的，而不是传统 IM 产品那种社交场景的组织方式。它更关注的是网络场景的组织：

- 你从谁那里拉过数据（读操作）
- 你往谁那里写过数据（写操作）
- 谁主动给你写过数据
- 你有没有订阅一些群组（类似加入一个 Group）

### 2.3 协议视图与 Message Center

我们对网络的观点是从社交网络的角度来开网络拓扑。系统中有一个叫 Message Center 的组件，它本质上定义的是两个节点之间写操作的协议。

核心观点：我们认为传统 UI 的时代已经结束了。在 AI 时代，UI 很多时候是一种不太必要的生产力。向外提供服务时，无非两种：

- **数据服务（读）**：类似 RSS，提供标准协议格式的数据，最好是自描述的，其他 Agent 可以通过系统级 API 直接读取
- **数据加工服务（写）**：通过 Message Center 的 Message Object 进行投递，只要定义好标准对象，就可以实现节点之间最宽粒度的写入

好处：每个人都可以用自己机器上安装的 App 去阅读数据。例如你用 WordPress App 写博客，我用自己喜欢的 RSS Reader 来读。我只要能从你的节点上接出标准数据就可以了，用什么 App 是我自己的选择。

关键区分：App（或称 Web UI）以后更多是"向内"的、内生的，是给自己看的。而"向外"的交互不是基于 UI，而是基于 Message 协议。比如我看了你的文章想发评论，发评论的方法就是通过投递一个 Message Object。

### 2.4 应用列表与 App Launcher

BuckyOS App 的另一个核心是 App Launcher，包含应用列表。我们定义了两类 App：

- **本地 App**：不需要连接 BuckyOS 就能使用
- **远程 App**：必须连接上一个 BuckyOS 节点后才能使用

由于 Agent 时代的到来，Agent 可以很快地为用户构建定制化的 App，因此 App 数量会很容易膨胀。它更像微信小程序的模式：跟 Agent 聊着聊着就可能构造出一个临时的 App（比如朋友聚会时做个麻将积分牌）。

因此应用管理不同于传统 iPhone 的 App 组织方式，更应该强调"最近使用"或"最近构造"的概念。

## 三、HomeStation（内容中心）

### 3.1 定位：用户的内容消费中心

从用户时间分配的角度看，普通人每天主要用的东西无非两类：一个是微信（社交/消息），另一个是抖音（内容/Feed）。HomeStation 对应的就是后者，是与 Message Center 平齐的另一个 UI 组件。

HomeStation 的核心关注的是"展现"。对大部分人来说，99% 的时间是在看内容，只有大约 1% 是在发布内容。

### 3.2 信息流转逻辑

每个用户有一个 Inbox。从外部来的信息会先进入 Inbox，很多时候只是元信息（引用的索引等还没下载）。用户需要通过一个第二层的筛选 App，将筛选后的内容汇集成一个实时的 Feed List。

在 Feed List 中，用户可以通过设置不同的过滤器或切换不同的 UI 表现方式，来查看各种新内容。

总结：Message Center 是通过 AI 整合的你与网络实体之间联系的中间节点（Message Hub），而 HomeStation 是你的 Content Hub，汇集所有来自网络上的新信息和新内容。

### 3.3 Content View（内容查看器）

Feed 流之外，还需要一个通用的内容查看页。有些内容是长内容，点进去后应该有沉浸式的展现逻辑（比如手机上全屏展示）。

内容查看器的展现方式可以与用户安装的 App 相关，也可以与发布者的偏好相关。例如：精美电影可能适合全屏展示，轻松娱乐视频可能需要弹幕功能。

核心理念："我怎么看"是用户自己的选择。今天的移动互联网产品会给你固化内容查看方式，但我们强调的是：内容发布者可以建议你用某种查看器，但你也可以强行用自己喜欢的方式看。

## 四、AI Native 时代的 UI 价值观

当前有大量软件是基于 UI 的，但已经有很多软件开始 CLI 化。如果软件本质上是工具，CLI 给 Agent 用效率更高（参考《Bash is All You Need》这篇论文）。但 UI 不会消失，反而以后值得用 UI 做的东西才是真正面向人的东西。

在 BuckyOS 中，需要做 UI 的东西通常就两大类：

- **人要去看的东西**：内容展示、信息浏览等，这是价值所在
- **保证系统基本工作的东西**：辅助普通用户在系统失效时拥有最低限度的系统管理能力

如果是数据向的、人不需要去看的东西，就不要做 UI。作为操作系统，更多的 UI 应该是用户自己做的、自己装的、自己生产的，而不是系统自带的。系统自带的 UI 主要围绕以上两个点。

## 五、系统桌面设计

### 5.1 基本思路

系统桌面应该往传统安卓手机桌面的方向走，中规中矩，不犯错就行。核心是效率导向：

- 平铺的图标列表，每个图标点进去是一个功能
- 支持小部件（Widget），类似监控面板或信息卡片
- 用户可以自定义桌面布局，形成稳定的使用习惯

作为去中心化的私有部署系统，我们没有像苹果和微软那样的商业利益驱动，桌面纯粹以效率为导向。

### 5.2 PC 端与移动端的窗口管理

需要同时支持 PC 桌面和手机版。核心区别在于点击图标后的行为：

PC 端：可能弹出页面内弹窗，也可能打开新的浏览器标签页。BuckyOS App 角度：会拉出内置浏览器来展现桌面。

窗口管理的大原则：

- 独立的 App 肯定弹新窗口
- 系统维护性功能（文件浏览、系统设置、诊断工具）不开新窗口，类似 Windows 的逻辑
- 具有应用性质的功能（Workspace、照片流等）开新窗口，因为用户会在其中停留较长时间

判断标准：预测用户点击图标后会在界面上停多久，是否需要频繁切换。简单来说，99% 的情况下系统维护功能不开新窗口，只有系统内置应用才开新窗口。

## 六、Vibe Coding 的开发思路

### 6.1 LLM 写代码的原理

大语言模型写代码的本质是：把所有需求和对现有实现的观测结果塞进一个超大的提示词窗口，然后生成一组写文件的操作。每次推理都是完全独立的，模型不会像人脑一样有对系统的潜意识整体观，必须总结成提示词给它才行。

### 6.2 两个核心原则

**原则一：分离不常改的项目和常改的项目。** 有些代码一遍做到位后就不应该再进入另一个需求的上下文中。好的分层就是区分系统中基础的 UI 和易变的 UI。

**原则二：工程目录上的分层。** 最好分到不同的目录、不同的工程；即使在同一个工程中也要分到不同的文件。尽可能让 AI 一次外包 Coding 只改一个目录，而且这个目录最好没有之前改过的东西（因为 AI 特别喜欢做向下兼容，即使只是第二次迭代）。

### 6.3 识别"独创部分"与"非独创部分"

核心思路是识别系统中的独创部分（易变的）和非独创部分（稳定的）。独创部分意味着不可能一口气想清楚，需要迭代；非独创部分有明确的对标对象（比如桌面对标安卓），相对稳定。

对于独创部分，Vibe Coding 的准备要做得更充分；非独创部分可以写得更泛一点、轻松一点。

## 七、各模块规划与现状

### 7.1 Message Hub

肯定要独立出来做。本质是一个多 DID 聚合的统一消息中心。UI 设计上跟 Telegram 比较像，因为 Telegram 在功能上做得最精细，聚合也做得很好。

### 7.2 Agent IDE（原 Workspace）

正确名称应叫 Agent IDE。定位是支持用户自己构建 Agent 的新时代自然语言编程入口。通常是从现有的 Agent Fork 下来进行定制。目前成熟度不高，现有版本主要服务于开发者自己的 Agent Loop 调试需求，已基本准备废弃。

未来定位的区别：当前版本更关心平台如何工作、Agent Loop 是否在跑、执行环境、Task 父子关系等；未来面向提示词开发者时，用户主要关心的是提示词和 Loop 状态。

### 7.3 Monitor（系统监控）

查看系统状态，重点是要加入 AI 相关的状态展示。Network 的功能可以与 Monitor 一体化。唯一要区分的是传统系统监控和 AI 监控。

创新点：BuckyOS 本质上是一个分布式系统，是多节点的。虽然未来 1–2 个月内都会是单节点，但做 UI 框架时要有分布式的概念。

### 7.4 Settings（系统设置）

**当前问题**：存在两个入口，一个是弹出页面的 Settings，另一个是独立页面的系统配置，且风格不统一。

**解决方案**：独立页面的 Settings 应该去掉。根据前述原则，Settings 是一个"搞完就立刻跑掉"的东西，应该是页面内弹窗。结构上没有创新点，标准的左边配置项、右边下滑页即可，但必须加入搜索功能（因为配置项会非常多）。

Container 等面向专业用户的功能都应该收到 Settings 里面。System Log 与 Monitor 都属于系统诊断相关。基本上系统功能分为两块：实时监控（Monitor）和调试状态（Settings）。

### 7.5 Storage 与文件系统

Storage 非常值得单独拿出来做，因为与后续的文件系统和 Knowledge Base（知识库）有关。但它不需要独立 UI，用户关心的是数据总量、分布式文件系统中新设备的空间导入等。

备份与恢复：应该是一个独立的 App，但不配拥有独立的桌面图标。可以从 Settings 和 Storage 中进入。日常查看备份点只需在 Monitor 中看一眼 Last Backup Time 就可以；需要调整备份或恢复时，给用户一个大页面。

### 7.6 AI Center

**重要基础假设**：BuckyOS 不默认用户拥有 AI 能力。不管是本地模型还是远程模型，AI 能力都是昂贵的。系统基础服务不允许依赖任何 AI 能力，所有基础服务不依赖 AI 就能工作。

App 安装时会声明是否需要 AI 能力。如果系统没有 AI 能力，依赖 AI 的 App 即使安装了也是灰色的，点不进去。这也是未来推出专用硬件的核心动机：硬件开箱即用，至少本地 AI（例如 3.5B 模型）已经帮你配好了。

**AI Center 内容**：

- **模型管理**：两套来源——云端模型（Provider）和本地模型（Models）。不同于 LM Studio 面向专业用户的细粒度配置，我们让用户只关心模型本身，通过类似应用商店的方式下载安装本地模型
- **用量监控**：与计费相关，从总体使用维度和 Agent 使用维度查看账单。这里也会有 Agent 列表，提供跳转到 Agent 详情页或快速打开 Agent 的入口

产品目标：站在普通用户视角，尽可能不让用户接触 API Token 等专用性设置。

### 7.7 Workflow（工作流）

这是一个边界还没完全清晰的创新模块。核心解决的是如何实现 Agent-Human Loop：以大语言模型作为意图驱动引擎，在某些环节需要人类确认。

举例：公司的审批流程。AI 可以帮你完成单据汇总，但最终需要有人点确认。用户可以通过与 Agent 聊天聊出一个 Workflow，上线后变成可用的东西，每个 Workflow 有自己独立的链接。

Workflow 本质上是 AI 的开放性智能能力、传统计算的确定性能力和人的审计能力三者串联的一套结构。

与 Dify、DCloud 等低代码平台的根本区别：他们是用 Workflow 开发系统，我们是用 Agent 生成 Workflow 配置文件，实体化后用 UI 查看即可。

底层实现上，调度器中有一个 Function Instance 调度体系，继承了之前 BuckyCloud 的 FaaS（Function as a Service）那套设计，将 Lambda（函数）作为第一类对象。

## 八、应用商店（Store）

应用商店肯定是一个独立图标，且有资格获得独立的窗口标签页。承载这套东西的系统叫 RepoService。

不一定叫 App Store，更准确的可能是 Content Store 或直接叫 Store。因为本质上它是一个数字发行平台，面向全内容，只是因为这个时代应用更重要。

核心功能：

- 搜索浏览有趣的内容
- 收藏内容（收藏 ≠ 安装，类似 Steam 的 Library）
- 下载/使用已收藏的内容
- 对已安装内容的配置管理
- 自己发布内容的入口（包括 Jarvis 生成的 App）

## 九、当前最小流程目标

当前要跑通的最小核心流程：让 Jarvis（Agent）接收一个指令，写一个小游戏，写完后给用户一个链接。点进去后，因为是未发布的 App，会有一个标准的 Header，在 Header 中可以选择发布或分享。主要是要把传统静态网页这套流程跑通。
