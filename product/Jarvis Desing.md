

# Jarvis Build-in 能力思考

> 基于用户视角整理，用于指导后续 OpenDAN 系统开发。

---

## 1. 文档目的

这份文档不是从“模型能力”出发，而是从**用户如何理解、使用、培养 Jarvis** 的角度出发，定义 Jarvis 在 OpenDAN 中应具备的内建能力（built-in capabilities）以及系统应提供的支撑能力。

核心目标不是把 Jarvis 做成“所有领域里最强的单体智能”，而是把它做成：

* 用户的**统一入口**
* 用户长期共生、持续培养的**主 Agent**
* 能不断接入、学习、编排外部工具与子能力的**能力中枢**

---

## 2. 核心理念

### 2.1 Jarvis 是统一入口，不是可随时替换的 App

在传统 App 世界里，用户的思路是：

* A 能做某事
* B 做得更好
* 那我就从 A 迁移到 B

但在 Agent 时代，用户的思路会发生变化：

* 用户不会天然希望“更换主 Agent”
* 用户更希望“让自己的 Jarvis 学会新的能力”
* 用户对能力提升的理解，更接近于“培养”“养成”“共生”，而不是“安装后替换”

因此，OpenDAN 不应假设未来会出现大量“主 Agent 替换市场”。更合理的产品理念是：

> **用户持续使用同一个 Jarvis，新的能力通过接入、学习、编排、训练、观察和记忆，不断内化到这个 Jarvis 身上。**

Jarvis 的独特性，也不是来自初始安装包，而是来自它与用户长期相处后形成的独特习惯、记忆、偏好与工作方式。

### 2.2 多 Agent 的本质是权限与服务对象的区分

系统中存在多个 Agent，不应理解成“多个平行主角”，而应理解成：

> **多 Agent 的本质，是不同服务对象与不同权限边界下的能力投射。**

可初步划分为三类：

#### 1）Jarvis

* 面向唯一主人（owner）
* 默认只响应主人
* 即使在群聊中，也应默认只听主人的指令
* 权限最大，能力覆盖面最广
* 是用户的主 Agent / everything agent

#### 2）Diplomat（外交官）

* 可被外部对象访问
* 可配置一个有限的允许访问列表
* 因为输入来源更复杂，其权限必须显著收缩
* 本质是“对外交互代理”，而不是主 Agent 的替代品

#### 3）Mia

* 以 **local LLM** 为核心前提
* 面向高敏感私有数据场景
* 主要价值不是最高智能，而是本地化、私有化、可控性
* 如果没有本地大模型支持，Mia 原则上不应默认启用
* Mia 的未来路线应以 **local-first / local-only** 为主

这三者不是能力高低的关系，而是：

* 服务对象不同
* 输入来源不同
* 权限控制不同
* 风险模型不同
* 模型部署策略不同

## 3. 用户对 Jarvis 能力演进的期待

### 3.1 用户期待的不是“你什么都会”，而是“你能学会新能力”

用户并不会要求 Jarvis 从第一天起就具备世界上最强的写代码能力、研究能力、设计能力、采购能力。

更现实、也更符合产品逻辑的期待是：

* 当某个新的 AI 工具很强时，用户希望 Jarvis 能接入它
* 当某种新方法被证明有效时，用户希望 Jarvis 能学会它
* 当用户已经形成自己的工作习惯时，用户希望 Jarvis 能继承并优化这些习惯

因此，Jarvis 的核心竞争力不应定义为：

> “在每个垂直领域都比所有专业工具更强。”

而应定义为：

> “能够持续吸收外部能力，并在用户上下文中把这些能力发挥得更好。”

### 3.2 Jarvis 的价值不在于替代一切，而在于统一调度一切

可以把 Jarvis 看成用户与世界之间的统一界面。用户不需要直接适配每一个新的 AI 软件、工具界面或交互协议，只需要向 Jarvis 表达目标。

Jarvis 的工作是：

1. 理解用户意图
2. 判断应调用哪类工具
3. 把用户指令翻译成目标工具所需的输入结构
4. 自动补足背景信息、历史偏好和上下文
5. 以比用户直接操作更高效的方式完成任务

也就是说：

> **如果某个 AI 工具人类能用，那么 Agent 原则上也应该能用。**

而 Jarvis 要做的不只是“会调用”，还要做到：

* 知道有这个工具
* 知道怎么安装和接入
* 知道基础使用方法
* 在用户具体场景里用得更好

### 3.3 “比用户直接用更好”主要体现在两点

#### 1）自动补全上下文

传统 AI 工具常常要求用户反复写提示词、重复介绍项目背景、风格要求、工作偏好。

Jarvis 可以替用户承担这部分“体力活”：

* 自动带入用户背景
* 自动带入项目上下文
* 自动带入历史偏好
* 自动带入行为规范与习惯

#### 2）统一入口与时机控制

未来会出现大量不同结构、不同界面、不同交互方式的 AI 工具。用户不应为了完成一个目标，不断切换工具、学习界面、重写指令。

用户只需要对 Jarvis 下达目标；Jarvis 负责：

* 选择工具
* 翻译指令
* 协调执行
* 决定调用时机
* 回收结果并继续迭代

## 4. 工具观：Jarvis 需要掌握的三类能力对象

从 Agent 使用工具的角度，可以把目标能力对象分成三类。

### 4.1 第一类：传统工具

传统工具数量极大，且仍将长期存在。Jarvis 必须具备使用传统工具的能力。

可进一步分为三种：

#### 1）命令行工具（CLI）

这是 Agent 最容易使用的一类工具：

* 接口明确
* 操作可组合
* 自动化友好
* 成熟稳定

典型场景包括：

* 写代码
* 跑脚本
* 文件处理
* 构建部署
* 数据转换

#### 2）网页工具（Web）

网页本质上仍是标准化程度较高的交互界面。

Agent 可通过以下方式操作：

* 浏览器自动化
* 无头浏览器
* HTTP 请求分析与调用
* 页面结构识别与表单操作

这类工具覆盖极广，包括各种 SaaS、搜索、后台系统、在线服务等。

#### 3）传统桌面 GUI 工具

这是最重、最难标准化的一类，包括大量 Windows/macOS 上的旧式软件：

* 没有命令行接口
* 没有统一网络协议
* 交互高度依赖视觉界面

这类工具通常需要：

* 屏幕读取
* 鼠标键盘模拟
* UI 自动化
* 视觉定位与状态判断

这类能力虽然最重，但不能缺失，因为大量真实用户工作流仍建立在这类工具之上。

### 4.2 第二类：AI 原生工具

例如 Codex 一类工具，本身已经是面向 AI 交互设计的产品。

相比传统工具，AI 原生工具的关键差异在于：

* 操作核心不再是点击界面，而是构造高质量输入
* 提示词、上下文和任务描述成为主要交互对象
* 使用方法更接近“如何更好地表达任务”

Jarvis 在使用这类工具时，重点不是“学会按钮在哪”，而是：

* 学会如何组织提示词
* 学会如何表达任务边界
* 学会如何补全背景上下文
* 学会如何在结果不佳时进行追问、修正与迭代

### 4.3 第三类：SubAgent / 组合型能力

这类对象介于“工具”和“完整 Agent”之间。

它们的特点是：

* 针对某一类任务有相对固定的提示词模板
* 内部组合了若干传统工具、AI 工具或网页流程
* 不要求像通用 Agent 那样灵活，但能高效解决一类重复问题

可以理解为：

> **SubAgent 是对某类任务套路的封装。**

典型场景例如：

* 某类 research 工作流
* 商品对比与购买决策支持
* 多站点内容收集与整理
* 特定行业或固定流程的任务执行器

SubAgent 的意义在于：

* 把经常重复的套路沉淀下来
* 让 Jarvis 不必每次都从零规划
* 让用户对“某类事情”获得稳定、可复用的自动化能力

## 5. 产品定位：Jarvis 不追求“单点最强”，而追求“总体验最强”

例如在“写代码”这件事上，Jarvis 不一定要从底层智能上胜过世界上最强的编程模型或最好的编程工具。

但它应该做到：

* 知道应该优先调用哪个编程工具
* 知道用户当前正在参与哪些项目
* 知道代码库背景、讨论上下文与编码习惯
* 知道用户平时如何使用某个工具
* 能把这些信息组织成更好的输入交给专业工具

因此，Jarvis 的价值在于：

> **让专业工具在“你的场景”里表现得更好。**

这与传统“工具替换工具”的逻辑不同，Jarvis 应成为一个能力放大器和个性化编排器。

## 6. OpenDAN 需要内建的系统能力

为了让上述能力演进真正跑起来，系统层面至少要提供以下 built-in 能力。

### 6.1 权限模型与 Agent 身份模型

系统必须把“谁能向 Agent 发指令”视为一等公民能力。

需要支持：

* owner-only Agent（如 Jarvis）
* limited-access Agent（如 Diplomat）
* local-private Agent（如 Mia）
* 基于输入来源和任务类型的权限收缩
* 群聊场景下的默认忽略策略与定向响应机制

这意味着：

* 多 Agent 架构首先是**权限架构**
* 其次才是能力编排架构

### 6.2 模型路由能力

不同 Agent 的模型策略应不同：

* Jarvis：可在云端模型、本地模型、成本优先模型之间灵活切换
* Mia：以本地模型为默认甚至唯一主路线
* Diplomat：权限更小，执行边界更保守

系统需要内建：

* 模型选择与切换策略
* 成本感知
* 隐私感知
* 任务级模型路由

### 6.3 工具发现、安装与接入能力

当用户说“以后写代码都用 Codex”时，Jarvis 不应只把这当作一句偏好设置，而应能把它解释成一个完整任务链：

1. 识别这是一个新工具接入请求
2. 判断 Codex 需要什么运行环境
3. 完成安装、初始化和依赖配置
4. 让该工具进入 Jarvis 的可调用能力池
5. 记录它适合的任务类型与调用方式

因此，OpenDAN 需要内建：

* 软件安装能力
* 依赖管理能力
* 环境准备能力
* 工具注册能力
* 工具可调用性检测
* 工具知识沉淀机制

### 6.4 “共享工作环境”能力

很多能力不是靠语言讲清楚的，而是靠“你做一遍，我看着学”。

因此系统需要支持一种共生式工作环境：

* Jarvis 安装好工具后，不一定自己先完全摸索
* 它可以把工具所在环境开放给用户使用
* 用户进入该环境，亲自操作一遍
* Jarvis 在旁观察、记录、理解、提炼

这是一种非常关键的能力获取方式。

可抽象为：

> **工具环境既是执行环境，也是教学环境。**

需要支持：

* 用户与 Agent 共享同一个工作环境
* Jarvis 对用户操作进行可解释观察
* 关键步骤录制
* 任务过程回放
* 从演示中提炼出可复用流程

### 6.5 语音伴随式教学能力

理想状态下，用户在操作工具时，可以一边做、一边讲：

* 解释自己为什么这样做
* 说明在什么场景下该用哪种方法
* 指出哪些操作是关键步骤
* 告诉 Jarvis 自己真正关心的判断标准

所以系统最好支持：

* 全程语音伴随
* 操作录制与语音对齐
* 从“示范 + 讲解”中生成任务教学资料
* 将一次演示沉淀为未来可调用的使用知识

这本质上是在构建 Jarvis 的“教学视频”能力与操作记忆能力。

### 6.6 用户上下文与长期记忆能力

Jarvis 想比用户直接用工具更好，前提是它比工具更懂用户。

因此系统必须持续积累：

* 用户项目背景
* 日常讨论内容
* 工作规范
* 偏好与习惯
* 过往任务结果
* 工具使用方式

这类记忆不是可有可无，而是 Jarvis 个性化能力的核心来源。

### 6.7 SubAgent 封装与复用能力

系统需要允许把一类已经稳定下来的工作流封装成可复用能力单元。

也就是说，OpenDAN 不仅要支持“通用 Agent 调工具”，还要支持：

* 固定提示词模板
* 工具链组合
* 参数化输入
* 某类任务的一键复用
* 被 Jarvis 主动调用或推荐调用

这类能力单元就是 SubAgent 的基础。

## 7. Infra 设计要求

从系统基础设施角度，Jarvis 需要的不是一个无限膨胀的单体运行时，而是一个可调度的多环境体系。

### 7.1 Jarvis 自身 runtime 应保持轻量

Jarvis 自己运行在一个小型 Linux 容器中是合理的，但这个环境不应成为所有工具安装与执行的主战场。

原因包括：

* 容器环境天然受限
* 工具越装越多，环境会快速复杂化和脆弱化
* 不适合承载大量异构软件
* 不利于隔离和维护

因此，Jarvis 自身 runtime 更适合承担：

* 指令理解
* 任务规划
* 资源调度
* 环境跳转
* 结果汇总

而不是成为所有能力的最终承载地。

### 7.2 需要多操作系统执行面

现实中的工具分布决定了 OpenDAN 需要支持多个主流执行环境，至少包括：

* Linux
* Windows
* macOS

Jarvis 需要能够根据任务需要，调度到正确的环境中完成工作。

### 7.3 执行环境应与 Agent runtime 解耦

更合理的结构是：

* Jarvis 运行在自己的主控环境中
* 通过 SSH、远程桌面、浏览器自动化或其他控制方式
* 跳转到外部完整执行环境中使用工具

也就是说：

> **Jarvis 自己是大脑，不必把所有工具都装进脑子里；它应学会调度“身体”和“工作台”。**

### 7.4 设备与虚拟机都应纳入统一管理

按照这个思路，OpenDAN 应具备统一的设备/环境管理平面，例如：

* 用户已有设备
* 远程 Linux 主机
* Windows 虚拟机
* macOS 虚拟机
* 按需创建的临时环境

Jarvis 需要知道：

* 有哪些环境可用
* 每个环境适合做什么
* 如何连入
* 当前成本与状态如何
* 是否可与用户共享

### 7.5 通过虚拟化能力按需创建环境

系统应支持类似 PVE 的资源管理逻辑，让 Jarvis 在需要时创建一个新的工作环境，例如：

* 新建一个完整 Linux 环境用于开发/爬取/数据处理
* 新建一个 Windows 环境用于桌面 GUI 工具
* 新建一个 macOS 环境用于 Apple 生态相关任务

原则上，不需要无限扩张环境数量，而是优先保障：

* 一套主 Linux 工作环境
* 一套主 Windows 工作环境
* 一套主 macOS 工作环境

在此基础上按需扩展。

### 7.6 环境应具备“共享”属性

某些环境不仅是 Jarvis 的执行环境，也应是用户可进入的共享环境。

这很重要，因为：

* 用户可以直接演示操作
* Jarvis 可以观察与学习
* 用户可以在不破坏自己主工作机环境的前提下尝试新工具
* Agent 与用户形成“共用同一身体”的关系

这种共享环境，可以视为 Jarvis 的一种“Body”。

## 8. 对 OpenDAN 的架构启示

基于以上判断，OpenDAN 后续架构设计应优先体现以下方向。

### 8.1 从“Agent Runtime”转向“Agent + Environment Mesh”

不要把系统重点放在一个不断膨胀的 Agent runtime 上，而应转向：

* 轻量 Agent runtime
* 多执行环境管理
* 工具注册与调度
* 统一身份与权限控制
* 长期记忆与教学沉淀

### 8.2 从“模型中心”转向“能力接入中心”

单纯提升模型智力不够。对用户来说，更关键的是：

* Jarvis 是否知道有什么工具可用
* 是否能把工具接进来
* 是否会用
* 是否能学会“怎么在我这里用得更好”

### 8.3 从“直接自动化”转向“观察—学习—复用”闭环

系统不应只支持“我命令你去做”，还应支持：

1. 你先安装和准备环境
2. 我自己演示一遍
3. 你观察并理解
4. 你把流程沉淀为能力
5. 下次你自己做

这是更符合真实用户习惯的能力增长方式。

### 8.4 从“工具调用”升级为“能力内化”

一个新工具被接入后，系统不应停留在“API 可以调用”这一层，而应继续追求：

* 知识化：知道它是什么、适合什么场景
* 经验化：知道怎么用更有效
* 个性化：知道在这个用户身上怎么用更好
* 组合化：知道如何与其他工具拼接成流程

## 9. 建议的开发优先级

### P0：身份、权限与 Agent 类型边界

先把以下问题做扎实：

* Jarvis / Diplomat / Mia 的职责边界
* 输入来源控制
* owner-only 与 shared-agent 的权限模型
* 本地私有数据场景的模型策略

### P1：多环境调度基础设施

重点建设：

* 轻量 runtime + 外部执行环境的结构
* Linux / Windows / macOS 环境管理
* SSH / RDP / 浏览器自动化接入
* 虚拟机创建与销毁
* 环境状态登记与能力标注

### P2：工具接入与安装框架

重点建设：

* 工具发现
* 工具安装
* 依赖检查
* 接入注册
* 可调用性测试
* 工具能力描述

### P3：共享环境与观察式教学

重点建设：

* 用户与 Jarvis 共用工作环境
* 操作录制
* 语音伴随
* 步骤提取
* 从演示生成工作流

### P4：SubAgent 封装能力

重点建设：

* 工作流模板化
* 固定提示词封装
* 多工具组合执行
* 任务类能力市场 / 注册机制

### P5：长期记忆与个性化能力增强

重点建设：

* 项目上下文记忆
* 用户偏好记忆
* 工具使用偏好记忆
* 自动补全上下文
* 结果反馈闭环

## 10. 一句话总结

Jarvis 的 built-in 能力，不应被理解为“预装了多少功能”，而应被理解为：

> **它是否具备持续接入工具、共享环境、观察用户、沉淀经验、形成子能力，并在长期共生中不断成长的系统能力。**

对 OpenDAN 来说，真正要构建的，不是一个静态的超级助手，而是一个：

* 以 Jarvis 为统一入口
* 以权限分层组织多 Agent
* 以多环境为身体
* 以工具接入为扩展手段
* 以观察教学为学习机制
* 以长期记忆为个性化来源

的可成长 Agent 操作系统。

## 11. 可直接落地的产品原则（便于团队共识）

1. **一个用户，长期只养一个主 Jarvis。**
2. **多 Agent 首先是权限分工，不是功能堆叠。**
3. **Jarvis 不必处处最强，但必须会接入最强外部能力。**
4. **工具环境既是执行环境，也是教学环境。**
5. **用户演示一次，系统应努力学会并复用。**
6. **Jarvis runtime 要轻，执行环境要丰富。**
7. **能力增长来自长期共生，而不是频繁替换。**
8. **SubAgent 是套路沉淀，不是另起一个主 Agent。**
9. **Mia 的核心不是更聪明，而是更本地、更私有、更可控。**
10. **OpenDAN 的重点不是做一个更大的模型外壳，而是做一个可成长的 Agent 基础设施。**
