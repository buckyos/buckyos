## 1. 产品概述

BuckyOS File Browser 是 BuckyOS / Personal Server 的基础文件管理组件，也是系统的“保底入口”。对有权限的用户而言，它应当提供一个足够熟悉、稳定、可控的文件系统视角，让用户能够像使用传统桌面文件浏览器一样浏览、定位、理解和管理自己的文件。

在 AI 时代，用户可能越来越少直接进入 File Browser，但一旦进入，通常只有两类核心诉求：

1. **浏览**：按文件夹、视图、路径等传统方式理解和整理文件。
2. **查找**：在不确定文件位置时，结合目录范围、传统检索和 AI 增强能力快速找到目标文件。

因此，本产品的核心不是“重新发明文件系统”，而是在**不破坏文件系统真实性**的前提下，为传统文件浏览体验加入适度的 AI-Native 增强能力。

---

## 2. 产品定位

### 2.1 产品角色

BuckyOS File Browser 的定位是：

* 面向终端用户的基础文件浏览器
* Personal Server 文件系统的可视化入口
* AI 能力进入文件系统场景的“温和网关”
* 用户获得“掌控感”的重要系统组件

### 2.2 设计取向

整体设计应尽量遵循用户熟悉的 Windows Explorer / macOS Finder 心智模型，不在 PRD 中重新定义所有传统交互细节，而是强调以下原则：

* **熟悉优先**：用户一眼就知道怎么用，不产生陌生感
* **真实优先**：所有 AI 聚合都不能让用户误以为文件被复制、移动或生成了多份
* **信息密度优先**：进入 File Browser 的用户通常有明确目的，需要高效获得更多上下文
* **控制感优先**：用户需要理解文件在哪里、为什么能被找到、为什么某个 AI/插件行为会发生或不会发生

## 3. 背景与问题

### 3.1 背景

BuckyOS 是 Personal Server 形态的系统。即便上层存在更 AI-Native 的内容管理方式，用户仍然需要一个以“文件系统”为核心的基础组件，作为最稳定、最可预期的入口。

同时，系统中的文件来源可能非常复杂，例如：

* 本地上传
* 好友通过共享文件夹上传
* 外部 IM / Telegram 等渠道导入
* 多设备、多节点上的分布式文件系统
* AI/知识库/插件在文件之上生成的语义信息

### 3.2 现有痛点

1. **传统文件系统仍然是用户最熟悉的心智模型**
   用户需要通过文件夹、路径、列表、图标等方式理解系统文件。

2. **文件真实位置与可访问路径不够直观**
   尤其是 `public` 文件夹中的文件，除了存储路径外，还存在公网 URL。

3. **跨来源文件难以按“事件/主题”聚合查看**
   用户找一组相关文件时，不一定记得文件夹，但记得“这是一趟旅行”“这是某次聊天中收到的资料”。

4. **传统搜索召回与解释能力不足**
   用户最大的痛苦往往不是“没有文件”，而是“文件明明在，却怎么也找不到”。

5. **AI 增强能力缺少可理解、可控制的入口**
   用户需要知道为什么某个文件被纳入知识库、为什么某个插件触发了处理、某些隐私目录为什么不能触发某类管线。

## 4. 产品目标与非目标

### 4.1 产品目标

1. 提供一个符合传统文件浏览习惯的基础文件管理体验。
2. 支持用户以多种熟悉方式浏览文件夹，例如列表视图、图标视图、多层目录展开、多标签页切换。
3. 在任何时候都能让用户方便、准确地获得文件路径。
4. 对特殊目录（如 `public`）明确展示其额外可访问路径（如 URL）。
5. 提供默认的 **DFS 逻辑视图**，同时为高级用户提供 **设备/裸盘视图**。
6. 通过 AI 引入 Topic、增强搜索、增强 Meta、Story 等能力，但不能破坏用户对真实文件系统的理解。
7. 提供上传、下载、分享、协作等网盘式能力入口。
8. 让 File Browser 成为终极用户理解插件、触发器、知识库管线与权限控制的入口之一。

### 4.2 非目标

1. 不在本 PRD 中重新定义所有 Explorer / Finder 的传统交互细节。
2. 不将 File Browser 做成完整的照片库、媒体库或知识库产品。
3. 不允许 Topic 视图制造“看似复制了多份文件”的错觉。
4. 不鼓励 AI 自动构造超过三层的复杂分组结构。
5. 右侧 Preview 区不是重度内容消费或编辑场景，重点应放在元信息与上下文。

## 5. 目标用户与权限分层

| 用户角色         | 典型能力                              | 主要诉求              |
| ------------ | --------------------------------- | ----------------- |
| 普通用户         | 浏览自己有权限的 DFS 目录；通常只能看到 Home 等逻辑空间 | 浏览、查找、下载、分享、获取路径  |
| 高级用户         | 可访问更多系统暴露的设备级文件管理接口               | 查看多设备、节点、裸盘目录结构   |
| 终极用户 / 控制型用户 | 关注插件、触发器、权限、后处理管线                 | 理解系统行为、建立规则、获得掌控感 |

### 权限原则

* 所有文件、文件夹、设备、搜索结果、Topic、Meta、Story、Trigger 展示均必须受权限约束。
* 除传统访问权限外，还应支持**管线/触发器级权限控制**，决定某些目录是否允许进入知识库、触发 AI 后处理等。

## 6. 关键概念定义

### 6.1 DFS 逻辑视图

面向普通用户的默认视图。用户看到的是系统暴露出来的逻辑文件空间，类似 NAS 的共享空间体验。

### 6.2 设备 / 裸盘视图

面向高级用户。若某个设备暴露了文件管理接口，则可以在树中展开该设备，直接查看其本地目录（例如某设备的 `/opt`、`/usr/local` 等）。

### 6.3 Topic

由 AI 基于用户近期活动、主题、会话、内容关系自动发现的“主题聚合入口”。Topic 不是新的真实路径，也不是复制品，而是帮助用户快速进入一组相关文件的入口。

### 6.4 Meta

文件的元信息，包括传统元信息（大小、时间、格式、EXIF、地理位置等）和 AI 生成元信息（语义描述、标签、主题归属、相关上下文等）。

### 6.5 Story

文件的上下文故事层。它可能来自聊天记录、共享历史、过去的会话、知识库关联等，用于帮助用户回忆该文件背后的事件与语境。

### 6.6 Public URL

对于位于 `public` 文件夹内的文件，系统除显示其存储路径外，还应显示基于当前 Personal Server 域名生成的可访问 URL。

## 7. 产品原则

### 7.1 熟悉但不过时

整体布局与交互遵循用户对传统文件浏览器的预期，但在搜索、Topic、Meta、Story 上体现 AI 时代价值。

### 7.2 AI 是增强层，不是底层替代

AI 负责聚合、理解、提示和解释；文件系统本身仍然保持真实、稳定、可验证。

### 7.3 聚合入口不能替代真实路径

任何 Topic、标签、Story 入口都必须能回到文件的真实存储位置，并向用户清楚展示“文件到底在哪里”。

### 7.4 高信息密度

进入 File Browser 的用户通常是来解决问题的，因此要在有限空间内展示尽可能多、但仍可理解的信息。

### 7.5 可解释、可控制

无论是搜索结果、Topic 分类还是触发器行为，都应让用户理解“为什么会这样”。

---

## 8. 信息架构与界面布局

### 8.1 总体布局

采用标准文件浏览器结构：

* **顶部**：自定义 Title Bar + 多标签页（Tab）+ 路径/位置栏（Location）
* **左侧边栏**：标准导航区 + 树状层级入口 + AI Topic/Focus 区
* **中间主内容区**：文件/文件夹内容展示区
* **右侧面板**：Preview / Meta / Story 面板
* **底部状态栏**：文件大小、选中数、路径、附加路径等状态信息

### 8.2 视图方式

主内容区至少支持以下熟悉视图：

* 列表视图
* 图标视图

其他传统视图和细节交互可参考系统通用文件管理习惯实现，不在本文逐一展开。

### 8.3 多标签页

需支持多标签页浏览。目标是让用户方便地：

* 对比多个目录
* 在不同路径之间快速切换
* 明确当前所在位置（Location）

## 9. 功能需求

### 9.1 基础浏览能力

#### 需求说明

用户可以按照熟悉的文件系统方式浏览目录结构，并逐层查看文件夹内容。

#### 具体要求

1. 支持逐层展开和进入文件夹。
2. 支持列表视图与图标视图切换。
3. 支持标准侧边栏导航。
4. 支持多标签页。
5. 需尽量延续用户熟悉的文件管理交互习惯。
6. 系统级基础操作（如新建、移动、重命名、删除等）应遵循标准文件浏览器习惯，但不在本 PRD 细化。

---

### 9.2 路径展示能力

#### 需求说明

用户应能方便、准确地获取文件路径，这是 File Browser 的基础价值之一。

#### 具体要求

1. 在顶部位置栏中明确展示当前所在路径。
2. 在底部状态栏或详情区展示选中文件/文件夹的路径。
3. 路径信息应可复制，并保证引用准确。
4. 对特殊目录中的文件，应展示多种有意义的访问路径。

#### 特殊要求：`public` 文件夹

对于 `public` 文件夹中的文件，至少需要展示：

* 主存储路径 / 主文件系统路径
* 基于当前 Personal Server 域名生成的公网 URL

用户必须能明确理解：

* 文件真实存放在哪里
* 从互联网访问时应使用哪个 URL

---

### 9.3 双视图模型：DFS 逻辑视图 + 设备视图

#### 需求说明

系统默认提供对普通用户更友好的 DFS 逻辑视图；对高级用户，则提供更底层的设备视图。

#### 具体要求

1. **默认视图为 DFS 逻辑视图**。
   普通用户通常只看到自己有权限的逻辑目录（如 Home）。

2. **高级用户可查看设备视图**。
   若集群中的设备暴露了文件管理接口，则允许在侧边栏中展开设备树。

3. **设备视图允许浏览设备上的真实目录**。
   例如：

   * 设备 A 的 `/opt`
   * 设备 B 的 `/usr/local`

4. 所有设备展示、目录展开和文件访问均受权限控制。

---

### 9.4 AI Topic（AI 聚合主题）能力

#### 需求说明

在左侧边栏顶部提供一个 AI 驱动的 Topic / Focus 区，基于用户最近活动与内容关系，自动形成“主题聚合入口”。

#### Topic 的产品定位

* Topic 是**帮助找文件的辅助入口**
* Topic 是**聚合视图**，不是新的真实文件系统
* Topic 的价值在于：用户可能不记得目录，但记得“主题”

#### 具体要求

1. Topic 以**自动发现**为主，手动创建为辅。
2. 系统可基于用户近期活动、会话、内容关系、分享来源等自动生成 Topic。
3. 手动创建入口可以存在，但不应成为主路径，以减少与系统已识别 Topic 的冲突。
4. Topic 内文件可来自多个来源，例如：

   * Telegram / IM 导入
   * 共享文件夹上传
   * 本地上传
   * 其他系统来源

#### Topic 分组规则

1. Topic 的第一层是主题本身。
2. 第二层可按基础维度分组，例如：

   * 按来源
   * 按地点
   * 按类型
   * 按人物/是否合影等
3. 必要时可再展开一层，但**总分组层级不应超过 3 层**。
4. 分组结构应由底层 AI 提示词根据文件量与聚合特征动态构造，但需保持克制，避免复杂化。

#### 重要约束

1. Topic 不能让用户误以为系统复制了文件。
2. 不应向用户暴露“为了 Topic 而新建的符号路径/逻辑路径”作为主要概念。
3. 当用户选中 Topic 中的具体文件时，状态栏/详情区仍需明确显示其真实路径与主存储位置。

---

### 9.5 搜索能力

#### 需求说明

搜索是 File Browser 中最重要的 AI 增强入口之一，其目标是同时满足“可理解”和“有惊喜”。

#### 搜索范围

用户可在以下范围内搜索：

* 当前权限范围内的全局文件空间
* 当前目录/当前路径限定范围（用于缩小查找范围）

#### 搜索能力构成

1. **传统搜索**

   * 文件名匹配
   * 文件夹名匹配
   * 文本内容搜索
   * 倒排索引式全文搜索

2. **AI 增强搜索**

   * 基于 AI 生成的语义 Meta 检索
   * 基于知识库管线产出的内容语义召回
   * 图片/文档等内容的语义理解结果召回

#### 搜索结果要求

1. 搜索结果需在统一结果面板中展示。
2. 结果排序需兼顾传统精确匹配与 AI 召回结果。
3. 结果需尽量提示或标识来源类型，例如：

   * 文件名命中
   * 文件夹名命中
   * 全文命中
   * AI 语义命中
4. 用户应能大致理解“为什么它会出现在结果里”。

#### 产品原则

* 对用户而言，“找到”比“技术上检索了什么”更重要。
* 但对控制型用户而言，结果的可解释性同样重要。

---

### 9.6 Preview / Meta / Story 面板

#### 需求说明

当用户选中文件时，系统应在右侧面板提供丰富但可理解的元信息与上下文信息。

#### 基础信息

在状态栏或基础信息区展示：

* 文件名
* 文件大小
* 路径
* 选中状态等可预期信息

#### Preview 面板定位

* 右侧面板主要用于信息展示，而非复杂内容消费
* 重点是 **Meta 密度**，不是在狭小区域里进行重度预览

#### Meta 内容

对于文件（尤其是图片），可展示：

1. **传统元信息**

   * 文件格式
   * 大小
   * 时间
   * EXIF
   * 拍摄位置等

2. **AI 生成元信息**

   * 语义标签
   * Topic 归属
   * 内容摘要/描述
   * 美学结构、构图、场景描述等

3. **可点击关联项**

   * 标签（tag）
   * Topic
   * 来源
   * 地点
   * 人物
   * 相关文件入口

用户点击这些关联项后，应能跳转到相应 Topic、筛选视图或相关文件集合。

#### Story 能力

对于有上下文来源的文件，系统可展示 Story 信息，例如：

* 它来自谁
* 它在什么事件中产生
* 与之相关的聊天/共享/会话背景
* 某张历史照片背后的上下文

Story 是可扩展能力。重点不是固定格式，而是帮助用户在长时间跨度后仍能“想起这是什么”。

---

### 9.7 上传、下载、分享、协作

#### 需求说明

由于 BuckyOS 运行在 Personal Server 上，File Browser 需要具备典型网盘能力入口。

#### 具体要求

1. 支持上传
2. 支持下载
3. 支持分享
4. 支持协作相关入口
5. 支持从文件夹视角理解共享与上传行为

对于用户来说，这些入口应当是自然的、文件夹视角下可理解的，而不是割裂的附加能力。

---

### 9.8 插件、触发器与管线入口

#### 需求说明

File Browser 应成为终极用户理解系统扩展能力的重要入口之一。

#### 典型触发器示例

* `on_new_file_upload`
* `on_new_topic_created`

上述触发器可能进一步触发：

* 知识库整理
* AI 后处理
* 分类/摘要
* 其他插件工作流

#### 具体要求

1. 用户应能在 File Browser 中感知某些触发器/插件的存在。
2. 用户应能理解某些动作为什么发生、为什么没发生。
3. 对隐私目录，应支持限制其进入特定 AI/知识库管线。
4. 除文件访问权限外，应存在面向“AI 后处理”的权限/控制维度。

#### 产品价值

对控制型用户而言，File Browser 是从“旧世界的文件系统”过渡到“新世界的 AI 工作流”的重要网关。

## 10. 关键用户场景

### 10.1 普通用户浏览 Home 目录

用户进入默认 DFS 视图，在 Home 下按列表/图标方式浏览文件夹，通过熟悉的层级结构定位文件，并在状态栏复制路径。

### 10.2 用户在 Topic 中找回一组旅行照片

用户不记得照片存在哪个目录，但记得“那次旅行”。
侧边栏 Topic 区出现该旅行主题。用户点入后，系统按“来源 / 地点 / 类型”等方式进行最多三层聚合，快速定位相关照片。
用户点开照片后，依然可以看到它的真实存储路径与相关上下文。

### 10.3 用户查找“怎么都找不到”的文件

用户通过搜索框输入关键词。结果面板同时返回：

* 文件名命中
* 全文命中
* AI 语义命中

并通过标签说明命中来源，提升信任感与可解释性。

### 10.4 高级用户查看设备真实目录

高级用户切换或展开设备视图，在某个节点上查看 `/opt` 或 `/usr/local` 等目录，确认底层文件布局。

### 10.5 用户理解 AI 为什么处理了某个目录

用户在某个文件夹中看到它开启了某类插件/触发器，因此上传文件后会自动进入知识库；而另一个隐私目录由于权限策略，不会触发同类后处理。用户由此建立对系统行为的掌控感。

---

## 11. 交互与体验要求

1. **不陌生**：整体布局必须让有桌面操作经验的用户迅速上手。
2. **高信息密度**：浏览时尽量提供更多上下文；查找时尽量降低试错成本。
3. **适度 AI 化**：AI 入口应“增强体验”，而不是改变文件系统本质。
4. **结果可解释**：搜索、Topic、Meta、Story 都要尽可能解释“为什么”。
5. **路径始终可信**：无论从 Topic、Search 还是 Preview 进入，都必须能回到真实路径。
6. **渐进式暴露复杂度**：普通用户看到的是稳定、简单的逻辑空间；高级用户和终极用户才看到更多底层能力。

---

## 12. 权限与隐私要求

1. 文件可见性受标准访问权限控制。
2. Topic 只可聚合用户有权限看到的文件。
3. Search 不得泄露无权限内容。
4. Story 不得泄露来自无权限上下文的数据。
5. Public URL 仅适用于允许公开访问的文件。
6. AI/知识库/插件管线必须受额外策略控制，尤其是：

   * 隐私文件夹
   * 敏感目录
   * 不允许进入外部处理/语义处理的内容

---

## 13. 建议分期

### 13.1 MVP

1. 标准文件浏览器框架（顶部、多标签、侧边栏、主内容区、右侧面板、状态栏）

3. 默认 DFS 逻辑视图
4. 高级用户设备视图入口
5. 路径展示与复制

11. 上传、下载、分享入口

## 15. 一句话总结

**BuckyOS File Browser 本质上是一个“传统而可信”的文件浏览器，但它会在不破坏文件系统真实性的前提下，引入 Topic、AI 搜索、Meta、Story、Trigger 等能力，让用户在 AI 时代依然能通过文件系统获得熟悉感、效率与掌控感。**
