# Module Tier and Admission Matrix

## 1. 文档目标

本文把 [Harness Engineering.md](/Users/liuzhicong/project/buckyos/harness/Harness%20Engineering.md) 中“模块分级与模式匹配”部分独立整理为准入规则，便于模块负责人和贡献者在任务开始前就确定开发模式边界。

---

## 2. 核心原则

不同模块不应适用同一套开发自由度。

判断某个 PR 是否可接受，不只看代码是否能跑，还要看：

- 模块属于哪一类；
- 允许哪种贡献模式；
- 该模式下需要哪些额外过程证据；
- 是否超出了模块负责人愿意承担的发布风险。

---

## 3. 模块分级

### 3.1 A 类：成熟适配型模块

特点：

- 以接入、封装、适配为主；
- 主要复用成熟组件与既有模式；
- 架构创新性较低；
- 风险集中在依赖控制、适配正确性与测试覆盖。

默认策略：

- 可以允许更高比例的 Agent 主导；
- 可以允许更强自动化；
- 审查重点放在“有没有正确组合现有轮子”。

### 3.2 B 类：常规功能型模块

特点：

- 业务边界明确；
- 与系统已有模块存在稳定接口；
- 会承载真实功能逻辑，但不直接决定底层架构原则。

默认策略：

- 建议采用混合模式；
- 对文档、测试和验收要求较高；
- 允许 Agent 主导实现，但必须受更强约束。

### 3.3 C 类：创新 / 核心 / 高耦合模块

特点：

- 牵涉关键机制、协议、架构原则；
- 需求边界更依赖领域经验；
- 一旦出错，影响范围广且难以靠自动化完全兜底。

默认策略：

- 应以人主导为主；
- AI 主要用于分析、补测试、原型化、局部辅助；
- 不默认允许全自动推进。

---

## 4. 贡献模式准入矩阵

| 模块类型 | 模式一 纯人工 | 模式二 Human-Agent Loop | 模式三 Agent-Human Loop |
| --- | --- | --- | --- |
| A 类 成熟适配型 | 允许 | 允许 | 有条件允许 |
| B 类 常规功能型 | 允许 | 允许 | 有条件允许，要求更强证据 |
| C 类 创新/核心/高耦合 | 允许 | 有条件允许 | 默认不允许 |

说明：

- “有条件允许”意味着模块负责人需要显式接受该策略；
- 若涉及协议、持久数据格式、共享类型、核心调度逻辑，默认按更高风险类别处理；
- 若模块尚未分级，默认按 **B 类偏 C 类** 处理，而不是按最宽松口径处理。

---

## 5. 模块负责人决策要点

在允许模式二或模式三前，模块负责人至少应回答：

1. 该任务的验收标准是否已经文档化；
2. 任务边界是否足够清晰；
3. 是否已有成熟实现模式可复用；
4. 若 Agent 做出错误判断，是否能被测试较早发现；
5. 是否存在高危上游文档，一旦错误会系统性误导后续实现。

若上述问题无法明确回答，默认不应放宽到高自动化模式。

---

## 6. 贡献者选择模式的建议

### 6.1 适合模式二的场景

- 设计判断已相对清楚；
- 需要提升编码和测试效率；
- 人仍希望保留关键设计控制权。

### 6.2 适合模式三的场景

- 任务边界极清楚；
- 输入文档较成熟；
- 可复用模式明显；
- 测试与验证手段足够强。

### 6.3 不适合模式三的场景

- 需要边做边探索系统边界；
- 协议、数据格式或架构仍未冻结；
- 主要风险在产品判断或跨模块 tradeoff，而不是编码速度。

---

## 7. 与 PR 审查的衔接

贡献模式声明是 PR 级信息，模块分级是模块级信息。

两者必须同时成立：

- PR 声明了模式；
- 模块允许该模式。

只满足其中之一，不足以进入可发布流程。
