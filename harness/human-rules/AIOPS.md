# BuckyOS AI Ops 规划

> 面向正式生产环境的 AI 驱动运维、发布与故障处理体系  
> 版本：v0.1  
> 说明：根据语音转文字记录整理成文，保留原有业务思路，并补齐结构化章节，便于评审与落地。

## 0. 文档说明

本文用于定义 BuckyOS 正式生产环境的运维管理方式。核心目标不是让 AI 完全替代人，而是让 AI Agent 或 Agent 生成的脚本承担日常重复性、信息整合型、偏体力型的工作；人只在关键动作上完成审批、授权与最终确认。

目标形态是：

- 日常运维流程可被标准化、模板化、脚本化。
- AI 能够基于资产、监控、历史故障和发布信息，主动给出建议、报告和执行方案。
- 人员不需要做大量前置操作，只在关键节点确认即可。
- 所有高风险动作都必须保留人工授权与审计记录。

> 备注：文中部分业务缩写和术语（如 SN、OOD、LMLoader 等）沿用口述原文，后续可单独补一份术语表统一命名。

## 1. 规划目标与边界

### 1.1 总体目标

围绕正式生产环境，建立一套以 **AIOps** 为核心的运维体系，覆盖以下内容：

1. 机器资产与扩容决策。
2. 机器初始化与基础运行环境安装。
3. 监控、告警与日常巡检。
4. 发布上线与版本变更管理。
5. 故障探测、诊断、修复建议与知识沉淀。
6. 核心业务、核心数据与权限边界管理。

### 1.2 当前阶段的基本边界

当前阶段不追求“全自动自治运维”，而是强调“**AI 驱动 + 人类授权**”：

- AI 可以出 proposal、脚本、报告、排查路径和修复建议。
- AI 可以做审计、观测、诊断、对比、生成。
- AI 暂不直接承担财务采购、关键写操作、不可逆高风险动作。
- 高风险动作仍由有权限人员确认后执行。

### 1.3 当前阶段不做的事情

现阶段原则上不做以下目标：

- 不让 AI 直接完成机器采购或财务闭环。
- 不以“完全自动修复”为第一目标。
- 不盲目扩大机器 SKU、技术栈 SKU 和部署方式 SKU。
- 不在底层先做一套过重、难验证可靠性的跨机器事务系统。

## 2. 核心原则

### 2.1 AI 驱动，人类授权

整个运维体系的核心思路是：

- AI 负责理解上下文、生成方案、审计脚本、汇总状态、给出建议。
- 人负责关键确认、权限使用和最终责任。
- 所有关键动作必须可追溯、可复核、可回看。

### 2.2 尽量减少 SKU

为了降低管理复杂度，必须严格控制以下对象的种类数量：

- 机器类型 SKU
- 业务模板 SKU
- 基础环境 SKU
- 运行时版本 SKU
- 发布流程 SKU

减少 SKU 的目的不是牺牲业务，而是尽量把复杂度提前收敛，让后续 AI 更容易理解、审计和执行。

### 2.3 标准化优先于自动化

在自动化之前，先把对象和流程标准化：

- 先有资产台账，再做 AI 扩容决策。
- 先有模板化初始化，再做自动部署。
- 先有统一告警分级，再做自动诊断。
- 先有统一知识沉淀，再做智能修复。

### 2.4 只做必要、可验证的自动化

自动化本身也会引入风险，因此基础系统应先覆盖“大家都能理解、容易验证”的部分：

- 通用监控
- 标准初始化
- 标准发布
- 只读探针
- 故障建议与历史知识库

不追求一次性把所有判断都做成自动执行。

### 2.5 无状态、幂等、易迁移

系统设计上优先追求：

- 无状态
- 幂等
- 依赖关系简单
- 容易扩容与替换
- 重要数据外部持久化

这不仅降低人工运维复杂度，也有利于 AI 做稳定的判断和建议。

### 2.6 观测权限与操作权限分离

Agent、探针、诊断脚本默认应该以“只读观测”为主：

- 可以看
- 可以比对
- 可以出报告
- 不能默认写
- 不能默认改
- 不能默认执行破坏性动作

## 3. 机器资产管理与扩容决策

### 3.1 机器资产台账

机器采购本身暂时不由 AI 直接执行，因为涉及财务与审批；但 AI 应该能够基于完整、结构化、Agent 友好的资产台账，持续辅助做扩容判断。

建议资产台账至少包含如下字段：

| 字段 | 说明 |
|---|---|
| 机器 ID / 主机名 | 唯一标识 |
| 机器类型 | 接入机 / 业务机 / 核心机 |
| 所属地区 / 云厂商 / 机房 | AWS / GCP / 自有小机房等 |
| 当前用途 | DNS、HTTP、中转、业务服务、缓存、数据库等 |
| 规格 | CPU、内存、磁盘、带宽 |
| 当前负载 | CPU、内存、磁盘、带宽、请求量等 |
| 业务归属 | 所属服务组 / 模板 |
| 数据等级 | 普通 / 重要 / 核心 |
| 备份等级 | 无 / 标准 / 高等级 |
| 权限等级 | 观测、发布、管理、核心受限 |
| 成本信息 | 月成本、采购成本、续费成本 |
| 生命周期状态 | 待启用、运行中、待下线、已回收 |

这份台账是后续 AI 做判断的基础数据源。

### 3.2 AI 在扩容中的角色

AI 在机器扩容中的角色是“**辅助决策**”，而不是“直接采购”：

1. 读取资产台账与当前业务负载。
2. 判断现有结构是否合理。
3. 判断是否真的需要买机器，还是先重分配或调优。
4. 在需要扩容时，输出一份 proposal，包括：
   - 要扩哪一类机器
   - 建议买多少
   - 推荐部署在哪些区域或平台
   - 对现有 SKU 的影响
   - 预期收益、成本与风险
5. 由人工完成评审、审批与财务动作。

### 3.3 扩容设计中的主要矛盾

扩容过程中存在一个长期矛盾：

- 从管理角度，希望机器种类越少越好。
- 从业务角度，又会出现不同负载、带宽、安全性和可靠性需求。

本规划的处理方式是：

- 先把机器收敛为少数几个大类。
- 在大类内部再做有限模板差异。
- 不允许无边界地增加特殊规格和特殊流程。

### 3.4 机器三大类

| 类型 | 典型职责 | 主要特点 | 成本/部署特点 | 备份与管理要求 |
|---|---|---|---|---|
| 接入机 | DNS、HTTP、TLS 流量第一跳 | 面向大带宽、靠近流量入口，多数场景下无状态 | 可分布在小机房或边缘位置；除 DNS 外更强调成本控制 | DNS 类接入机按核心标准管理；普通接入机可偏轻量 |
| 业务机 | 常规业务服务、计算、缓存等 | 配置较“水桶”，通用性强，主要承载常规业务组 | 主要部署在 AWS / GCP 等云平台，可靠性优先 | 标准备份，更多考虑成本控制 |
| 核心机 | 订单类业务、财务敏感业务、核心数据库 | 高安全等级、隔离环境、稳定性要求最高 | 可以接受更高成本，优先保障可靠性与数据安全 | 需要高等级、定时、严格执行的备份 |

### 3.5 特殊说明：DNS 机虽然是接入机，但要按核心机标准管理

DNS 机虽然属于流量入口，但其重要性极高：

- DNS 不工作会造成明显雪崩效应。
- DNS 相关状态与恢复依赖核心数据库。
- 对外身份认证与访问入口高度依赖 DNS。

因此，DNS 机在管理策略上应按核心机标准对待：

- 轻易不变更
- 权限更严格
- 审批更严格
- 备份与恢复链路更严格

### 3.6 数据落盘与备份策略

当前整体策略是：

- 前面的“水桶机”更多承担 cache 或临时承载功能。
- 真正重要的数据优先落到 S3 这类对象存储。
- 如果后续提供备份类增值服务，也优先采用“前端 cache + 后端 S3 落盘”的模式。

由此推导出备份原则：

- **核心机**：必须做高等级、定时备份。
- **业务机 / 普通接入机**：备份密度和要求可以更强调成本控制。
- **无状态中转类机器**：原则上尽量不依赖本地持久状态。

## 4. 机器初始化与基础运维

### 4.1 模板化初始化

当一台机器被申请下来时，应该在申请阶段就明确其所属模板，而不是上线前再临时判断。

初始化的基本原则：

- 针对少数常见业务类型准备标准初始化模板。
- 模板中包含系统安全基线、运行时、目录结构、日志与监控接入等内容。
- 不允许未经讨论随意引入新的初始化路径。

### 4.2 AI Agent 的落地方式

初期出于安全和责任考虑，不建议一开始就让 Agent 以“自动触发、自行执行”的方式存在于生产环境。

更稳妥的方式是：

- 先由 AI 生成标准脚本。
- 由人审查通过后，再把脚本安装到目标机器上。
- AI 负责审计脚本是否完善、是否符合模板和业务要求。

也就是说，现阶段更适合采用：

**Agent 生成的脚本 + 人工过目 + 挂载运行**

而不是：

**全权限自治 Agent 直接长期写操作**

### 4.3 初始化后的基础组件

一台机器完成基础初始化后，至少还应安装以下组件：

1. **AI Ops Bootstrap/运维脚本**：作为后续审计、巡检、执行入口。
2. **监控组件**：采集系统与业务相关指标。
3. **最小化权限配置**：明确只读、发布、管理边界。
4. **标准日志/报告输出**：便于 AI 与人工一起审计。

## 5. 监控、告警与日常巡检

### 5.1 监控组件的目标

监控组件不仅要做通用的黑盒监控，也要在知道“这台机器跑了什么业务”的前提下做更有针对性的判断。

监控范围至少包括：

- 内存
- 带宽
- 存储空间
- 进程存活情况
- CPU 异常变化
- 与业务类型不匹配的资源行为

例如：

- 某个流量型进程平时 CPU 不高，突然 CPU 暴涨，应触发异常判断。
- 某类服务平时有固定流量画像，偏离过大应被识别。

### 5.2 监控需要与调度/分配系统联动

由于平台内部本身有服务调度或机器分配信息，因此监控系统不应只看机器本身，还应结合：

- 机器当前被分配的服务类型
- 所属业务组
- 预期流量画像
- 预期资源画像

这样才能做“有业务上下文的监控”。

### 5.3 双通道告警机制

建议明确分成两个 channel：

#### Channel A：Keepalive / 平安通道

用于定期发送“我还活着”的保活信息，并附带少量关键指标。

特点：

- 人平时基本不看。
- 主要作用是形成历史记录。
- 如果连 keepalive 都报不上来，本身就是重大问题。

#### Channel B：重要告警通道

用于真正的异常、故障和需要处理的事件。

特点：

- 必须能快速触达相关人员。
- 需要结合告警级别触发后续处理流程。
- 是后续诊断 Agent 和人工处理的主要入口。

### 5.4 三层告警级别（建议初版）

录音中明确提到告警应分三级。建议初版定义如下：

| 级别 | 含义 | 典型场景 | 处理方式 |
|---|---|---|---|
| Level 1：提示 | 已出现偏离，但未形成明显业务影响 | 单机资源异常、少量错误率抬升、keepalive 指标波动 | AI 自动汇总，人工按班次查看 |
| Level 2：警告 | 已影响局部业务，或存在明显升级风险 | 服务组异常、发布后行为异常、非核心链路中断 | 触发人工介入，AI 给出诊断建议 |
| Level 3：严重 | 影响核心链路、核心数据、核心入口或存在安全/财务风险 | DNS 异常、核心数据库异常、订单/计费链路异常 | 立即升级处理，必须人工确认关键动作 |

后续可以在此基础上补充：

- 处理 SLA
- 升级路径
- 值班责任
- 是否允许执行自动化动作

## 6. 发布与上线流程

### 6.1 发布流程必须模板化

机器准备好后，进入业务上线阶段。这里同样强调减少 SKU：

- 机器在申请时就应确定属于哪套模板。
- 上线时只允许走对应模板下的发布流程。
- 不允许临时创造“这次特殊处理”的灰色流程。

### 6.2 发布权限模型

具备上线权限的同学，应能完成从 **CI 编译结果 → 目标机器组上线** 的标准化流程。

但这里的关键点是：

- 运维不接受工程师私下发来的包。
- 上线输入必须来自统一 CI 体系。
- 人只确认来源、范围、报告和动作。

### 6.3 发布输入的唯一合法来源

上线前，必须明确以下输入：

1. **CI 产物**：一个已经编好的标准结果。
2. **Git Hash**：由有权限的人手动触发 Git Action / CI 工作流生成同样的标准结果。

无论是哪一种，最终都必须落在统一 CI 结果之上。

**禁止的方式**：

- 某工程师临时给一个包让运维上线。
- 某台机器手工编译一个“看起来差不多”的版本。
- 同一版本在不同机器上用不同方式得到结果。

### 6.4 上线前的报告

上线前应有一份由开发流程产出的标准报告，供运维确认。运维不需要关心报告如何构造，但需要基于它确认：

- 本次上线更新什么服务
- 来自哪个 CI workflow / 哪个 Git Hash
- 作用于哪些机器组
- 风险点是什么
- 是否属于模板内的正常变更

### 6.5 当前阶段的跨机器发布策略

给定一个 CI 固定结果，再给定一个机器组列表，理论上可以一口气完成全量推送；但现阶段并不要求底层实现严格的跨机器事务性发布。

当前更现实、可控的做法是：

1. 由人工发起脚本。
2. 先把目标服务全部停下来。
3. 再把新产物统一 push 到目标机器组。
4. 再把服务全部启动起来。
5. 人可以看到整体进度，并在必要时终止流程。

这套方式的前提是：

- 现阶段系统对“绝对不间断服务”要求没有那么强。
- 人工可接受一个短时、可控的停机窗口。
- 需要优先确保流程简单、稳定、可验证。

### 6.6 对业务开发的约束

为了让统一发布成立，开发侧必须遵守：

- 依赖关系清晰。
- 配置管理规范。
- 尽量不做启动顺序强依赖。
- 同一服务组内的实例应该可随意启动。

也就是说，系统设计上应尽量避免：

- A 服务必须在 B 服务之前启动。
- 某一台机器承担不可替代的特殊角色。
- 发布时需要大量人工 patch 或临时改配置。

## 7. 基础环境与技术栈规范

### 7.1 原则：减少环境复杂度

基础环境要尽量简单、统一、可复制。

如果开发希望引入新的技术环境、运行时或系统依赖，原则上应触发更高一级评审，因为这会直接带来运维和安全风险。

### 7.2 当前推荐的技术栈收敛方向

根据口述整理，当前可接受的基础环境主要包括：

- `servicegateway` 及基础设施前端接入层
- `nginx`
- `Next.js`（服务端渲染场景）
- `Go`（优先）
- `Java`（在外部 SDK 强依赖时接受）
- `Rust`（优先编译为 musl 版本）
- `Node.js LTS`
- `Deno`（可用于部分场景，但服务端可不强制）

### 7.3 运行时规范

为了减少环境冲突，所有机器上的同类组件原则上必须统一版本：

- 所有机器上的 JRE 版本统一。
- Node 统一使用固定 LTS 版本。
- Rust 尽量编译为 `musl` 目标，降低 CRT 依赖问题。
- Go 尽量维持 zero-dependency 风格，避免运行时差异。

### 7.4 依赖冲突治理原则

原则上在设计初期就消灭大部分依赖冲突问题：

- 不允许软件 A 依赖库 B、软件 C 又依赖冲突版本的 B，最后只能在机器上硬拼。
- 不允许同类业务长期运行在多个不兼容基础版本之上。
- 不允许“为了赶上线”临时引入一个后续没人维护的特殊依赖。

运维环境的目标不是“什么都能装”，而是“**能稳定、可复制、好审计**”。

## 8. 故障探针、事故处理与知识库闭环

### 8.1 故障处理的核心原则

事故处理部分是 AI 可以发挥较大价值的区域。总体原则是：

**早规划、早发现、早解决。**

### 8.2 探针体系的基本设计

探针不一定部署在生产机本身，可以部署在外部观察位，以便：

- 探测服务依赖是否正常
- 做反向探测
- 基于预期流量进行异常识别

例如：

- 某组件在正常情况下应收到一定数量请求，但某时段请求明显缺失；此时探针可以做反向判断，推测依赖链路或入口链路出了问题。

### 8.3 探针需要的权限模型

为了支持 Agent 探针，建议给探针开放目标节点的 **SSH 只读权限**：

- 可以观察环境
- 可以采集信息
- 可以比对状态
- 不能直接改配置
- 不能直接重启服务
- 不能直接执行高风险写操作

这种“只读探针 + AI 诊断”的模式，有助于在不扩大风险面的情况下，显著提升定位速度。

### 8.4 探针与人工告警的结合方式

探针不一定全自动触发，可以有两种入口：

1. **自动触发**：监控发现异常后拉起标准探针流程。
2. **人工触发**：工程师收到告警后，手工启动某类探针进行排查。

### 8.5 AI 在事故处理中的角色

探针采集完环境后，Agent 应该基于：

- 当前环境信息
- 历史故障案例
- 机器/业务模板
- 已知的典型错误模式

快速给出：

- 当前故障的可能分类
- 建议排查顺序
- 可能的典型解决方案
- 风险提示

当前阶段原则上：

- **允许 AI 给方案**
- **不要求 AI 自动执行修复**

### 8.6 故障知识库闭环

Agent 生成的诊断和建议天然应保留历史记录。随着系统运行时间增长，应逐步形成一套以“故障上下文”为核心的知识库：

- 故障现象
- 探针采集结果
- 环境信息
- 根因判断
- 最终处理动作
- 修复后的效果

最终目标是：

- 越常见的故障，定位越快。
- 越典型的环境，建议越准。
- 越往后，MTTR 越低，人工心智负担越轻。

## 9. 当前业务模板整理

### 9.1 模板一：SN DNS 解析 + HTTPS 中转入口

这是当前最重型、最关键的一类入口业务。

特征：

- DNS 服务为自研，不依赖第三方 DNS 服务器。
- 路口型 HTTPS 能力也偏自研，不完全依赖第三方入口。
- 这是整个外部访问链路中的核心入口。

规划要求：

- 至少部署两台 DNS 相关机器。
- 最好分布在两个地区。
- NS 记录至少应能指向两台机器。
- 管理标准按核心机执行，轻易不动。

原因：

- DNS 故障会造成明显的全局影响。
- 入口一旦失效，雪崩效应明显。
- 相关恢复路径依赖核心数据库和核心状态。

### 9.2 模板二：用户 OOD HTTP/TLS 流量中转机

这类机器用于给用户分配流量中转入口。

特征：

- 用户的证书不在中转机上，而在用户自己的 OOD 上。
- 中转机通过 HTTPS 的 SNI 信息做原样转发。
- 因为不承载用户证书，整体更接近无状态转发节点。
- 理论上比较容易扩容。

但该模板存在一个典型难点：

1. 用户的 OOD 需要先知道自己被分配到了哪台中转机。
2. OOD 需要主动保持到该中转机的 keepalive / 长连 / 注册关系。
3. DNS 入口还需要根据分配结果生成正确的解析记录。

问题在于：

- 用户侧的分配关系相对可控。
- 浏览器侧访问用户域名时，会受到 DNS 缓存影响。
- 因此流量可能打到“旧机器”或“错误机器”。

这意味着该模板的后续重点，不只是“扩容”，还包括：

- 错误落点时如何平滑迁移
- 分配关系变更时如何降低抖动
- DNS 缓存尚未失效时如何做兼容过渡

这是当前体系中一个明确的待解决问题。

### 9.3 模板三：3W 服务提供机（官网/登录/充值/表现层）

这类机器承载传统 Web 服务，例如：

- 官网
- 用户登录页面
- 充值页面
- 其他展示层页面

特征：

- 压力通常不会特别大。
- 性能不是第一优先级。
- 稳定性和“能打开”比高性能更重要。

针对下载类业务，建议遵循双来源策略：

- 一个来源是自有站点当前版本。
- 一个来源是 S3 上的版本。

也就是说，任何下载链接都应尽量具备双路径，以提升稳定性和兜底能力。

#### 证书管理要求

浏览器访问类业务受证书影响极大，因此必须采用严格的证书管理策略：

- 尽量不要多个业务共用一个泛化证书。
- 每个业务使用自己的证书。
- 证书更新不依赖开发同学临时处理。
- 运维应有固定周期的 renew 流程。

建议节奏：

- 每月检查
- 最长不超过每两个月完成一次统一 renew 操作

### 9.4 模板四：SN / LMLoader / AI 转发服务集群

> 注：此处名称按录音原文整理，后续需统一正式命名。

这类服务的特点是：

- 用户多通过 RPC 调用，而非浏览器调用。
- 请求量可能非常大。
- 本质上偏转发型服务。
- 核心业务约束之一是计费正确性。

因此，这类模板的设计原则必须是：

#### a. 请求尽量无状态

- 不要把单点状态压在某一台机器上。
- 尽量让节点可随时替换和扩展。

#### b. 请求必须强调幂等

特别是在计费相关场景下，必须遵循：

- **宁可少计，不可多计。**

也就是说：

- 如果用户本地看到用量高于服务端记账，那用户通常还能接受。
- 如果服务端记账高于用户实际感知用量，用户很难接受。

因此，系统设计上要尽量避免重复记账、放大记账和链路抖动导致的多计费。

#### c. 客户端应尽量具备选择节点能力

不要把所有请求都固定压到单一 API 入口。

更理想的方式是：

- 在客户端或本地请求构造过程中，具备节点选择逻辑。
- 优先让用户总是访问对自己最合适的节点。
- 对非浏览器、自有系统发起的请求，可以不必完全受限于标准 HTTPS 入口模型。

## 10. 权限与安全边界

### 10.1 关键原则

所有与生产环境有关的系统，必须首先明确权限边界，而不是先追求功能完备。

### 10.2 基本分层

建议至少分成以下几类权限：

- **观测权限**：看状态、读日志、读配置摘要、跑只读探针。
- **发布权限**：从标准 CI 产物向目标机器组部署。
- **管理权限**：修改配置、重启服务、调整分配。
- **核心权限**：涉及核心机、核心数据库、证书、计费、订单、DNS 等高风险动作。

### 10.3 AI 的权限边界

AI 默认只应拥有：

- 读资产
- 读监控
- 读历史
- 读运行状态
- 生成报告和脚本
- 给出建议

AI 不应默认拥有：

- 财务采购权限
- 核心环境写权限
- 批量不可逆执行权限
- 无审计的后台自动执行能力

### 10.4 核心对象的重点保护

以下对象应进入更高等级的保护范围：

- DNS 入口链路
- 用户核心数据库
- 订单/计费类服务
- 证书管理体系
- 核心机的备份与恢复流程

## 11. 建议的分阶段落地路径

### 阶段 1：先把基础标准化

目标：先把“对象”和“流程”统一起来。

交付内容：

- 机器资产台账
- 三类机器模板
- 常见业务初始化模板
- CI 产物唯一上线入口
- 双通道监控与告警
- 证书统一 renew 流程

### 阶段 2：引入 AI 辅助判断

目标：让 AI 开始承担“理解和建议”工作。

交付内容：

- 扩容 proposal 自动生成
- 初始化脚本审计
- 发布前报告汇总
- 业务感知监控
- 只读探针与故障诊断建议
- 初版故障知识库

### 阶段 3：形成半自动闭环

目标：让有权限人员只在关键位置确认。

交付内容：

- 关键操作确认界面
- 一键化发布脚本
- 一键化证书续期流程
- 告警到探针到报告的自动串联
- 典型故障的一键排查流程

### 阶段 4：在可控范围内开放低风险自动动作

目标：只对低风险、可回滚、可审计动作开放更高自动化。

可考虑的方向：

- 自动采集上下文
- 自动补齐诊断报告
- 自动执行只读巡检
- 对无状态、低风险节点执行有限自愈动作（需单独审批）

## 12. 当前待明确或待攻关的问题

### 12.1 DNS 与 HTTP 中转机的平滑迁移

当前最明显的技术问题之一，是流量在 DNS 缓存尚未失效时可能仍会打到旧机器。后续需要专门设计：

- 中转关系变更时的兼容期
- 错误机器收到请求后的转发策略
- 分配关系与 DNS 更新之间的双写/双活过渡策略

### 12.2 告警三级定义与响应机制

虽然已确定要分三级告警，但仍需进一步补充：

- 每一级的明确判定条件
- 处理时限
- 升级路径
- 是否支持 AI 触发预案

### 12.3 业务机是否引入 Kubernetes

录音中明确提到当前并未使用 Kubernetes，但未来在业务更复杂时可以考虑在“业务机层”引入。该事项目前不作为前置依赖。

原则上：

- 现在先把模板化、标准化做好。
- 业务复杂度足够高时，再评估是否需要 K8s。

### 12.4 术语与模板命名统一

当前录音中存在一些口述缩写和内部术语，后续应统一整理为：

- 正式业务模板名称
- 正式机器模板名称
- 正式告警类别
- 正式发布流程名称

这样才能便于文档、系统、脚本与 Agent 使用统一语言。

## 13. 结论

BuckyOS 生产环境的 AI Ops 规划，核心不是“让 AI 替人做所有决定”，而是把运维体系重构为一套：

- 机器类型有限
- 环境模板有限
- 发布入口统一
- 告警路径清晰
- 故障诊断留痕
- 人在关键处确认
- AI 在大部分重复劳动中承担主力

的运营方式。

从落地顺序上看，最重要的不是先追求复杂的自动化，而是先完成：

1. 资产数据标准化；
2. 机器与业务模板标准化；
3. 发布与监控流程标准化；
4. 只读探针与知识库闭环；
5. 高风险动作的人类授权机制。

当这些基础打牢后，AI Agent 才能真正成为生产环境的“可控助手”，而不是新的不确定性来源。
