# 系统服务持久数据格式设计任务

## 所属 Feature / 流程阶段
- Feature：
- 阶段：持久数据格式设计
- 负责人：

## 任务目标
- 在服务实现前冻结 durable data 的结构、位置、版本和兼容策略。
- 区分 durable data 与 disposable data，避免把缓存语义误设计成长期格式承诺。

## 输入
- 上一步产物：
  - 已批准的协议文档
- 相关文档：
  - `harness/Service Dev Loop.md`
  - 同类服务的数据模型、迁移策略、存储方案
- 相关代码：
  - 现有 RDB instance / object 管理使用方式
- 约束条件：
  - 服务代码进入实现前必须有持久数据格式文档
  - 数据格式变更必须可触发兼容性检查

## 任务内容
- 定义 durable data 的结构、字段语义、存储位置、schema version。
- 标明哪些数据可丢弃，哪些需要跨安装/升级保留。
- 设计升级兼容、迁移或重建策略。
- 说明哪些字段允许扩展，哪些字段不得改变语义。

## 处理流程
### Step 1
- 动作：根据协议与业务主路径识别持久数据、缓存数据和派生数据。
- Skill：`design-durable-data-schema`
- 输出：数据分类表。

### Step 2
- 动作：设计 durable data schema、版本策略、存储位置与兼容路径。
- Skill：`design-durable-data-schema`
- 输出：持久数据格式文档初稿。

### Step 3
- 动作：确认是否处于“新服务未上线、无需兼容”的开发模式，并记录重建边界。
- Skill：`design-durable-data-schema`
- 输出：定稿文档与兼容性说明。

## 输出产物
- 产物 1：持久数据格式文档
- 产物 2：数据分类与兼容策略说明
- 产物 3：迁移 / 重建决策记录

## 如何判断完成
- [ ] durable data 与 disposable data 已清楚区分
- [ ] schema version、存储位置、升级兼容策略已定义
- [ ] 已明确当前是否需要承担旧数据兼容成本

## 下一步进入条件
- [ ] 模块负责人确认数据格式可作为实现输入
- [ ] 不存在未决的数据兼容策略阻塞项

## 风险 / 注意事项
- 过早把 cache 或中间态定义成 durable data，会抬高长期兼容成本。
- 若必须直接依赖 filesystem 作为核心数据模型，必须显式标为高风险。

## 允许修改范围
- 可以修改：
  - feature 文档
  - 持久数据格式文档
  - 数据兼容性说明
- 不可以修改：
  - 业务实现代码
  - 构建与调度接入代码
  - UI 相关实现
