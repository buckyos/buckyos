# 系统服务协议设计任务

## 所属 Feature / 流程阶段
- Feature：
- 阶段：协议设计
- 负责人：

## 任务目标
- 基于 feature 文档与现有系统能力，冻结系统服务的外部边界。
- 形成可进入实现阶段的 KRPC 协议文档。

## 输入
- 上一步产物：
  - feature proposal / goal / PRD（若有）
- 相关文档：
  - `harness/Service Dev Loop.md`
  - 同类系统服务的协议文档或实现
- 相关代码：
  - 现有相似服务的 KRPC 接口、SDK 暴露方式、错误码定义
- 约束条件：
  - 协议文档必须先于实现提交
  - 协议进入实现后视为冻结边界，后续变更属于高风险

## 任务内容
- 明确服务边界：做什么、不做什么。
- 定义 KRPC 方法列表、输入输出结构、错误码语义、幂等性与副作用。
- 标出状态流转、兼容风险、SDK 暴露方式。
- 识别不清晰需求并补充到待确认问题列表。

## 处理流程
### Step 1
- 动作：阅读 feature 核心文档和同类服务设计，确认服务边界与调用方。
- Skill：`design-krpc-protocol`
- 输出：边界说明、接口草图、待确认问题列表。

### Step 2
- 动作：撰写协议文档，补齐方法、输入输出、错误码、幂等性和状态流转。
- Skill：`design-krpc-protocol`
- 输出：协议文档初稿。

### Step 3
- 动作：做一次协议自检，确认兼容性检查触发点与高风险变更位置。
- Skill：`design-krpc-protocol`
- 输出：可评审的协议文档定稿。

## 输出产物
- 产物 1：协议文档
- 产物 2：兼容性风险说明
- 产物 3：待确认问题与边界决策记录

## 如何判断完成
- [ ] 协议文档已存在且覆盖服务边界、方法、输入输出、错误码、幂等性
- [ ] SDK 暴露方式与主要调用路径已说明
- [ ] 协议兼容风险和后续检查触发规则已标出

## 下一步进入条件
- [ ] 模块负责人确认协议边界可冻结
- [ ] 不再存在阻塞实现的关键需求歧义

## 风险 / 注意事项
- 协议阶段遗漏错误码或副作用定义，会在 DV Test 阶段产生高成本返工。
- 不要在未冻结协议前启动大规模实现。

## 允许修改范围
- 可以修改：
  - feature 文档
  - 协议文档
  - 设计说明与兼容性说明
- 不可以修改：
  - 服务运行时代码
  - scheduler / build 接入代码
  - UI 相关实现
