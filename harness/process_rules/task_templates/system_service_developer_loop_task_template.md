# 系统服务 Developer Loop 收敛任务

## 所属 Feature / 流程阶段
- Feature：
- 阶段：Developer Loop
- 负责人：

## 任务目标
- 建立“运行测试 -> 看日志 -> 修代码 -> 重建/重部署 -> 再验证”的自动化闭环。
- 将本地单点环境收敛到可稳定通过关键业务测试的状态。

## 输入
- 上一步产物：
  - 可执行的 DV Test 脚本
  - 服务已能在本地环境启动
- 相关文档：
  - `harness/Service Dev Loop.md`
  - 日志路径、测试入口、部署脚本说明
- 相关代码：
  - `start.py` / `stop.py`
  - 构建脚本
  - DV Test 脚本
- 约束条件：
  - 必须明确什么算测试通过
  - 必须明确日志在哪里看、哪些日志用于定位当前问题
  - 必须明确何时覆盖安装、何时 `start.py --all`

## 任务内容
- 固化测试入口、日志路径、重建重部署命令和成功判定标准。
- 根据 DV Test 结果和日志反复修复实现问题。
- 对数据格式变更场景明确是否需要兼容旧数据，并据此选择安装模式。

## 处理流程
### Step 1
- 动作：整理本任务的 runbook，明确测试通过标准、日志入口、部署命令和数据兼容模式。
- Skill：`implement-system-service`
- 输出：Developer Loop 执行说明。

### Step 2
- 动作：执行标准 Loop：运行测试、读日志、修代码、`stop.py`、`buckyos build`、`start.py`、重新测试。
- Skill：`implement-system-service`
- 输出：问题修复记录与代码修改。

### Step 3
- 动作：确认关键业务测试、状态数据和错误日志均达到收敛标准。
- Skill：`service-dv-test`
- 输出：收敛证据与剩余风险列表。

## 输出产物
- 产物 1：Developer Loop runbook
- 产物 2：为收敛而产生的代码修复
- 产物 3：收敛结果与剩余风险说明

## 如何判断完成
- [ ] 关键业务测试通过
- [ ] 状态数据符合预期
- [ ] 无关键错误日志
- [ ] 服务能稳定运行并重复通过验证

## 下一步进入条件
- [ ] 本地环境已足够稳定，可转入安装包环境验证
- [ ] 若有 SDK 改动，已准备正式提交

## 风险 / 注意事项
- 不要在未记录原因的情况下频繁切换覆盖安装与全量重装。
- 如果每轮都需要大幅改协议或数据模型，应该回退到前置设计任务重新拆分。

## 允许修改范围
- 可以修改：
  - 服务代码
  - 构建与部署脚本中的本任务必要部分
  - DV Test 脚本
  - 与问题定位直接相关的日志和配置
- 不可以修改：
  - 无关模块的大范围实现
  - 为掩盖错误日志而关闭关键观测点
  - UI 相关实现
