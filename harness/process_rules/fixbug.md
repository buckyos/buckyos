# Bug 修复流程

1. 理解与分析
   (a) 首先要充分理解 Bug 发生时的相关信息。
   (b) 对 Bug 进行修正时，重点是根据 Bug 汇报的信息先进行分析。
   (c) 必须坚持“先分析、不要立刻开始修复”的原则。

2. 修复路径
   (a) 分析后，如果通过阅读代码和相关的关键标志位可以直接明确问题，则直接进行修改。
   (b) 如果分析后仍不能明确问题，则需要进入复现流程。复现 bug 需要经过相对仔细的规划。也就是说，从修复 bug 的角度来看，最简单的情况是能从代码上直接看出问题并明确修复；如果无法直接看出问题，就必须走复现流程。

3. 复现-修复循环主要分为以下几个步骤：

3.1 构造 Bug
   由于很多 bug 都源自生产环境，我们需要思考如何根据 bug 描述，尽可能在我们的单点测试环境（DV Test）中将其构造出来。

3.2 跑流程与观察
   在单点环境中运行相关流程。通过阅读相关的日志信息，判断 bug 产生的真实原因。

3.3 调整与迭代
   根据判断出的原因调整相关组件的实现。调整完成后，重复上述过程进行测试，直到结果符合预期。

在修复 bug 的过程中，可以充分利用日志机制：

* 可以将日志级别调低来获得更多信息

edit $BUCKYOS_ROOT/logs/log_settings.cfg for dynamic control logger.
```toml
[$module_name]
level = "debug"
```

4. 完成的定义

- 在代码中确认了Bug的根源，能明确的回答"为什么会有这个Bug"，“我是如何修复的"，而不是"测试通过了"
- 修复后，不影响系统的基准功能即可，无需进行繁重的测试（这个工作有CI完成)
