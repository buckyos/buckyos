# TaskCenter_mgr 需求文档

> 文档状态：Draft v1.0  
> 组件暂名：任务管理器（定位更接近“任务中心 + 系统事件查看器”）  
> 文档目标：基于语音记录整理 TaskCenter / Task Manager 的产品需求，用于产品、设计与客户端/服务端实现对齐。  
> **已确认决策**：在“系统事件”中，**保留同一任务的多条记录**，但仅保留 **生 / 死 / 关键拐点** 三类记录，避免事件流被高频状态变更污染。

---

## 1. 背景与定位

TaskCenter 是系统级基础应用/组件，用于承载系统内的重要长时间运行任务的统一跟踪，以及关键系统通知与用户确认的统一入口。

它不是传统意义上的“下载管理器”或“通知中心”，而是一个更偏系统基础设施的能力，承担以下职责：

1. 统一展示系统中长期运行任务的状态与结果。
2. 统一承接系统级、严肃型通知与需要用户确认的操作。
3. 为系统内其他应用/模块提供一个标准的任务查看、状态展示、事件追踪与兜底详情页能力。

整体产品定位更接近：
- **任务中心（Task Center）**
- **系统事件查看器（System Event Viewer）**
- **任务详情兜底页 / 标准状态面板**

---

## 2. 设计目标

### 2.1 核心目标

1. **让用户能快速看到“当前最需要关注的任务”**  
   重点是运行中任务、刚失败的任务、刚结束的任务、需要立刻确认的系统通知。

2. **让系统内任意任务都可被统一追踪**  
   无论任务由系统自动发起，还是由用户手工创建，都应该可在 TaskCenter 中查询与查看。

3. **让关键系统通知有统一、可信、严肃的处理入口**  
   TaskCenter 中出现的通知仅代表系统级重要通知，不承载泛滥的应用通知。

4. **为各类任务提供统一的保底展示能力**  
   对系统已知 schema 的任务提供结构化展示；对未知 schema 的任务至少能展示原始字段与状态。

5. **控制信息噪音，保证可读性与追踪性平衡**  
   首页强调“现在要处理什么”；任务页强调“任务全量查询”；系统事件强调“系统内发生过什么”。

### 2.2 非目标

1. 当前版本**不做泛应用通知中心**。
2. 当前版本**不支持以子任务为独立查询对象进行单独浏览**；用户查看的基本对象始终是“根任务”。
3. 当前版本**不要求替代业务应用自己的专属任务页面**；TaskCenter 提供的是统一入口与兜底展示。
4. 当前版本**不以社交/人际消息为主要承载场景**；人与人之间的消息/审批优先放在 MessageHub。

---

## 3. 目标用户与典型场景

### 3.1 目标用户

- 普通用户：查看下载、安装、大型工作流等任务进度与结果。
- 高级用户 / 运维型用户：查看系统事件、追踪任务生命周期、回看系统通知处理记录。
- 应用/系统开发者：通过统一任务详情页、TaskInfo Panel 接入任务状态展示能力。

### 3.2 典型场景

1. 用户安装一个体积很大的应用，需要持续跟踪安装状态。
2. 用户运行一个耗时很长的 workflow，需要查看运行进度、阶段变化、失败原因。
3. Agent 在执行过程中触发授权/二次确认，需要系统级严肃通知承接。
4. 系统出现关键告警（如磁盘空间只剩 10%），需要用户显式确认已读。
5. 用户需要查看历史任务、筛选某类任务、搜索某个任务。
6. 系统内其他页面仅需悬浮展示某个任务状态，点击后再跳转到完整详情页。

## 4. 核心概念定义

### 4.1 任务（Task）

任务是 TaskCenter 的核心对象。一个任务可以包含多个子任务，但产品层面的默认查看对象始终是**根任务（Root Task）**。

#### 任务来源
- **系统自动创建**：由系统或其他模块触发生成。
- **用户手工创建**：由用户在 TaskCenter 中主动创建。

#### 任务类型
- **一次性任务**：执行一次即结束。
- **计划任务**：按计划重复执行。
- **可扩展手工任务**：通过系统扩展注册的新任务类型，如下载、目录同步等。

### 4.2 系统通知（System Notification）

TaskCenter 首页中的通知是**系统级严肃通知**，来源必须是 **System**。  
这类通知包含：
- 需要用户立即确认/决策的通知
- 关键系统告警
- 需要手工标记已读的系统消息

### 4.3 MessageHub 与 TaskCenter 的边界

MessageHub 中可能会看到“系统来源”的消息，但两者心智不同：

- **MessageHub**：偏消息流、沟通流、Agent 发来的消息。
- **TaskCenter**：偏系统级状态、系统通知、需要明确处理动作的严肃入口。

尤其是系统安全拦截或双确认（double confirm）场景，在产品心智上应归入 TaskCenter 的系统通知能力。

### 4.4 系统事件（System Event）

系统事件是系统层面的完整事件流查看页，偏低频、偏高级用户、偏追踪/调试视角。  
其中任务相关事件是高频事件的一种，需要做专门的事件压缩与里程碑保留策略。

### 4.5 TaskInfo Panel

TaskInfo Panel 是系统级悬浮/嵌入式状态面板，用于在其他页面临时展示某个任务的状态摘要，并支持跳转到完整任务详情页。

## 5. 产品信息架构

TaskCenter 由以下三个主板块组成：

1. **首页**
2. **完整任务页**
3. **系统事件页**

此外，系统对外提供一个基础能力：

4. **TaskInfo Panel（任务信息悬浮面板）**

---

## 6. 首页需求

### 6.1 定位

首页用于承接“用户现在最需要看到与处理的信息”，强调即时性与优先级，而不是全量信息浏览。

### 6.2 首页信息优先级

首页应按以下优先级组织内容：

1. **运行中的任务**
2. **最近失败 / 最近结束的任务**
3. **需要用户立即决策的系统通知**
4. **需要用户手工确认已读的系统告警/系统消息**
5. **创建任务入口**

### 6.3 首页任务区

#### 功能要求

1. 首页应优先展示**运行中的任务**。
2. 首页可展示**最近失败或最近结束**的任务。
3. 较久之前结束的任务**不应继续占用首页空间**。
4. 首页任务对象以**根任务**为单位展示，不单独展示子任务条目。
5. 首页应提供**创建任务入口**。

#### 创建任务要求

1. 用户可在首页进入创建任务流程。
2. 任务创建入口应支持：
   - 一次性任务
   - 计划任务
   - 通过系统扩展注册的手工任务类型
3. 未来可扩展支持的手工任务示例：
   - 下载任务（类似 Download Manager / Download Station）
   - 定期同步目录
   - 其他由系统扩展提供的任务模板

#### 展示要求

首页任务卡片至少需要展示：
- 任务名称
- 来源
- 当前状态
- 最近更新时间 / 关键时间
- 主要进度信息（若有）
- 进入详情页入口

### 6.4 首页系统通知区

#### 通知范围

首页只展示**系统级重要通知**，不展示泛应用通知。

#### 通知类型

1. **需要立即确认/决策的通知**
   - 如授权确认、风险确认、执行确认等
   - 通常需要明确的 Yes / No 或类似确认动作

2. **系统告警**
   - 如磁盘空间不足（例如剩余 10%）
   - 需要用户显式确认“我知道了”或手工标记已读

#### 展示与交互要求

1. 首页中的系统通知应支持直接处理，不要求先进入详情页。
2. 若通知可跳转，则点击后应跳转到实际处理页面，而不是额外的通知详情页。
3. 若通知无需跳转，则应允许用户直接：
   - 确认
   - 拒绝
   - 标记“我知道了”
   - 标记已读
4. 用户完成处理后，该通知应从首页消失。
5. 未处理的关键通知应在首页维持可见，并可触发未读/待处理提示。

#### 设计原则

1. 首页中的系统通知**不再额外设计单独详情页**。
2. TaskCenter 中的通知是“严肃通知”，只允许系统发送，避免通知泛滥后用户失去敏感度。

## 7. 完整任务页需求

### 7.1 定位

完整任务页用于查看系统内的全部任务，是 TaskCenter 的全量任务管理入口。

### 7.2 列表要求

1. 展示历史上的**所有任务**。
2. 默认展示对象是**根任务**。
3. 不支持把子任务当作独立一级列表项单独浏览。
4. 子任务应在根任务详情页内展开查看。

### 7.3 查询与筛选要求

完整任务页至少支持以下能力：

#### 过滤器
- 按任务类型过滤
- 按任务来源过滤
- 按任务状态过滤

#### 搜索
- 支持搜索任务
- 搜索对象至少包括：
  - 任务名称
  - 任务 ID
  - 来源
  - 相关关键字（若有）

#### 排序
- 默认按最近更新时间倒序
- 后续可扩展其他排序方式（非当前 P0 必需）

### 7.4 单个任务详情入口

1. 用户点击任务列表项后进入任务详情页。
2. 任务详情页的输入参数既可以是：
   - 根任务 ID
   - 任意子任务 ID
3. 无论传入的是根任务 ID 还是子任务 ID，最终都应落到**完整的根任务详情页**。

即：  
**“带任意任务 ID 打开任务详情页” 是系统级基础能力。**

---

## 8. 任务详情页需求

### 8.1 定位

任务详情页用于展示某个任务的完整信息，是所有任务的标准兜底详情页。

### 8.2 展示原则

1. 详情页关注的是“这件事情”的完整上下文，而不是某个孤立子任务。
2. 当传入子任务 ID 时，也必须展示其所属根任务的完整详情。
3. 详情页应能体现任务与子任务的组织关系。

### 8.3 详情内容要求

任务详情页至少应展示以下信息：

- 根任务基础信息
- 当前总体状态
- 来源
- 类型
- 创建时间 / 更新时间
- 关键生命周期时间点
- 子任务结构与状态
- 最近进展 / 阶段信息
- 错误信息 / 失败原因（如有）
- 用户可执行操作（若有）
- 原始字段 / 扩展字段展示（兜底）

### 8.4 展示策略：结构化优先，原始数据兜底

#### 已知 schema 任务
对于系统已知 schema 的任务，TaskCenter 应提供结构化展示，包括但不限于：
- 标准状态字段
- 进度
- 阶段
- 错误原因
- 操作入口

#### 未知 schema 任务
对于系统未知 schema 的任务，TaskCenter 仍需提供可用的保底展示：
- 原始字段 Key-Value
- 状态
- 时间信息
- 基础元数据

#### 产品原则
TaskCenter 是系统兜底，不阻止应用提供自己的更优体验：
- 若某类任务有更适合的业务专属详情页，鼓励应用自己做深度展示。
- TaskCenter 负责提供通用、稳定、可追踪的保底能力。

---

## 9. TaskInfo Panel 需求

### 9.1 定位

TaskInfo Panel 是供系统内其他位置调用的轻量级任务状态面板，用于临时查看任务摘要。

### 9.2 能力要求

1. 支持通过任务 ID 打开。
2. 支持通过子任务 ID 打开，并自动归并到对应根任务上下文。
3. 能展示该任务当前的关键状态摘要。
4. 提供跳转到完整任务详情页的入口。

### 9.3 使用场景

- 某个业务页面需要悬浮展示任务当前状态
- 某个系统通知需要附带查看任务摘要
- 用户无需离开当前页面即可快速了解任务状态

---

## 10. 系统事件页需求

### 10.1 定位

系统事件页是首页通知与系统事件的完整归档与追踪页。  
它面向低频但高价值的查看场景，偏高级用户、偏系统追踪、偏调试用途。

与“完整任务页”不同：
- **任务页**关注“有哪些任务”
- **系统事件页**关注“系统里发生过什么”

### 10.2 基本要求

1. 默认按时间倒序排列。
2. 支持按时间线浏览历史事件。
3. 支持事件过滤与搜索。
4. 用户应能在这里看到自己处理过的系统通知历史。

### 10.3 事件类型范围

当前系统事件页至少包括：
- 系统通知事件
- 任务相关事件

当前版本**不纳入泛应用通知**。

### 10.4 任务事件展示策略（已确认）

#### 最终决策
系统事件中，**同一任务保留多条记录**，但只保留以下三类：

1. **生**：任务创建 / 开始
2. **死**：任务结束（完成 / 失败 / 取消等终态）
3. **关键拐点**：对用户追踪有价值的重要状态变化

#### 设计目的

1. 保留任务生命周期的连续性，避免完全压扁后失去追踪价值。
2. 避免所有状态变化都进入系统事件，造成事件流污染。
3. 兼顾普通可读性与高级调试追踪需求。

### 10.5 关键拐点定义

“关键拐点”不是所有状态变化，而是少量、对用户有意义的节点。  
P0 阶段建议纳入以下类型：

- 等待用户授权 / 等待用户确认
- 被系统阻塞 / 被安全策略拦截
- 进入关键阶段（如下载完成进入安装、准备阶段进入执行阶段）
- 暂停 / 恢复（仅当对用户有显著意义时）
- 重试升级为异常
- 触发显著告警或风险
- 其他由系统定义为 milestone 的节点

#### 不应进入系统事件的噪音状态示例
- 高频进度刷新（如 31%、32%、33%）
- 无业务意义的内部中间态
- 对用户不可感知、不可操作的短暂状态切换

### 10.6 任务事件记录规则

1. 同一任务允许出现多条事件记录，但必须遵循“生 / 死 / 关键拐点”原则。
2. 任务事件条目的时间戳应使用**该事件实际发生时间**，而不是统一使用任务最后更新时间。
3. 系统应避免把同类高频抖动状态反复写入事件流。
4. 事件条目点击后应跳转到对应任务详情页。
5. 事件页中的任务条目仍以根任务为承载对象，但事件可指向具体子任务上下文。

### 10.7 系统通知历史

1. 用户在首页处理过的系统通知，应在系统事件中可追溯。
2. 需要保留通知的：
   - 产生时间
   - 来源
   - 通知内容摘要
   - 用户处理动作
   - 处理时间
3. 对于“手工标记已读”的系统告警，也应保留“已读/确认”记录。

### 10.8 与未来“应用事件”能力的关系

当前版本不做应用通知中心。  
若未来确有应用级事件展示需求，可在系统事件体系下扩展一个 **应用事件（App Events）** 板块，但其展示风格可与系统事件不同：

- 更偏可读性
- 更少调试意味
- 更接近信息流/动态流

当前版本不纳入 P0 实现范围。

## 11. 交互与行为规则

### 11.1 首页行为

1. 未处理的系统通知保留在首页。
2. 用户完成处理后，从首页移除。
3. 系统告警需要手工确认或手工标记已读。
4. 首页不承担历史归档功能，历史统一进入系统事件页。

### 11.2 任务跳转行为

1. 从任意页面传入任务 ID 或子任务 ID，都应能进入完整任务详情页。
2. TaskInfo Panel 只是轻量入口，完整信息必须可继续跳转。

### 11.3 展示一致性

1. 首页、任务页、系统事件页中的同一任务，应保持一致的任务身份与状态定义。
2. 已知 schema 与未知 schema 的展示风格虽可不同，但基础字段与状态表达必须一致。

---

## 12. 数据模型与展示约束

### 12.1 根任务与子任务关系

1. 系统中的任务允许存在层级结构。
2. 列表级默认只展示根任务。
3. 子任务只在详情页内作为上下文展示。
4. 任意子任务 ID 都可作为详情页入口参数。

### 12.2 必备字段（建议）

以下为任务对象建议具备的基础字段（供产品与技术对齐）：

| 字段 | 说明 |
|---|---|
| rootTaskId | 根任务 ID |
| taskId | 当前任务 ID |
| parentTaskId | 父任务 ID（若有） |
| source | 任务来源 |
| type | 任务类型 |
| status | 当前状态 |
| title | 任务标题 |
| summary | 摘要 |
| createdAt | 创建时间 |
| updatedAt | 最近更新时间 |
| startedAt | 开始时间 |
| endedAt | 结束时间 |
| progress | 进度信息 |
| schemaType | schema 类型 |
| payload | 原始扩展数据 |

### 12.3 事件对象建议字段

| 字段 | 说明 |
|---|---|
| eventId | 事件 ID |
| eventType | 事件类型 |
| source | 事件来源 |
| relatedRootTaskId | 关联根任务 ID |
| relatedTaskId | 关联任务/子任务 ID |
| title | 事件标题 |
| summary | 事件摘要 |
| occurredAt | 事件发生时间 |
| actionState | 用户处理状态 |
| actionAt | 用户处理时间 |
| payload | 原始扩展数据 |

---

## 13. 权限与来源约束

1. **首页中的严肃通知仅允许系统发送**。
2. 应用层如需通知用户，当前应优先走：
   - MessageHub（作为消息来源）
   - 未来扩展的 App Events（非当前版本）
3. TaskCenter 不应演变为传统移动端那种可被任意应用滥发通知的中心。

---

## 14. 非功能性要求

### 14.1 可扩展性

1. 手工任务类型需要支持通过系统扩展注册。
2. 任务详情页展示 schema 需要支持扩展。
3. 事件类型需要支持扩展，但仍受系统级约束控制。

### 14.2 信息密度控制

1. 首页只保留当前最重要的信息。
2. 系统事件对任务状态变化做里程碑压缩。
3. 避免高频状态与噪音事件污染主视图。

### 14.3 兜底可用性

1. 即便任务 schema 未被系统识别，TaskCenter 也必须能够展示其基础信息与状态。
2. 即便业务应用未提供自定义详情页，用户也必须能在 TaskCenter 完成查看与追踪。

---

## 15. P0 范围建议

以下建议作为首版（P0）必须交付范围：

1. 首页
   - 运行中任务展示
   - 最近失败/结束任务展示
   - 创建任务入口
   - 系统通知直接处理
   - 已处理后从首页移除

2. 完整任务页
   - 全量任务列表
   - 过滤器（类型 / 来源 / 状态）
   - 搜索
   - 任务详情跳转

3. 任务详情页
   - 根任务视图
   - 子任务结构展示
   - 已知 schema 结构化展示
   - 未知 schema 原始字段兜底

4. TaskInfo Panel
   - 支持通过任务 ID / 子任务 ID 打开
   - 支持跳转完整详情

5. 系统事件页
   - 时间线浏览
   - 过滤器 / 搜索
   - 系统通知历史
   - 任务事件“生 / 死 / 关键拐点”保留策略

## 16. 后续演进方向（P1 / P2）

1. 更丰富的手工任务模板体系
2. 下载类、同步类任务的专属创建器与专属展示器
3. 任务详情页的可插拔自定义渲染
4. 应用事件（App Events）板块
5. 更强的事件检索、诊断与导出能力

## 17. 结论

TaskCenter 是一个系统级的“任务中心 + 系统事件查看器”基础组件。  
其核心不是简单地管理任务列表，而是：

- 承接系统内长时任务的统一追踪
- 承接系统级严肃通知与确认动作
- 提供统一的任务详情兜底能力
- 提供面向高级用户的系统事件追踪视图

### 本次讨论最终确认的关键产品决策

在 **系统事件** 中，针对任务相关记录：

- **不采用“同一任务只保留一条事件”的方案**
- **采用“保留多条，但仅保留生 / 死 / 关键拐点”的方案**

这样既能保留生命周期与时序价值，又能控制事件流噪音，适合作为当前版本的设计基线。
