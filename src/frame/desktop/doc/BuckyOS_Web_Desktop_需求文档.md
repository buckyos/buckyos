# BuckyOS Web Desktop 需求文档



---

## 1. 文档目的

本文用于将前述关于桌面层、窗口系统、布局系统与 App 接入方式的讨论，整理为一份可供产品、设计、前端、客户端与平台工程共同使用的需求文档。

本文聚焦以下内容：

1. Web Desktop 的产品定位与边界
2. 跨端一致的基础交互模型
3. 桌面布局系统（Grid / Page / Icon / Widget）
4. 窗口系统（Window / Frame / Title Bar / Open Mode）
5. App 接入与运行容器
6. 布局状态的数据结构边界与后续扩展方向

本文**不讨论视觉风格与审美语言**，仅讨论可交互性、系统能力、数据结构与实现边界。

---

## 2. 产品定位

### 2.1 核心定位

BuckyOS Web Desktop 的定位是一个**尽可能轻量的桌面层（Thin Desktop Layer）**。

它不是一个重型操作系统，也不是一个试图重新发明整套图形界面的系统；它的核心职责是：

1. 提供桌面入口与桌面布局
2. 提供应用启动能力
3. 在需要时提供窗口承载与窗口管理能力

可概括为：

> Desktop = App Launcher + Layout Container + Window Container

实现完成后，得到一个能过通过UIDataModel设置好的，类似Android风格的可用桌面。
该桌面包括时钟,notepad2个默认的小部件，以及正常Mobile App Icon，帮助产品经理体验后最终确认。

### 2.2 设计原则

- 尽量少做事情
- 能交给系统处理的能力，不在 Web Desktop 内重复实现
- 不轻易重新发明交互范式
- 优先采用成熟系统已验证的交互经验
- 优先保证 App 开发者面对的是稳定、可预期的模型

### 2.3 非目标

当前阶段不作为目标的内容包括：

- 自定义视觉风格体系
- 重型桌面特效驱动的系统设计
- 在桌面 App 容器中重做 OS 级窗口管理
- 一开始就定义完整的任务栏 / 开始菜单 / 启动器样式方案
- 一开始就做复杂的多端同步策略与冲突合并策略

## 3. 顶层硬性要求

### 3.1 不讨论视觉风格

当前阶段不定义整体 UI 风格，不讨论审美与品牌视觉层面的具体方案。

当前只讨论：

- 可交互性
- 跨端适配
- 主题能力
- 多语言能力
- 布局模型
- 窗口模型

### 3.2 多语言

系统必须支持**八国语言切换**。

要求：

- 系统级 UI 文案必须可翻译
- 所有基础组件必须支持 i18n
- 布局与组件应考虑不同语言长度差异

### 3.3 主题

系统必须支持**明亮 / 黑暗**两套主题切换。

要求：

- 主题能力为系统级能力
- 基础组件默认支持 light / dark
- 不将主题适配责任下放给各个 App 独自处理

### 3.4 移动优先

系统整体遵循**移动优先（Mobile First）**原则。

要求：

- 先保证移动端成立
- 桌面端在移动端语义之上做增强
- 触屏交互为基础约束之一

### 3.5 跨平台

同一套桌面体系需要同时覆盖：

- 移动平台
- 桌面平台
- 浏览器运行环境

这里的“同时支持”并不要求内部完全同构实现，但要求对 App 开发者提供尽可能稳定的一致性语义。

## 4. 交互参考基线

### 4.1 主要参考对象

整体操作逻辑优先参考 Android 已经成熟的跨端经验，原因包括：

- Android 手机端交互已充分成熟
- Android Tablet / 大屏适配已有较稳定经验
- 有助于降低从零发明桌面与移动双端交互的风险

### 4.2 参考策略

参考 Android，并不意味着复制 Android；其意义在于：

- 提供跨端一致性的参考源
- 避免系统级 UI 犯低级错误
- 尽量复用用户已经熟悉的交互语义

### 4.3 跨端一致性的目标

内部实现上，移动端与桌面端可以存在差异；但对于 App 开发者，系统应尽量避免暴露两套完全不同的交互契约。

应遵循：

> 系统内部可以分化实现，但面向 App 的打开语义与窗口语义要尽量统一。

---

## 5. 桌面层职责边界

### 5.1 桌面层负责什么

Web Desktop 的核心职责包括：

1. 桌面布局管理
2. 应用启动与入口管理
3. 在同一 WebView 内运行多个 App 时的窗口管理
4. 为系统内与接入 App 提供基础承载能力

### 5.2 桌面层不负责什么

以下内容不应默认归入桌面层核心职责：

- App 内部业务逻辑
- 复杂系统功能入口
- 所有系统级浮层与通知中心
- 浏览器或操作系统已经提供的窗口管理

### 5.3 核心抽象

对桌面层而言，App 的本质是：

> 一个被打开并被放入某个容器中的内容对象。

因此，桌面层的核心问题不是“App 内部如何工作”，而是：

- App 如何被打开
- App 在哪个容器中被打开
- App 以何种显示模式被承载

---

## 6. 运行容器（Runtime Container）

系统存在三类真实运行容器：

### 6.1 标准浏览器容器（Browser）

这是最基础的运行环境，包括：

- 桌面浏览器
- 移动浏览器

在该模式下：

- Desktop 自身是一个 WebUI 页面
- App 可以在当前 WebUI 内打开，也可以在新的浏览器容器中打开
- 该模式下通常需要自行实现 Web 层窗口管理

### 6.2 桌面 App 容器（Desktop App）

在桌面客户端环境中：

- BuckyOS 自身运行于独立桌面 App 内
- App 被打开后，可优先使用操作系统原生窗口承载
- App 内容通过独立 WebView 加载到 OS Window 中

该模式下应优先采用：

> 每个 App 使用新的 OS Window 打开

这样做的结果是：

- 不需要在 Web Desktop 层重复实现复杂窗口管理
- Desktop 只负责启动 App
- OS 负责窗口系统

### 6.3 移动 App 容器（Mobile App）

在移动客户端环境中：

- 应尽可能复用移动操作系统已有的应用打开与任务切换能力
- App 打开后默认应尽可能表现为全屏独立应用
- 能由系统管理的多任务，优先交给系统管理

平台差异：

- Android：更适合进行多任务与窗口相关实验
- iPad：支持多窗口与多场景
- iPhone：窗口能力较受限

---

## 7. 打开模式（Open Mode）

BuckyOS Web Desktop 中，App 的打开模式是核心抽象之一。

### 7.1 两类核心打开模式

#### 模式 A：当前容器内打开（In-place）

特点：

- App 在当前 Desktop 所在的 WebView 内打开
- 不创建新的 WebView / Tab / OS Window
- 需要由 Desktop 管理该 App 的显示与窗口状态

适用场景：

- 浏览器中的 Web Desktop
- 系统 UI
- 已接入 SDK 且允许嵌入的 App

#### 模式 B：新容器打开（New Container）

特点：

- App 在新的 WebView / 新的浏览器标签页 / 新的 OS Window / 新的移动任务中打开
- Desktop 只负责发起启动
- 后续容器生命周期主要由浏览器或操作系统管理

### 7.2 复杂度分界线

整个系统的复杂度，主要取决于：

> App 是否在同一 WebView 内被打开。

结论：

- **新容器打开**：Desktop 复杂度低
- **同一容器内打开**：Desktop 必须实现窗口系统

---

## 8. 显示模式（Display Mode）

从 Desktop 的视角，App 被打开后最终表现为某种显示模式。

### 8.1 支持的显示模式

#### 1）普通窗口模式（Windowed）

- 可拖动
- 可改变位置
- 可与其他窗口并存
- 主要用于桌面端

#### 2）最大化模式（Maximized）

- 占满当前工作区
- 仍属于窗口系统的一部分
- 移动端默认表现通常等价于此

#### 3）全屏模式（Fullscreen）

- 尽量脱离桌面 UI
- 适合沉浸式场景

#### 4）浏览器级全屏（Browser Fullscreen）

- 脱离浏览器常规界面
- 比普通全屏更彻底

### 8.2 最小化（Minimize）

最小化不是关闭。

最小化的语义为：

- App 仍然存活
- 仅从前台显示中移除
- 系统需要提供便捷的窗口 / 应用切换能力

最小化会隐含引入：

- 运行中 App 集合
- 前后台状态管理
- 最小化窗口的恢复能力

---

## 9. 核心工程拆分

整个 Web Desktop 的实现可收敛为两个主要工程模块：

### 9.1 模块一：Desktop Layout System

负责“纯桌面态”的布局与交互。

包含：

- Grid / Page / Icon / Widget 布局规则
- 拖拽交互
- 分页管理
- Layout State 数据结构定义
- 本地持久化

### 9.2 模块二：Window System

负责“App 打开后”的窗口承载与窗口管理。

包含：

- Window Frame
- 窗口状态机
- 窗口拖动 / 排列 / 最大化 / 最小化 / 关闭
- 移动端 Title Bar 模型
- App 与窗口系统之间的配置协议

---

## 10. 原始桌面（Base Desktop）

### 10.1 原始形态

桌面的原始形态应尽可能简单，仅包含：

1. 分页网格（Grid Pages）
2. 当前页指示器（小圆点）

除此之外，不默认包含：

- taskbar
- start menu
- launcher bar
- 复杂状态栏
- 额外系统 UI

### 10.2 设计原因

当前应优先把桌面底层抽象稳定下来，而不是过早绑定某一种系统 UI 样式（例如偏 Windows、偏 macOS 或偏 Android 的具体入口结构）。

---

## 11. System UI 与 Dead Zone

### 11.1 Desktop 与 System UI 解耦

后续可能存在的：

- taskbar
- launcher
- 系统入口
- 最小化窗口管理区
- 顶栏 / 底栏 / 侧栏

这些都属于 **System UI**，而不是原始桌面本身。

### 11.2 Dead Zone 抽象

为避免过早决定 System UI 的具体形式，桌面层应支持定义四个方向的不可放置区域（Dead Zone）：

- top
- bottom
- left
- right

Dead Zone 的作用：

- 不影响壁纸铺设
- 只限制桌面元素（Icon / Widget）的可放置区域
- 同时决定窗口工作区可用范围

### 11.3 Dead Zone 的价值

通过 Dead Zone，可以在不破坏桌面布局与窗口系统的前提下，在后续版本中迭代：

- 顶部系统栏
- 底部任务区
- 侧边启动器
- 移动端系统切换区域

---

## 12. 桌面布局系统（Grid Layout System）

### 12.1 布局基础

桌面本质是一个**分页二维网格系统**。

模型：

- 桌面由多屏（Pages）组成
- 每一屏由 rows × columns 的网格构成

### 12.2 跨端语义一致

移动端与桌面端都使用网格布局，但网格规模不同：

- 移动端：网格较小
- 桌面端：网格较大

原则上：

> 桌面端并不是另一套语义，只是拥有更多网格空间。

### 12.3 图标尺寸与网格关系

图标大小不作为独立抽象优先定义，而是通过网格大小间接影响。

即：

- 调整网格密度
- 间接改变图标视觉占用

### 12.4 元素类型

桌面网格中只存在两类基础元素：

#### 1）App Icon

- 默认占用 1 × 1
- 点击后触发应用打开

#### 2）Widget / Panel

- 支持占用多个网格单元
- 例如 2 × 1、2 × 2 等
- 开发时需要声明自身支持的尺寸范围

### 12.5 Widget 的核心约束

Widget 需要先定义其可支持尺寸，例如：

- 1 × 1
- 2 × 1
- 2 × 2
- 4 × 1

跨端要求：

- 使用统一网格语义
- 通过不同平台的网格数量与空间条件体现信息密度差异
- 同一个 Widget 在桌面端与移动端可展示不同的信息密度，但仍基于同一套尺寸抽象

### 12.6 分页机制

#### 自动分页

- 当当前页已被占满时，应自动创建下一页

#### 手动分页

- 用户即使在当前页未满时，也可以将元素拖动到下一页

---

## 13. 桌面交互

### 13.1 基础交互方式

桌面元素应支持的标准交互包括：

- 点击打开 App
- 拖拽移动位置
- 长按进入编辑 / 拖动状态（移动端）
- 鼠标拖动（桌面端）
- 右键打开上下文菜单（桌面端）

### 13.2 拖拽语义

拖拽的核心目标是修改布局状态，而不是直接操作 DOM 结果。

拖拽支持：

- 同页位置调整
- 跨页移动
- 拖到空格子后落位
- Widget 与 Icon 的位置占用判断

### 13.3 保存目标

所有桌面交互操作的最终目标，都是稳定地更新并保存布局状态。

---

## 14. Layout State（布局状态）

### 14.1 核心地位

Layout State 是整个 Desktop Layout System 的核心数据模型。

它需要满足：

- 可以完整表达桌面布局
- 能持久化保存
- 可以在未来做版本演进
- 向下兼容历史数据
- 为同步与跨端扩展预留空间

### 14.2 Layout State 的最小语义

需要至少表达：

- 当前有哪些页面
- 每个页面上有哪些元素
- 每个元素的位置
- 每个元素的尺寸
- 元素类型（App / Widget）
- Widget 的实例标识与配置

### 14.3 建议的最小数据结构（草案）

```json
{
  "version": 1,
  "formFactor": "desktop",
  "pages": [
    {
      "id": "page-1",
      "items": [
        {
          "id": "app-mail",
          "type": "app",
          "appId": "mail",
          "x": 0,
          "y": 0,
          "w": 1,
          "h": 1
        },
        {
          "id": "widget-monitor-1",
          "type": "widget",
          "widgetType": "system-monitor",
          "x": 1,
          "y": 0,
          "w": 2,
          "h": 1,
          "config": {}
        }
      ]
    }
  ]
}
```

说明：

- `version` 用于未来升级与兼容
- `formFactor` 用于区分 desktop / mobile
- `pages` 为分页集合
- `items` 为页面内元素集合

### 14.4 存储策略

当前阶段：

- 浏览器环境下优先使用 `localStorage`

未来阶段：

- 支持同步到用户 Personal Server
- 支持不同设备与会话的扩展作用域

---

## 15. Layout Scope（布局作用域）

### 15.1 Desktop / Mobile 分离

桌面布局与移动布局必须视为两套独立布局。

结论：

- 桌面布局变更不能污染移动布局
- 移动布局变更不能覆盖桌面布局

### 15.2 本地存储要求

即使在同一个浏览器中，如果用户切换了桌面模式与移动模式，也必须使用不同的 `localStorage key` 保存布局。

### 15.3 未来同步方向

后续在 Personal Server 上，至少维护两类布局：

- desktop layout
- mobile layout

并可在未来扩展到：

- device-specific layout
- session-specific layout

### 15.4 当前阶段的边界

当前阶段优先任务是：

- 定义稳定的 Layout State 结构
- 定义 desktop / mobile 布局隔离方式

当前阶段暂不深入：

- 同步协议设计
- 冲突合并策略
- 设备级布局策略细则

---

## 16. 窗口系统（Window System）

### 16.1 启动后承载职责

窗口系统负责在 App 已经被打开后，对其提供显示容器与基本管理能力。

### 16.2 窗口系统职责

窗口系统至少应覆盖：

- 打开
- 关闭
- 最小化
- 最大化
- 聚焦与失焦
- 遮挡顺序（z-order）
- 工作区适配（受 Dead Zone 影响）

### 16.3 逐步实现原则

窗口系统应分阶段实现，不建议一开始做过重的窗口管理器。

初期优先实现：

- 基础 Window Frame
- 基础状态切换
- 基础拖动
- 基础最大化 / 最小化 / 关闭

后续再考虑：

- 自动平铺排列
- 高级分屏
- 更复杂的窗口编排算法

---

## 17. Window Frame（窗口外框）

### 17.1 桌面端窗口外框

桌面端窗口外框遵循轻量原则：

- 左上显示标题
- 右上提供控制按钮
- 四周边框尽量轻
- 不做重型系统框架

至少应包含三个基础按钮：

- 最小化
- 最大化
- 关闭

可选扩展按钮：

- 全屏
- 其他 App 自定义功能按钮

### 17.2 视觉要求

视觉层面可追求精致，如：

- 毛玻璃
- 半透明
- 轻阴影
- 圆角

但这些属于视觉实现细节，不影响当前结构性需求。

### 17.3 移动端窗口外框

移动端不适合照搬桌面端传统边框模型。

移动端窗口语义应压缩为：

> 以 Title Bar 为核心的最大化容器

---

## 18. 移动端 Title Bar 模型

### 18.1 前提

在移动端，即使支持 In-place 打开，窗口通常也应以“接近全屏”的形式出现。

### 18.2 两类最大化语义

移动端至少需要区分两种最大化：

#### 1）覆盖 Dead Zone 的主流全屏模式

特点：

- 尽可能给 App 最大空间
- 更接近浏览器全屏 / 沉浸式使用
- 适合作为默认移动端模式

#### 2）保留 Dead Zone 的系统 UI 模式

特点：

- 允许系统保留外部切换区域
- 更适合设置等轻量系统面板
- 更容易快速切回其他区域

### 18.3 Title Bar 是移动端的核心控制区

移动端窗口控制的重点集中在 Title Bar。

Title Bar 需要承担：

- 标题展示
- 可选副标题 / 提示
- 返回 / 退出语义
- 控制按钮承载

### 18.4 系统兜底 Title Bar

如果 App 未提供自身配置，则由系统提供标准 Title Bar。

系统 Title Bar 的默认能力包括：

- 显示标题
- 左侧返回或回桌面入口
- 右侧控制按钮区
- 在必要时支持描述文本

### 18.5 App 自定义优先

如果 App 在 Manifest 中声明了其窗口配置，则优先使用 App 的声明式配置。

---

## 19. 窗口内容嵌入方式

在同一 WebView 内运行窗口时，窗口内容存在多种嵌入方式：

### 19.1 模式一：直接嵌入系统内组件

适用于：

- 系统内置 App
- 与 Web Desktop 同构、同源、同编译流程的内容

特点：

- 最容易实现
- 控制力最强
- 适合系统设置、系统面板等内容

### 19.2 模式二：iframe 嵌入

适用于：

- 标准 Web App
- 需要一定隔离性的 App

特点：

- 通用性较强
- 集成灵活
- 需要额外关注窗口层级、遮挡与通信问题

### 19.3 模式三：同源异构 App 嵌入

例如：

- 系统主框架使用 React
- 某个 App 使用 Vue 或其他技术栈
- 仍然处于受控的同源环境中

该模式需要进一步定义更合适的接入桥接方案，但不改变“窗口系统提供承载容器”的总体模型。

---

## 20. App 接入分级

从接入深度上，可将 App 分为三类：

### 20.1 系统内置 App

特点：

- 与系统同源、同构或同编译流程
- 优先使用系统 Manifest 与系统窗口能力
- 最容易做到体验统一

### 20.2 SDK 接入 App

特点：

- 已使用 BuckyOS 提供的 SDK 或接入协议
- 可在同一 WebView 内运行
- 能感知自身运行模式
- 能参与统一窗口系统与切换流程

### 20.3 外部 / 套壳 App

特点：

- 没有接入 BuckyOS SDK
- 不应默认直接运行在同一 WebView 内
- 应优先通过独立 WebView 或新容器打开

原因：

- 避免双 Title Bar
- 避免系统 UI 与 App UI 冲突
- 降低集成成本与异常风险

结论：

> 未接入 SDK 的 App，默认走独立容器；接入 SDK 的 App，才允许更深入参与同一 WebView 内的窗口系统。

---

## 21. 窗口配置协议（Window Manifest）

### 21.1 目标

为了让窗口系统快速落地，并保证系统控制力，App 在被打开时应主动提供一个 Manifest / 配置文件，用于描述其窗口需求。

该协议是窗口系统落地的关键前置条件。

### 21.2 初期策略

初期建议优先采用：

> 声明式配置（Manifest）优先，运行时高度自定义能力后置。

原因：

- 更容易统一系统行为
- 更容易先让系统应用跑通
- 减少每个 App 自行处理 Title Bar 的工作量
- 更适合先完成内置 App 的接入

### 21.3 Manifest 至少应描述的内容

建议至少包含：

- 默认打开模式
- 是否允许最小化 / 最大化 / 关闭
- Title Bar 使用系统还是自定义
- 移动端是否覆盖 Dead Zone
- 是否允许在同一 WebView 内嵌入

### 21.4 建议的 Manifest 草案

```json
{
  "window": {
    "defaultMode": "maximized",
    "allowMinimize": true,
    "allowMaximize": true,
    "allowClose": true,
    "allowFullscreen": false
  },
  "titleBar": {
    "mode": "system",
    "title": "Settings",
    "subtitle": "",
    "buttons": ["minimize", "maximize", "close"]
  },
  "mobile": {
    "fullscreenBehavior": "cover_dead_zone"
  },
  "integration": {
    "placement": "inplace"
  }
}
```

说明：

- 该结构仅为需求整理阶段的建议草案
- 最终字段命名可在工程实现阶段统一
- 需要保留版本与兼容升级空间

### 21.5 优先级规则

Manifest 生效逻辑建议为：

1. App 提供 Manifest，则优先使用 App 的配置
2. App 未提供 Manifest，则系统使用默认窗口规则

---

## 22. 移动端 App 接入原则

### 22.1 无 SDK App 的默认处理

在移动端，如果 App 没有接入 BuckyOS SDK，也没有提供窗口配置，则：

- 默认不应在当前 WebView 中 In-place 打开
- 应优先通过独立 WebView 打开

这样可避免：

- 双 Title Bar
- 系统与 App 的导航冲突
- 复杂的适配与兼容问题

### 22.2 SDK App 的能力

如果 App 已接入 SDK，则可以：

- 参与同一 WebView 内的窗口系统
- 使用系统 Title Bar 或声明式配置 Title Bar
- 获得更好的切换体验

### 22.3 核心原因

同一 WebView 内运行的主要价值，在于：

- 切换成本低
- 系统 UI 与 App 结合更紧
- 更适合内置 App 与深度集成 App

## 23. 任务与切换能力（当前边界）

当前阶段虽未定义完整的 Task UI，但窗口系统必须为后续任务切换保留基础语义。

至少要求：

- 最小化窗口仍存在于运行中集合
- 系统未来可以在某个 System UI 区域中展示已打开但未关闭的窗口
- 该区域未来可承载窗口恢复、切换与简单管理能力

具体表现形式，如：

- 底部任务区
- 顶部入口
- Recent Apps 视图

可留待后续版本定义，不应反向影响当前布局与窗口系统抽象。
