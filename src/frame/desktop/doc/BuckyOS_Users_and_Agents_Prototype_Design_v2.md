# BuckyOS Users & Agents 产品原型设计

## 1. 文档说明

- **文档名称**：BuckyOS Users & Agents 产品原型设计
- **产品名称**：Users & Agents
- **技术域名称**：DID 管理 / 实体管理 / 联系人管理
- **文档目标**：基于语音记录与最新结构调整，整理出可用于产品评审、交互设计与原型制作的统一方案
- **适用范围**：BuckyOS 桌面端优先，兼顾移动端后续轻量适配
- **关联模块**：MessageHub、DID Document、Profile、Message Tunnel Binding、权限/分组系统、Agent 运行管理

---

## 2. 背景与产品定位

BuckyOS 需要一个高频使用的“用户与实体管理”组件。它并不是传统意义上单纯的“操作系统用户管理”，也不是单纯的“通讯录”或“IAM 联系人管理”，而是三者的融合：

1. **操作系统用户管理**：支持在当前系统空间内创建可登录的新用户；
2. **联系人与群组管理**：承接用户日常的联系人、导入、分组、整理与聚合；
3. **DID 实体管理**：统一承载用户、Agent、群组等实体的 DID / Profile / Document / Binding 信息。

由于系统底层复用了同一套分组和权限逻辑，产品层不能把这些能力拆得过于“工程化”，否则普通用户会感到理解成本过高。因此，产品表达应更贴近 IM / 通讯录心智，让用户先理解“人、Agent、群、集合”，再逐步接触 DID、Binding、权限等能力。

**对外产品名称建议采用：Users & Agents。**

这个命名既覆盖“用户”与“联系人”，又明确传达 Agent 在系统中与用户并列的重要地位。

---

## 3. 产品目标

### 3.1 目标

本组件需要同时满足以下目标：

- 对普通用户足够友好，看起来像一个易用的“用户 / 联系人 / Agent 管理器”；
- 能承载系统级权限、分组、实体身份与外部通道绑定能力；
- 让“自己”“Agent”“本空间用户”“联系人”“Group”“集合”在同一交互框架中被统一理解；
- 成为 MessageHub、HomeStation 等系统级模块可复用的实体信息入口；
- 为未来的 DID 联合登录、可信绑定、AI 辅助整理预留结构。

### 3.2 设计原则

1. **用通讯录心智承载 DID 能力**  
   用户先看到的是“实体”和“集合”，不是协议细节。

2. **左侧只承载导航，不承载复杂层级树**  
   左侧栏只放“卡片”，不直接堆叠联系人明细项，也不做传统树形权限管理器。

3. **实体与集合分离**  
   实体是可被直接管理的对象；集合是组织和浏览这些对象的视图入口。

4. **高频功能前置，低频复杂功能后置**  
   查看、绑定、导入、基础整理优先；合并、手工清洗等低频能力收敛到二级交互。

5. **与 MessageHub 高度协同，但职责分明**  
   Users & Agents 更偏结构化管理、导入与整理；MessageHub 更偏即时互动、好友请求、单次添加和会话。

6. **安全优先**  
   尤其对于未来“外部一级 DID 登录本系统”的场景，绝不能让用户在当前系统中输入其外部身份体系的原始密码。

## 4. 最新结构决策（本次更新）

本次产品文档的核心更新如下：

### 4.1 左侧栏只保留卡片，不再显示树形联系人层级

左侧栏不再承载联系人树、分组树或混合的一行 / 两行列表。  
**左侧栏只展示两类卡片：**

1. **实体卡片**
2. **集合类卡片**

也就是说，左侧栏是“入口栏”，不是“联系人明细栏”。

### 4.2 左侧栏的固定结构

左侧栏分为上下两段：

#### A. 实体卡片区
用于展示当前用户可以直接管理、直接进入详情的实体，例如：

- 自己（Self）
- 自己的 Agent
- 本空间用户
- 自己创建 / 持有 / Host 的实体群
- 其他需要高权重展示的系统实体

#### B. 集合类卡片区
固定包含以下结构：

- **我的好友**：点击后显示所有个人联系人
- **我加入的 group**：点击后显示所有已加入的实体群
- **手工创建的集合（多个）**：点击后显示该集合内的条目

这里的“我的好友”“我加入的 group”“手工创建的集合”都被定义为：
**集合类卡片（Collection Cards）**。

### 4.3 集合类卡片被点击后的页面结构

当用户点击任意集合类卡片时，主界面切换为三栏结构：

```text
实体列表 ｜ 集合元素列表（单行） ｜ 元素详细信息
```

含义如下：

- **第一栏：实体列表**  
  仍然是左侧栏本身，显示实体卡片区 + 集合类卡片区。

- **第二栏：集合元素列表（单行）**  
  显示当前集合中的全部条目，采用高密度单行列表，用于浏览、搜索、筛选和批量整理。

- **第三栏：元素详细信息**  
  显示第二栏当前选中元素的详细信息，可以是联系人详情、Group 详情、实体详情等。

### 4.4 实体卡片与集合类卡片的模式区别

- **点击实体卡片**：进入该实体的详情视图，主界面保持“左栏 + 详情”的两栏模式；
- **点击集合类卡片**：进入集合浏览模式，主界面切换为“三栏模式”；
- **点击集合元素**：右侧第三栏展示该元素详情，但不会改变左侧导航结构。

这意味着产品有两种稳定工作模式：

1. **实体详情模式（两栏）**
2. **集合浏览模式（三栏）**

## 5. 核心概念模型

### 5.1 实体（Entity）

实体是可以被直接管理、拥有独立详情页、可以承载 DID / Profile / Document / Binding 的对象。

Users & Agents 中需要统一表达以下实体：

#### A. 一级对象（系统首页级对象）
默认至少包含两个：

- **自己（Self）**
- **Agent**

这两个对象都应拥有 DID，并在左侧实体卡片区中以最高权重展示。

#### B. 本空间用户（Local User）
指在当前系统空间内新增、可实际登录系统的用户。  
本期主要通过创建 **二级 DID** 的方式生成。

#### C. 联系人实体 / 联系人条目（Contact）
联系人是用户维护的个人关系对象。它可能是：

- 原生 DID 联系人；
- 从 Telegram / Email 等外部来源同步而来；
- 从 CSV / XML / 手机通讯录导入而来；
- 尚未完成归并的影子联系人 / 候选联系人。

联系人通常不直接出现在左侧栏，而是主要出现在“我的好友”或手工集合打开后的第二栏列表中。

#### D. 实体群（Entity Group）
真实存在、可被 MessageHub 作为消息对象处理的群组；该类对象可拥有自己的 DID、Owner、成员与详情页。

实体群既可能：

- 作为被自己 Host / 创建的实体，出现在左侧实体卡片区；
- 也可能作为已加入对象，出现在“我加入的 group”的集合元素列表中。

---

### 5.2 集合类卡片（Collection Cards）

集合类卡片不是“实体本身”，而是一个组织和查看实体 / 联系人条目的入口。

#### 系统内置集合类卡片

1. **我的好友**  
   用于查看个人联系人全集。

2. **我加入的 group**  
   用于查看当前用户加入的全部实体群。

#### 用户手工创建的集合类卡片

- 用户可以创建多个手工集合；
- 每个集合用于组织一批条目；
- 集合本身更接近“视图对象 / 逻辑对象”，默认不等于一个新的实体群。

#### 集合元素

集合中承载的条目可以是：

- 联系人
- 实体群
- 某些特殊实体
- 后续可扩展的条目类型

### 5.3 集合与 Group 的区别

这是本模块中必须严格澄清的概念：

#### 集合（Collection）
- 主要用于组织、浏览、筛选和整理；
- 是产品层视图结构；
- 默认不是一个新的消息对象；
- 默认不必拥有 DID。

#### 实体群（Entity Group）
- 是真实存在的群体实体；
- 可在 MessageHub 中发消息；
- 可拥有 DID、Owner、成员、Profile；
- 会出现在“我加入的 group”中，也可能作为实体卡片被单独管理。

---

### 5.4 DID 层级

#### 一级 DID
- 独立存在于当前系统空间之外；
- 可理解为全局身份；
- 示例：`Alice`

#### 二级 DID
- 依附于某个系统空间存在；
- 当前空间销毁后，该身份随之消失；
- 常用于本空间内新增用户或 Agent；
- 示例：`Bob.Alice`

### 5.5 Binding

Binding 即 **Message Tunnel Binding**，表示将某个现实世界中的消息通道账号绑定到本系统实体上，例如 Telegram。

Binding 是 MessageHub 变得“有内容”的关键入口。  
没有 Binding，用户只能在本系统内部实体之间通信；有 Binding 后，系统才具备聚合外部消息网络的能力。

### 5.6 Profile 与 DID Document

用户公开信息分为两层：

#### Info（轻量公开资料）
- 存放于标准 Profile 路径；
- 例如头像、公开简介等；
- 修改成本低，立即生效。

#### DID Document（可信身份资料）
- 承载更严肃、可信的身份信息；
- 未来可能涉及链上或空间外写入；
- 修改可能存在手续费或额外确认成本；
- 例如正式姓名、公钥等。

这两层在 UI 上必须明显区分，避免用户把所有资料都当成“随手可改的个人资料”。

## 6. 产品信息架构

### 6.1 页面总体结构

Users & Agents 采用两种稳定页面模式：

#### 模式 A：实体详情模式（两栏）

```text
实体列表 ｜ 实体详细信息
```

适用于用户点击左侧实体卡片时，例如：自己、Agent、本空间用户、自建实体群。

#### 模式 B：集合浏览模式（三栏）

```text
实体列表 ｜ 集合元素列表（单行） ｜ 元素详细信息
```

适用于用户点击左侧集合类卡片时，例如：我的好友、我加入的 group、某个手工创建的集合。

---

### 6.2 第一栏：实体列表（左侧固定导航区）

左侧栏不再使用树形结构，而是固定为卡片导航区。

#### 第一部分：实体卡片区
建议展示：

- 自己（Self）
- Agent
- 本空间用户
- 自己创建 / 持有 / Host 的实体群
- 其他需要被直接管理的系统对象

#### 第二部分：集合类卡片区
固定展示：

- 我的好友
- 我加入的 group
- 手工创建的集合（可多个）

#### 设计要求

- 左侧不直接列出联系人明细项；
- 左侧不展开树形分组层级；
- 左侧只负责“进入某个实体”或“进入某个集合视图”；
- 左侧卡片风格应与 MessageHub 的实体卡片体系保持一致。

---

### 6.3 第二栏：集合元素列表（仅在集合模式下出现）

第二栏用于展示当前集合中的全部条目，采用**单行高密度列表**。

#### 第二栏职责

- 浏览集合内全部元素；
- 搜索当前集合；
- 按来源 / 类型 / 状态筛选；
- 支持排序；
- 支持多选整理；
- 作为批量操作和管理操作的主场景。

#### 第二栏元素类型示例

在不同集合下，第二栏元素含义不同：

- **我的好友**：显示个人联系人
- **我加入的 group**：显示所有已加入的实体群
- **某个手工集合**：显示该集合内的联系人 / Group / 其他条目

#### 第二栏表现规则

- 统一使用单行列表，强调密度和可扫描性；
- 支持来源后缀、状态图标、备注、未整理标识；
- 支持桌面端多选、右键菜单、批量移动、批量删除、批量合并候选等能力。

---

### 6.4 第三栏：元素详细信息

第三栏用于展示第二栏当前选中元素的完整详情。

可能展示的详情类型包括：

- 联系人详情
- 实体群详情
- 用户实体详情
- Agent 详情
- 未来扩展对象详情

#### 设计要求

- 第三栏是信息深度区；
- 所有需要查看资料、Binding、来源、分组、Document、整理动作的内容都在这里承载；
- 第三栏不改变集合浏览上下文，用户始终知道自己当前是从哪个集合进入的。

---

### 6.5 同一对象的多处出现原则

同一个对象可以同时出现在不同位置，这是允许且必要的：

- 某个自建实体群可以出现在左侧实体卡片区；
- 同时它也可以出现在“我加入的 group”的第二栏列表中；
- 某个联系人也可以同时出现在“我的好友”和某个手工集合中。

这里的“多处出现”是视图复用，不是对象复制。

---

## 7. 页面与对象详情设计

### 7.1 自己（Self）详情页

自己是默认最高频对象之一，建议结构如下：

1. **头部信息**
   - 头像
   - 显示名
   - DID
   - 用户类型 / 身份标签

2. **Binding（置顶）**
   - 当前已绑定的 Message Tunnel 账号
   - 支持新增、查看状态、解除绑定
   - 这里是理解 Message Tunnel 的关键入口

3. **Info**
   - 头像
   - 昵称
   - 简介
   - 其他轻量公开资料
   - 修改后立即生效

4. **DID Document**
   - 展示可信身份信息
   - 点击“修改”后进入向导
   - 明确提示：可能产生额外确认或成本

5. **安全与账号**
   - 修改密码
   - 重置密码
   - 查看账号状态

### 设计重点

- Binding 的优先级应高于普通 Profile；
- Document 与 Info 必须分区；
- 修改 Document 不应是“就地编辑”，应通过向导完成。

---

### 7.2 Agent 详情页

Agent 与用户在语义上应被设计为“并列的重要对象”，这是 BuckyOS AI Native 心智的重要入口。

### Agent 与用户的共同点

- 有 DID
- 有 Profile
- 有 DID Document
- 有 Binding

### Agent 与用户的差异点

- **没有密码**
- 需要额外展示运行态信息

### Agent 页面建议分区

1. **头部信息**
   - Agent 名称
   - Logo / Avatar
   - DID
   - 所属关系

2. **Binding**
   - 对 Agent 尤其关键；
   - 决定用户是否能通过外部消息通道与 Agent 交互。

3. **Info**
   - 展示对外描述、能力摘要、头像等

4. **DID Document**
   - 可信身份与配置入口

5. **运行与工作信息**
   - 最近工作日志
   - UI Session / Work Session 数量
   - Workspace 数量
   - 跳转到 Agent 工作目录或专属管理页

---

### 7.3 本空间用户详情页

本空间用户是“真正可登录系统”的用户对象，本期主要通过创建二级 DID 用户实现。

页面建议包含：

- 头像 / 名称 / DID
- 用户类型
- 初始账号信息状态
- 可用应用列表
- 所属默认组
- 密码管理
- Binding（如允许）
- Profile / Document

### 默认组约束

当前系统空间中的所有本空间用户，应默认属于同一个不可移除的基础组。  
原因在于系统内存在共享资源（例如公共共享目录），需要一个稳定的基础权限边界。

---

### 7.4 联系人详情页

联系人详情页主要在集合浏览模式下展示，尤其是从“我的好友”或“手工集合”进入时。

建议包含：

1. **基础身份**
   - 名称
   - 备注
   - 来源标识
   - DID（如有）
   - 是否为双向关系

2. **来源信息**
   - 导入来源
   - 最近同步来源
   - 历史导入批次
   - 创建时间 / 最近更新时间

3. **Binding / Reachability**
   - 此联系人有哪些已知可达通道
   - 例如 DID、Telegram、Email 等
   - 未来可展示为“可达会话列表”

4. **所属集合 / 群组**
   - 属于哪些手工集合
   - 是否在“我的好友”中
   - 是否与某些实体群存在关系

5. **整理操作**
   - 删除
   - 移入 / 移出某个集合
   - 合并到其他联系人
   - 交给 Agent 自动整理（未来）

### 来源后缀展示建议

对于从外部系统同步来的名字，可在视图层使用来源后缀，例如：

- `Bob.telegram`
- `Alice.imported`

其作用不是长期暴露“脏数据”，而是帮助用户理解来源、避免命名冲突，并为后续合并提供依据。

---

### 7.5 实体群详情页

实体群详情页主要通过两种路径进入：

1. 左侧实体卡片区中直接点开自己创建 / Host 的实体群；
2. 从“我加入的 group”的第二栏中选择某个 group。

页面建议内容：

- 群组名称
- 群组类型
- Owner / Host 信息
- 群组 DID（如有）
- 成员列表
- 标签与描述
- 是否可在 MessageHub 发消息
- 与权限系统的关联说明

---

### 7.6 集合本身的视图信息

当用户点击某个集合类卡片时，第二栏顶部应展示该集合的轻量元信息，例如：

- 集合名称
- 元素数量
- 最近更新时间
- 来源类型（手工 / 导入 / 同步）
- 集合级操作入口

对于手工集合，还应支持：

- 重命名集合
- 删除集合
- 批量加入元素
- 批量移出元素

## 8. 关键用户流程

### 8.1 新建本空间用户（主流程）

这是本模块最关键的系统级能力之一。

#### 触发入口
- 左侧实体区顶部“+”
- 或主视图区显式“添加用户”

#### 向导流程

##### 第一步：说明操作性质
必须明确告诉用户：

- 这是在当前系统空间中创建一个“真实可登录”的新用户；
- 该用户会占用系统资源；
- 如果你的目标只是添加联系人，不应该使用这个入口。

##### 第二步：选择用户来源
建议原型中保留两个选项：

1. **创建新用户（本期支持）**
   - 创建二级 DID 用户

2. **使用已有一级 DID 加入（本期预留）**
   - 未来采用联合登录 / 凭证方式
   - 本期可灰置或标注“即将支持”

##### 第三步：填写基础信息
- 用户名
- 用户类型
- 初始密码

##### 第四步：确认应用可用性
本期建议以“信息展示”为主：

- 当前有哪些应用默认对该用户可用；
- 当前版本通常由 Owner 统一安装应用；
- 后续添加用户暂不开放自主安装应用。

##### 第五步：完成创建
- 自动加入默认基础组；
- 完成后可进入该用户详情页继续设置。

---

### 8.2 外部一级 DID 登录（预留流程）

该流程在概念上非常重要，但本期可仅保留入口和交互占位。

#### 目标形态
用户不需要在新系统中重新注册账号，而是使用自己已有的一级 DID 加入某个组织或系统空间。

#### 安全原则
**绝不能要求用户在当前系统输入其外部账号的原始密码。**

理想方案应为：

- 用户先在自己的身份体系中完成认证；
- 由其 DID Document 对应的认证公钥签发凭证（如 JWT）；
- 当前系统只验证凭证，不接触原始密码。

---

### 8.3 修改个人信息

#### Info 修改
- 直接编辑；
- 即时生效；
- 成本低。

#### DID Document 修改
- 进入单独向导；
- 显示风险 / 成本提示；
- 明确告诉用户：这是更正式、可信的身份资料。

---

### 8.4 添加 / 管理 Binding

Binding 是用户首次理解“Message Tunnel”概念的重要入口。

#### 页面要求

- 展示当前支持的 Binding 类型；
- 说明该 Binding 将带来什么能力；
- 清楚展示绑定状态、最近同步时间、异常状态。

#### 价值说明
例如绑定 Telegram 后：

- 可以把 Telegram 中的大量消息与联系人同步进系统；
- MessageHub 将拥有更完整的消息上下文；
- 联系人聚合能力显著提升。

---

### 8.5 联系人导入（核心流程）

导入是联系人系统的主入口，优先级高于手工逐个添加。

#### 原则

- 用户今天不太可能从零开始重建联系人体系；
- 大多数联系人来自历史系统迁移或外部同步；
- 导入过程必须尽量“安全、可理解、不覆盖旧数据”。

#### 导入结果的默认落点
导入完成后，联系人首先进入：

- **我的好友** 的总联系人池；
- 必要时可由用户继续加入某个手工集合；
- 不建议导入后直接把联系人塞进左侧导航区。

#### 支持方式建议

##### 桌面端
- CSV 导入
- XML 导入
- 其他常见通讯录导出格式

##### 移动端（后续）
- 导入系统通讯录
- 从手机通讯录读取手机号等信息

#### 导入后需要处理的问题

- 新导入联系人如何与历史联系人整合；
- 是否自动识别重复项；
- 是否产生“影子联系人”或候选合并项；
- 是否保留来源追踪；
- 如何保证“不会随便覆盖现有资料”。

---

### 8.6 管理“我加入的 group”

“我加入的 group”是系统内置集合类卡片。

点击后，第二栏显示当前用户加入的全部实体群，第三栏显示选中群的详情。

该视图重点承载：

- 查看已加入的群
- 快速定位某个 group
- 识别哪些 group 是自己 Host 的、哪些是外部加入的
- 查看群资料与成员
- 在需要时跳转到 MessageHub 中与该 group 交互

---

### 8.7 手工创建集合与整理

手工集合是用户进行结构化整理的重要能力。

#### 典型操作

- 新建一个集合
- 给集合命名
- 向集合中添加若干联系人 / 群 / 条目
- 从集合中移除条目
- 删除整个集合

#### 适用场景示例

- 某次活动参与人
- 某项目相关联系人
- 某个旅行临时名单
- 某类业务线索集合

手工集合本质上是整理工具，而不是强制要求用户理解权限或实体建模。

---

## 9. 联系人关系与状态设计

### 9.1 DID 联系人的单向 / 双向关系

原生 DID 联系人不是“知道 DID 就自动互为好友”。

#### 单向关系
- 我添加了对方；
- 对方未必添加我；
- 我可以读取其公开资料；
- 我给对方发消息，对方未必能看到或愿意接收。

#### 双向关系
- 双方都已确认彼此；
- 可以放心地双向通信；
- 是更接近传统“好友”的状态。

这一状态模型尤其适合去中心化体系，能够降低垃圾消息问题。

---

### 9.2 外部通道联系人

对于 Telegram、Email 等外部通道联系人，产品逻辑应尽量遵循来源系统本身的关系模型。

例如：

- 如果用户已经绑定 Telegram，那么 Telegram 联系人应主要通过 Telegram 同步得到；
- 不建议在 Users & Agents 中单独提供“手输 Telegram 账号直接加好友”的主入口；
- 但可以允许用户把 Telegram 身份作为单向记录加入联系人体系中。

---

### 9.3 通过手机号 / 邮箱发现 DID（未来能力）

系统未来可以支持“已知手机号 / 邮箱，但不知道 DID”时的发现流程。

#### 基本思路
- 用户本地拥有对方手机号 / 邮箱；
- 系统基于标准规则计算哈希；
- 去公开系统中匹配 DID Document 或相关索引；
- 若匹配成功，自动补全对方 DID。

#### 产品注意事项
- 这是“发现 DID”的能力，不等于自动建立双向关系；
- 需要明确隐私提示；
- 可信度与匹配置信度应可视化。

---

## 10. 与 MessageHub 的关系

Users & Agents 与 MessageHub 高度耦合，但二者职责不同。

### Users & Agents 更关注
- 实体结构
- 集合管理
- 导入与来源
- Binding 管理
- 分组与删除
- 资料展示与整理
- Agent 管理入口

### MessageHub 更关注
- 对话
- 好友请求
- 会话状态
- 消息过滤
- 单次添加与即时互动

### 协同建议
- 左侧实体卡片风格保持一致；
- 用户详情悬浮卡、实体卡片等尽量做成系统通用组件；
- 同一实体在两处的名称、头像、状态应保持一致；
- 用户在 Users & Agents 中看到的是“结构化关系”，在 MessageHub 中看到的是“互动关系”。

---

## 11. 低频管理操作设计

### 11.1 备注
备注是高频能力，但很可能在 MessageHub 中更常发生。  
因此建议将“备注编辑”设计为系统通用组件，Users & Agents 和 MessageHub 共用。

### 11.2 删除
删除是 Users & Agents 中相对合理的高频整理操作，主要发生在集合浏览模式下。

适用场景：

- 清理错误导入；
- 移除无效联系人；
- 删除临时集合中的条目；
- 删除整个手工集合。

### 11.3 调整集合归属
这是 Users & Agents 中最典型的结构化整理行为，建议主要在第二栏列表的多选操作中完成。

### 11.4 手工合并
手工合并是必要但低频、且交互复杂的能力。

#### 桌面端建议方案
优先考虑以下方式：

- 第二栏多选后右键“合并”
- 弹出确认框
- 选择主联系人（保留对象）

这种方式比“拖拽到详情页合并”更符合常见桌面管理心智。

### 11.5 AI 辅助整理
未来可增加“让 Agent 帮我整理”入口，让用户通过自然语言完成联系人清洗、归类、合并、重命名等操作。

- 点击实体卡片进入两栏模式；

## 13. 原型设计建议

### 13.1 首版原型应覆盖的页面

至少包含以下页面 / 状态：

1. Users & Agents 首页（默认实体模式）
2. Self 详情页
3. Agent 详情页
4. 本空间用户详情页
5. 我的好友（三栏模式）
6. 我加入的 group（三栏模式）
7. 手工集合（三栏模式）
8. 联系人详情页
9. 实体群详情页
10. 新建本空间用户向导
11. Binding 管理弹层 / 子页
12. 导入联系人流程
13. 多选合并确认弹层（桌面端）

### 13.2 首版原型必须体现的状态

- 空状态（无联系人、无绑定、无手工集合）
- 导入中 / 导入完成
- 同步来源状态
- DID 单向关系 / 双向关系
- 可编辑的 Info 与需向导修改的 Document
- Agent 已绑定 / 未绑定
- 联系人在多个集合中同时出现
- 某个实体群同时出现在左侧实体卡片区和“我加入的 group”中

### 13.3 视觉层级建议

- **最高层级**：自己、Agent、自己创建 / 持有的关键实体
- **次高层级**：本空间用户、自己 Host 的实体群
- **集合层级**：我的好友、我加入的 group、手工集合
- **元素层级**：集合第二栏中的单行条目
- **辅助层级**：来源、状态、标签、后缀、候选合并提示

## 14. 未来演进方向

### 14.1 安全联合登录
支持已有一级 DID 用户无密码泄露地加入新组织 / 系统空间。

### 14.2 联系人可信绑定公开化
允许用户在 Profile 中公开自己的外部可达方式，并逐步建立可信验证机制。

### 14.3 AI 辅助联系人整理
包括：

- 自动识别重复联系人；
- 推荐合并；
- 推荐加入某个集合；
- 自动清理无效条目；
- 通过自然语言完成整理任务。

### 14.4 手机号 / 邮箱 -> DID 发现
降低 DID 使用门槛，帮助用户从传统联系方式平滑进入 DID 世界。

### 14.5 移动端轻量化管理
移动端优先保留查看、搜索、导入与基础整理，不承载复杂批量操作。

## 15. 关键开放问题

1. 自己创建的实体群，是否默认同时出现在“我加入的 group”中？
2. 集合第二栏的搜索、排序、筛选条应采用内嵌 header 还是悬浮工具栏？
3. 手工集合是否支持嵌套，还是坚持扁平化？
4. 联系人自动合并的阈值、解释性与回滚机制如何定义？
5. 外部 Binding 的可信验证机制如何设计？
6. “添加联系人”入口是否完全迁移到 MessageHub？
7. Document 修改涉及成本时，用户支付与确认路径如何设计？

## 16. 最终产品判断

Users & Agents 不应被设计成一个“后台管理页”，而应被设计成：

- 一个面向普通用户的实体管理中心；
- 一个通过“实体卡片 + 集合类卡片”组织信息的系统级入口；
- 一个将 **用户、Agent、联系人、实体群、集合、DID、Binding** 统一起来的基础组件；
- 一个既服务当前 BuckyOS 使用体验，又为未来 AI Native 身份与协作模型铺路的产品底座。

它的核心价值不在于“把对象列出来”，而在于帮助用户逐步理解：  
**在 BuckyOS 里，用户与 Agent 是实体；好友与群可以通过集合被组织；而系统正在把这些关系转化为可管理、可连接、可协作的统一结构。**
