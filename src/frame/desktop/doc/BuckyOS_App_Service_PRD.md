## 1. 文档目的

本文档用于定义 **BuckyOS App Service** 的产品定位、信息架构、核心交互、安装流程、状态模型与诊断逻辑，作为设计、研发与后续评审的基础版本。

App Service 不是一个“内容生态商店”，而是一个更偏系统层、工程化、保底型的应用服务管理组件。它用于在生态尚未成熟、应用分发形态尚不统一的阶段，承担应用安装、运行观测、基础控制与故障解释的职责。

## 2. 背景与问题定义

### 2.1 已有能力：Library

系统中已经有一个更偏生态型的能力（暂称 **Library**），其目标是承载：

- 内容分发
- 内容购买
- 内容使用
- 通过 Agent / Spider 等机制向生态补充内容与标准化资产

从长期设计目标看，Library 是一个用户完成付费内容网络核心行为的统一入口。

### 2.2 当前现实问题

尽管 Library 的目标完整，但在当前阶段存在以下现实约束：

#### 2.2.1 生态冷启动问题

早期生态内容有限，Library 很可能“空空如也”，无法作为用户进入系统后的主要可用入口。

#### 2.2.2 开发资源与迭代节奏限制

当前版本在该方向上的开发投入有限，而 Library 又依赖若干基础服务，因此其稳定性、完整性与覆盖面不足以承担“系统保底入口”的职责。

#### 2.2.3 内容分发形态不统一

并不是所有应用、内容或服务都适合进入商店式流程，例如：

- 开发中的组件
- 内部测试应用
- 临时分发服务
- 工程师之间直接交付的包
- 仅供开发与联调使用的运行对象

### 2.3 由此引出的产品机会

系统仍需要一个更“冰冷”、更直接、更偏工程化的组件，来承担以下职责：

- 观测系统中的应用层服务运行情况
- 提供更直接的安装入口
- 支持开发 → 安装 → 运行 → 测试 的闭环
- 在生态能力不足时作为系统保底方案

## 3. 产品定位

### 3.1 一句话定义

**App Service 是 BuckyOS 中面向应用层服务的系统级管理组件，用于提供应用安装、运行态观测、基础控制和故障解释能力。**

### 3.2 与 Library 的关系

| 维度 | Library | App Service |
|---|---|---|
| 定位 | 生态/内容网络入口 | 系统级应用服务管理入口 |
| 阶段角色 | 中长期生态核心 | 当前版本保底能力 |
| 主要目标 | 分发、购买、消费、生态增长 | 安装、运行、调试、观测、诊断 |
| 内容来源 | 标准化生态内容 | URL / 本地对象 / 文件等任意来源 |
| 风格 | 运营/生态导向 | 工程/控制台导向 |
| 目标用户 | 普通用户 + 生态参与者 | 开发者、进阶用户、系统使用者 |

### 3.3 设计原则

1. **去生态化**：不承担内容发现、推荐、运营等职责。
2. **工程化优先**：界面风格更接近 `ps` / `docker ps` / 控制台，而非商店。
3. **保底能力优先**：即使 Library 不可用，App Service 也应可独立完成基础安装与运行观测。
4. **直达能力**：用户能快速看到系统在运行什么，快速进入详情并执行控制。
5. **可解释性**：即便不提供所有修复入口，也必须告诉用户为什么应用无法运行。

## 4. 产品目标与非目标

### 4.1 产品目标

本版本 App Service 的目标包括：

1. 展示当前系统中运行中的应用层服务。
2. 提供统一的应用详情页，用于查看状态、配置、版本与运行信息。
3. 提供标准控制操作：启动、停止（重启可选）。
4. 提供直接安装入口，支持 URL / 对象 ID / 上传文件 三类输入。
5. 支持基于安装文档的向导式安装流程。
6. 将安装过程视为系统任务，并在界面中体现“安装中”状态。
7. 在应用异常时提供人类可理解的原因解释，尤其是与 Docker 运行链路相关的问题。

### 4.2 非目标（当前版本不重点承载）

1. 不做商店式内容浏览与推荐。
2. 不要求提供完整的 Docker 管理入口。
3. 不承担内核配置能力。
4. 不要求覆盖复杂的应用发布/分发市场能力。
5. 不以日志分析平台为目标（日志能力可后续评估）。

## 5. 目标用户

### 5.1 普通用户

主要诉求：

- 看到当前运行中的应用
- 知道应用是否正常工作
- 在必要时对应用执行开始/停止操作
- 在应用出问题时获得可理解的原因说明

### 5.2 开发者 / 早期集成者

主要诉求：

- 快速安装开发中的应用
- 通过 URL / 本地对象 / 文件直接接入应用
- 完成开发 → 安装 → 测试的闭环
- 快速判断失败是出在镜像、引擎、配置还是安装资源

### 5.3 系统维护者 / 进阶用户

主要诉求：

- 从统一入口看到应用层、系统服务层、内核层的运行结构
- 判断故障是应用自身问题还是基础运行环境问题

## 7. 信息架构

### 7.1 首页信息层级

首页的核心目标是：**展示当前系统“正在运行的一切”**。

系统对象按三层结构展示：

1. **应用层（App Services）**
2. **系统服务层（System Services）**
3. **内核层（Kernel）**

### 7.2 排序与视觉权重

- 默认首先展示 **应用层**
- 其次展示 **系统服务层**
- 最后展示 **内核层**

这样做的原因是：

- 对普通用户来说，“应用”是最自然的认知层。
- 系统完整结构仍然存在，但不会在视觉上过于突兀。
- 内核层维持更“冰冷”的表达方式，减少对普通用户的干扰。

### 7.3 展示形式建议

| 层级 | 展示形式 | 说明 |
|---|---|---|
| 应用层 | 卡片列表 | 有图标、名称、状态、简介 |
| 系统服务层 | 简化列表或次级卡片 | 关注服务名称与状态 |
| 内核层 | 极简文本/列表 | 强调只读与低干预 |

## 8. 首页设计

### 8.1 页面目标

首页是一个运行态总览页，重点不是展示“已安装的所有东西”，而是展示“当前在运行的东西”。

其设计逻辑接近：

- `ps`
- `top`
- `docker ps`

但对普通用户更友好，优先可视化应用层对象。

### 8.2 页面元素

#### 8.2.1 顶部区域

- 页面标题：App Service
- 安装入口：“+” 按钮

#### 8.2.2 主体区域

按三层分组展示运行对象：

```text
Running Apps
System Services
Kernel
```

### 8.3 App Card（应用卡片）

应用层对象以卡片展示，建议包含：

- 应用图标
- 应用名称
- 简短描述（如有）
- 当前状态

### 8.4 应用状态（首页可见）

至少支持以下状态表达：

- Running / 运行中
- Starting / 启动中
- Stopped / 已停止
- Error / 异常
- Installing / 安装中

### 8.5 首页核心交互

点击任意运行中的应用卡片，直接进入该应用详情页。

设计意图：

- 快速到达详情
- 减少层级
- 让首页真正成为“系统运行态入口”

## 9. 应用详情页设计

### 9.1 页面目标

应用详情页用于承载单个应用的：

- 状态查看
- 标准控制操作
- 安装配置查看
- 自定义设置查看
- 版本与运行环境信息查看
- 异常原因解释

### 9.2 页面结构建议

#### 9.2.1 Header

- 图标
- 名称
- 状态
- 快速操作按钮（Start / Stop / Restart）

#### 9.2.2 Status Overview

展示应用当前状态及关键依赖状态。

#### 9.2.3 Configuration

展示：

- 安装配置（Spec）
- 自定义设置（Settings）

#### 9.2.4 Runtime Info

展示：

- 应用版本
- Docker 镜像名
- 容器状态
- 其他与运行时相关的元信息

#### 9.2.5 Diagnostics

展示应用无法运行或状态异常的原因解释。

### 9.3 标准控制操作

当前版本至少支持：

- Start
- Stop

可选支持：

- Restart

要求：

- 操作后有即时状态反馈
- 操作结果在详情页中可见

### 9.4 配置查看能力

#### 9.4.1 Spec

用户可统一查看安装时使用的配置/安装描述信息。

典型内容包括：

- 依赖信息
- 镜像信息
- 运行方式
- 权限需求

#### 9.4.2 Settings

展示应用自身的自定义设置项。

> 注：当前口述需求主要强调“查看”，是否支持编辑尚未明确，暂记为 TBD。

### 9.5 运行信息查看能力

详情页应至少展示以下信息：

- 应用版本
- Docker Image Name
- Docker Container 状态
- 与当前应用运行相关的关键元数据

## 10. Docker 运行依赖模型

由于很多应用基于 Docker，App Service 需要能够解释以下依赖关系：

```text
App
  ↓
Container
  ↓
Image
  ↓
Docker Engine
```

### 10.1 需要暴露给用户的关键状态

#### 10.1.1 Docker Engine 是否工作

如果 Docker Engine 未运行，则应用不工作是合理结果。用户需要在 App Service 中看到这一解释。

#### 10.1.2 应用所需 Image 是否存在

存在“应用已安装，但镜像未拉取成功”的情况。此时应用无法启动，系统应明确指出是 **image missing**，而不是仅笼统显示失败。

#### 10.1.3 Container 是否已创建/已运行

如果镜像存在但容器启动失败，系统也应给出相应诊断。

### 10.2 运行态依赖状态建议展示方式

在详情页中建议统一展示：

- Docker Engine: Running / Not Running
- Image: Present / Missing
- Container: Running / Stopped / Error

### 10.3 设计要求

App Service 不一定需要为 Docker 问题提供完整修复入口，但必须具备：

- **观测能力**
- **原因解释能力**

## 11. 诊断与错误表达

### 11.1 目标

系统在应用不可运行时，不应只展示技术性失败状态，而应尽可能输出用户可理解的原因。

### 11.2 最低支持错误类型

- Docker 引擎未运行
- Docker 镜像不存在 / 拉取失败
- 容器启动失败
- 安装文档无法解析
- 安装资源下载失败

### 11.3 表达形式

推荐以“原因描述”为主，而不是直接暴露大量底层日志。

示例：

- Docker 引擎未运行，应用无法启动
- 应用镜像尚未下载完成，因此当前无法运行
- 容器启动失败，请检查安装配置或运行依赖

> 注：更详细日志是否在后续版本提供，当前记为 TBD。

## 12. 安装入口

### 12.1 入口位置

在 App Service 首页提供右上角 **“+” 按钮** 作为统一安装入口。

### 12.2 入口目标

安装入口强调的是：

> 将一个已经存在的应用来源，快速接入系统。

它不是“商店浏览入口”，而是“工程交付入口”。

---

## 13. 安装来源

安装向导第一步需要支持三类输入来源。

### 13.1 URL

用户输入一个 URL，系统按既定安装协议拉取对应安装描述或资源。

适用场景：

- 开发者直接分享安装链接
- 内部分发
- 非商店化交付

### 13.2 对象 ID

用户输入或粘贴一个对象 ID，用于安装本地已存在的对象。

典型来源包括：

- 本机已存在资源
- 通过 MessageHub 等渠道收到的对象引用
- 其他方式同步到本地但尚未安装的对象

为避免混淆，产品层建议统一称为：

- **对象 ID 安装**

### 13.3 上传文件

用户可以拖拽或上传本地文件完成安装。

候选文件形式包括：

- `.pkg`
- `meta.json`
- 系统定义的其他安装文件格式

其中 `meta.json` 可作为轻量安装描述文件，典型可能包含：

- Docker 镜像信息
- 权限声明
- 基础元数据

## 14. 安装向导流程

安装采用向导式流程。

### 14.1 Step 1：选择输入方式并提供安装源

用户选择：

- URL
- 对象 ID
- 上传文件

并填写/拖入对应内容。

### 14.2 Step 2：解析安装文档

系统对输入源执行解析，读取应用描述文档（App Document）。

解析目标包括：

- 应用基础信息
- 安装元信息
- 权限声明
- 依赖信息
- 运行相关字段

若解析失败，则在当前步骤直接报错并终止安装。

### 14.3 Step 3：展示应用信息与权限控制

当文档解析成功后，系统展示：

#### 14.3.1 应用基础信息

- 图标
- 名称
- 版本
- 描述
- 来源（如可识别）

#### 14.3.2 权限与资源信息

- 所需权限
- 目录访问范围
- 网络或其他能力声明
- 系统资源访问要求

此步骤的作用是让用户在安装前完成安全确认。

### 14.4 Step 4：管理员确认

安装属于敏感操作，在确认安装前要求用户输入：

- 管理员密码

该步骤用于：

- 确认具备安装权限
- 与系统整体权限模型保持一致

### 14.5 Step 5：执行安装

用户确认后，系统开始执行安装流程。执行阶段可能包括：

- 下载资源
- 拉取镜像
- 写入安装记录
- 注册应用服务
- 准备运行环境

## 15. 安装任务与状态流转

### 15.1 安装是一个任务

安装不应被视为瞬时动作，而应视为系统任务（Task）。

### 15.2 在 App Service 中的表现

安装开始后：

- App Service 中出现应用“安装中”状态
- 系统任务管理器中出现对应安装任务

用户在 App Service 中知道“正在安装”，如需更详细进度，可前往任务管理器查看。

### 15.3 状态流转

推荐状态流转如下：

```text
未安装
→ 安装中
→ 安装成功
→ 运行中
```

若安装完成后默认自动启动，也可表现为：

```text
未安装
→ 安装中
→ 运行中
```

### 15.4 安装结果反馈

安装成功后，首页中对应应用卡片从“安装中”切换为“运行中”。

---

## 16. 安装失败处理

### 16.1 解析阶段失败

例如：

- URL 无效
- 对象 ID 无法解析
- 上传文件格式错误
- App Document 非法

处理方式：

- 在向导内直接提示
- 不进入执行安装阶段

### 16.2 执行阶段失败

例如：

- 下载失败
- Docker Pull 失败
- 权限确认后执行中断
- 运行环境准备失败

处理方式：

- App Service 中显示安装失败或不可运行状态
- 任务管理器中可查看更详细失败信息

---

## 17. 交互原则总结

1. **首页纯粹**：用户一进来先看到系统正在运行什么。
2. **详情直达**：点卡片即可进入详情页，无额外层级。
3. **安装直接**：一个“+”就能进入安装流程。
4. **解释优先**：异常时至少要知道问题出在哪里。
5. **任务解耦**：安装进度的细粒度查看交给任务系统，App Service 只承担核心状态表达。
6. **工程风格**：这个组件不是商店，而是控制面板。

## 19. 功能需求清单（Functional Requirements）

### FR-01 首页运行态展示

系统应展示当前运行对象，并按“应用 / 系统服务 / 内核”三层分组。

### FR-02 应用优先展示

系统应将应用层对象置于默认首屏位置，并以更友好的卡片形式呈现。

### FR-03 详情直达

点击应用卡片后，系统应直接进入该应用详情页。

### FR-04 详情页状态展示

系统应在详情页展示应用状态、安装配置、自定义设置、版本与运行环境信息。

### FR-05 标准控制操作

系统应支持对应用层服务执行启动与停止操作。

### FR-06 安装入口

系统应在首页提供统一“+”安装入口。

### FR-07 多来源安装

系统应支持通过 URL、对象 ID、上传文件三种方式安装应用。

### FR-08 安装文档解析

系统应在安装前解析 App Document，并提取应用信息、权限与依赖信息。

### FR-09 权限确认

系统应在安装前展示权限申请与资源访问说明。

### FR-10 管理员鉴权

系统应在开始安装前要求输入管理员密码。

### FR-11 安装任务化

系统应将安装流程作为系统任务执行，并在 App Service 中显示安装中状态。

### FR-12 异常原因解释

系统应在应用无法正常运行时，向用户呈现可理解的原因描述。

### FR-13 Docker 依赖状态可见

系统应显示 Docker Engine、Image 与 Container 的关键状态。

---

## 20. 非功能要求（Non-functional Requirements）

### 20.1 可理解性

错误状态应尽量以自然语言表达，降低普通用户理解门槛。

### 20.2 即时反馈

应用控制操作与安装状态变化应有明确反馈。

### 20.3 一致性

安装流程、状态命名与详情页字段应保持一致，避免同一概念在不同页面用不同称呼。

### 20.4 可扩展性

后续应能平滑扩展：

- 更新
- 卸载
- 日志
- 更详细运行指标
- 与 Library 的联动

## 21. 依赖与前置条件

App Service 的部分能力依赖：

1. 安装协议与 App Document 格式定义
2. 系统任务管理器
3. 管理员鉴权机制
4. Docker 运行环境与状态查询接口
5. 应用/服务运行状态查询接口

## 22. 风险与注意事项

### 22.1 状态定义需统一

“安装中 / 已安装 / 运行中 / 异常 / 不可运行”等状态需要在产品、前端、后端之间统一定义。

### 22.2 Docker 异常原因需收敛

底层错误很多，但产品层需要收敛为少数可解释类别，否则用户会被技术细节淹没。

### 22.3 普通用户与开发者视角需平衡

如果表达过于底层，会影响普通用户理解；如果过于抽象，则开发者无法有效调试。需在信息默认层级上做好平衡。

## 25. 附：产品总结

BuckyOS App Service 在当前版本中承担的是一个非常明确的角色：

> **它不是生态，不是商店，而是系统应用层服务的控制台。**

它需要在 Library 尚未成熟的阶段，为系统提供最基础、最可靠、最直接的应用接入与运行管理能力；同时，也要为开发者提供一个足够顺手的开发测试闭环入口。
