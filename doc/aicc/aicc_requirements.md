# AICC 需求文档

## 1. 文档目的

本文整理 AICC 当前设计文档和现有能力后形成产品需求说明，只描述 AICC 应满足的用户价值、功能边界和验收口径，不描述内部实现方案。

## 2. 产品定位

AICC 是 BuckyOS 中统一的 AI 计算中心。它面向系统服务、应用和 Agent 提供一致的 AI 能力使用体验，让调用方不需要关心不同 AI 服务来源之间的差异。

AICC 的核心价值是：

1. 统一接入文本、图像、语音、视频、视觉理解和 Agent 辅助类 AI 能力。
2. 根据任务需求、用户偏好、隐私要求、成本和可用性自动选择合适的 AI 服务来源。
3. 对耗时任务提供可追踪、可取消、可审计的执行体验。
4. 让管理员和用户能够管理 AI 服务来源、模型用途偏好、预算和隐私策略。

## 3. 目标用户

### 3.1 应用开发者

应用开发者希望用稳定、统一的方式调用 AI 能力，而不是分别适配每个模型或服务商。开发者需要关注任务类型、输入输出和业务约束，而不需要关注底层服务差异。

### 3.2 Agent 开发者

Agent 开发者需要为规划、推理、代码、总结、视觉理解、语音处理等不同角色选择合适模型，并希望同一会话内的行为保持稳定。

### 3.3 普通用户

普通用户希望 AI 能力可用、稳定、成本可控，并能在本地优先、云端优先、质量优先、速度优先等偏好之间做选择。

### 3.4 系统管理员

系统管理员需要统一配置可用的 AI 服务来源，控制密钥、额度、预算、隐私策略、可用性和故障处理。

## 4. 范围

### 4.1 包含范围

1. AI 能力统一入口。
2. 多种 AI 服务来源的统一管理。
3. 按任务用途选择模型。
4. 成本、速度、质量、隐私和可用性策略。
5. 耗时任务的状态、进度、取消和结果管理。
6. 使用量、成本和审计记录。
7. 面向用户和开发者的模型选择说明。
8. 正式环境和远程调用场景的验收。

### 4.2 不包含范围

1. 不提供独立的文件存储产品。
2. 不替代系统已有的任务、权限、登录、事件和资源管理能力。
3. 不负责训练或微调模型。
4. 不承诺所有 AI 服务来源都具备完全相同的功能。
5. 不在本文中规定内部落地方案。

## 5. 核心使用场景

### 5.1 应用调用通用 AI 能力

应用可以提交一次文本、图像、语音或视频相关请求，并获得结构一致的结果。应用只需要表达任务目标和约束，例如“生成图片”“识别图片文字”“总结音频”“生成视频”“进行对话”。

### 5.2 Agent 执行复杂任务

Agent 可以把一个复杂目标拆成多个 AI 步骤，例如规划、搜索、图片理解、文案生成、结果总结。AICC 需要保证这些步骤能使用合适的模型，并让同一任务上下文中的模型选择保持可预期。

### 5.3 用户选择隐私优先

用户可以要求某类任务只使用本地可信能力。若本地能力不可用，AICC 必须明确失败或提示原因，不能自动转交给云端能力。

### 5.4 管理员接入多个 AI 服务来源

管理员可以配置多个云端、本地或代理型 AI 服务来源。AICC 需要展示可用能力、健康状态、模型用途和费用信息，并在服务异常时自动避开不可用来源。

### 5.5 长时间 AI 任务

图片生成、视频生成、音乐生成、长音频转写、大批量文本处理等任务可能无法立即完成。AICC 需要返回任务状态，让调用方查询进度、取消任务，并在完成后获取结果。

## 6. 功能需求

### 6.1 AI 能力覆盖

AICC 应支持以下能力类别：

| 类别 | 需求 |
|---|---|
| 文本对话 | 支持多轮对话、结构化回答、工具调用意图、总结、规划、代码、推理等用途。 |
| 文本补全 | 支持传统文本续写场景，并鼓励新场景使用对话能力。 |
| 文本向量化 | 支持将文本或多模态内容转为可检索、可比较的表示。 |
| 重排序 | 支持对候选文本、文档或结果进行相关性排序。 |
| 图像生成 | 支持从文本生成图像。 |
| 图像编辑 | 支持图生图、局部重绘、超分辨率、去背景等处理。 |
| 视觉理解 | 支持图片文字识别、图片描述、目标检测、区域分割等任务。 |
| 语音处理 | 支持文字转语音、语音转文字、音乐生成、音频增强等任务。 |
| 视频处理 | 支持文本生成视频、图像生成视频、视频转换、视频延展、视频超分辨率等任务。 |
| Agent 辅助 | 支持 Agent 需要的电脑使用、规划、执行辅助等能力。 |

### 6.2 模型用途分类

AICC 应提供稳定的模型用途分类，让应用和用户可以按用途选择模型，而不是必须记住具体模型名称。

必须支持的用途包括：

1. 通用对话。
2. 任务规划。
3. 代码生成与理解。
4. 快速轻量回答。
5. 深度推理。
6. 视觉理解。
7. 长上下文处理。
8. 兜底可用能力。
9. 图像、语音、视频各自的常见任务用途。

当某个具体用途没有可用能力时，AICC 可以按管理员或用户允许的规则切换到更通用的能力；若策略要求严格匹配，则必须明确失败。

### 6.3 AI 服务来源管理

AICC 应允许系统接入多个 AI 服务来源，并对每个来源维护以下可见信息：

1. 支持哪些 AI 能力。
2. 支持哪些模型或模型家族。
3. 是否为本地能力、云端能力或可信代理能力。
4. 当前是否可用。
5. 大致成本、免费额度、包月或计费模式。
6. 质量、速度、上下文长度、多模态能力等用户可理解属性。

服务来源发生变化时，AICC 应能更新可用能力列表，不要求用户重启整个系统。

### 6.4 自动选择与切换

AICC 在选择 AI 服务来源时，应综合以下因素：

1. 任务类型是否匹配。
2. 用户或应用指定的模型用途是否匹配。
3. 隐私策略是否允许。
4. 预算或单次成本上限是否允许。
5. 用户偏好的质量、速度、成本优先级。
6. 当前服务来源是否健康。
7. 同一会话中是否需要保持一致体验。

当首选服务来源不可用时，AICC 可以在策略允许范围内自动切换到备用来源。切换必须遵守隐私和预算限制，不得为了成功率绕过用户明确约束。

### 6.5 会话一致性

对于连续对话、Agent 工作流或同一用户任务，AICC 应尽量保持模型选择一致，避免同一会话中频繁切换导致风格、能力或结果不稳定。

当原选择不可用、超出预算、违反新策略或健康状态异常时，AICC 可以重新选择，并应给出可理解的原因。

### 6.6 耗时任务管理

AICC 对耗时任务应支持：

1. 返回可识别的任务编号。
2. 查询任务当前状态。
3. 查询进度或阶段信息。
4. 获取最终结果。
5. 取消正在执行的任务。
6. 记录失败原因。
7. 防止重复提交导致重复收费或重复生成结果。

短任务应尽量直接返回结果；耗时任务应进入后台处理。

### 6.7 输入与输出资源

AICC 应支持文本、图片、音频、视频和结构化内容作为输入，也应支持这些类型作为输出。

资源处理应满足：

1. 大文件可以复用系统已有资源，不要求反复上传。
2. 临时小资源可以直接提交，但必须有大小和类型限制。
3. 输出结果应便于应用继续使用，例如用于后续图片编辑、视频处理或 Agent 工作流。
4. 日志和审计记录不得保存原始敏感内容。

### 6.8 结构化结果

AICC 应尽量把不同 AI 服务来源的结果整理成一致格式，使调用方能稳定读取：

1. 文本结果。
2. 结构化数据结果。
3. 图片、音频、视频等生成物。
4. 工具调用意图。
5. 结束原因。
6. 使用量和成本摘要。
7. 错误原因。

当某个服务来源返回额外信息时，AICC 可以保留摘要，但必须避免暴露密钥、原始敏感输入和过大的原始响应。

### 6.9 使用量、成本和预算

AICC 应记录每次成功完成的 AI 使用情况，用于后续账单、统计、预算控制和审计。

使用记录应至少支持：

1. 用户和应用维度统计。
2. 任务类型维度统计。
3. 模型用途和实际服务来源维度统计。
4. 输入、输出和总使用量统计。
5. 估算成本和实际成本。
6. 免费额度、包月额度和付费额度的区分。
7. 去重，避免同一次任务被重复计费。

预算策略应支持：

1. 单次任务成本上限。
2. 用户、应用或任务类型的额度限制。
3. 额度不足时拒绝或改选更低成本能力。
4. 向用户解释因预算导致的选择变化或失败原因。

### 6.10 隐私与安全

AICC 必须把隐私策略作为硬性要求。

必须满足：

1. 本地优先或仅本地任务不得自动切换到云端能力。
2. 高隐私任务不得交给未知可信级别的服务来源。
3. 用户级策略不得覆盖更高优先级的组织或系统安全策略。
4. 密钥、访问凭据、原始提示词、原始文件内容不得进入普通日志。
5. 取消、查询和结果访问必须遵守用户和应用权限。
6. 跨用户或跨租户任务不得互相查看或取消。
7. 对服务来源可信级别的判断必须来自系统可信设置，不能只依赖服务来源自我声明。

### 6.11 可解释性

AICC 应能向开发者、管理员和必要时向用户解释一次 AI 请求为什么选择了某个服务来源。

说明内容应包括：

1. 最终选择的模型用途或模型家族。
2. 选择依据，例如成本、速度、质量、可用性或隐私限制。
3. 是否发生自动切换。
4. 被排除的主要原因，例如能力不匹配、超预算、不可用、隐私策略不允许。
5. 不包含敏感输入内容。

用户界面应展示简明解释；开发和排障场景可以展示更详细说明。

### 6.12 管理与配置

管理员应能完成以下操作：

1. 添加、启用、禁用 AI 服务来源。
2. 设置密钥和访问凭据。
3. 设置默认模型用途偏好。
4. 设置用户、应用或全局级别的成本、速度、质量偏好。
5. 设置隐私策略。
6. 查看可用能力、健康状态和异常原因。
7. 触发配置重新生效。
8. 查看使用量和成本统计。

普通用户应能在授权范围内完成以下操作：

1. 选择偏好的模型用途。
2. 选择本地优先、速度优先、质量优先或成本优先。
3. 查看自己可用的 AI 能力和使用情况。
4. 理解任务失败或被拒绝的原因。

## 7. 非功能需求

### 7.1 可用性

1. 单个 AI 服务来源故障不应导致整个 AI 能力不可用。
2. 可自动切换的任务应尽量使用备用来源完成。
3. 不可自动切换的任务必须给出明确原因。
4. 服务来源健康状态异常时，应减少或停止使用该来源。

### 7.2 性能

1. 普通短任务应尽量低延迟返回。
2. 选择模型和服务来源的过程不应成为主要耗时瓶颈。
3. 耗时任务不应阻塞调用方继续执行其他操作。
4. 高并发场景下应保持稳定的任务状态和选择结果。

### 7.3 稳定性

1. 重复请求不应造成不可控的重复执行或重复收费。
2. 系统重启或服务来源刷新后，应尽量保持已有任务可追踪。
3. 任务状态变化应完整记录，便于恢复和排障。

### 7.4 可维护性

1. 新增 AI 服务来源不应影响已有应用的调用方式。
2. 新增模型或能力时，应优先纳入已有用途分类。
3. 旧模型名称或旧用途名称应提供迁移提示。
4. 能力差异应通过清晰的可见信息表达，而不是让调用方试错。

### 7.5 合规与审计

1. 使用记录应支持审计和问题追踪。
2. 审计记录不应包含原始敏感内容。
3. 用户应能区分本地、云端和代理型服务来源。
4. 管理员应能追踪隐私策略、预算策略和服务来源变更对任务结果的影响。

## 8. 验收标准

### 8.1 基础能力验收

1. 文本对话任务可以成功返回结果。
2. 图片生成任务可以成功返回可复用图片结果。
3. 图片理解任务可以成功返回描述、识别或检测结果。
4. 语音转文字和文字转语音任务至少有一个可用服务来源时可以成功执行。
5. 视频类耗时任务可以进入后台处理并最终获取结果。
6. 模型用途列表可展示给用户或开发者。

### 8.2 选择与切换验收

1. 同一用途存在多个可用服务来源时，AICC 能按成本、速度、质量和健康状态选择。
2. 首选来源不可用且策略允许时，AICC 能切换到备用来源。
3. 用户要求仅本地时，AICC 不会切换到云端来源。
4. 用户指定严格模型或严格用途时，不满足条件必须明确失败。
5. 同一会话内连续请求应尽量保持选择一致。

### 8.3 任务管理验收

1. 耗时任务能返回任务编号。
2. 调用方能查询任务状态和最终结果。
3. 调用方能取消自己有权限取消的任务。
4. 无权限用户不能查看或取消他人的任务。
5. 失败任务能返回可理解原因。

### 8.4 使用量与成本验收

1. 成功任务会生成使用记录。
2. 重复记录不会导致重复计费。
3. 用户、应用、任务类型维度可以查询统计。
4. 单次成本上限和额度限制能影响任务选择或拒绝任务。

### 8.5 安全与隐私验收

1. 日志和审计记录不包含密钥、原始提示词和原始文件内容。
2. 高隐私任务不会交给未知可信级别来源。
3. 跨用户、跨租户访问会被拒绝。
4. 选择说明中不暴露敏感输入。

### 8.6 运维验收

1. 管理员可以查看已配置服务来源和健康状态。
2. 服务来源配置变更后可以重新生效。
3. 服务来源不可用时有明确诊断信息。
4. 正式环境远程调用可以完成基础文本、多模态和耗时任务用例。

## 9. 优先级

### 9.1 P0

1. 文本对话、图像生成、视觉理解、语音处理和基础视频任务。
2. 多服务来源选择和故障切换。
3. 耗时任务状态、结果和取消。
4. 本地优先和仅本地隐私策略。
5. 使用量与成本记录。
6. 管理员配置服务来源。
7. 简明选择说明和错误原因。

### 9.2 P1

1. 更完整的图像编辑、音频增强、音乐生成和视频处理能力。
2. 用户和应用级模型用途偏好。
3. 更精细的预算、额度和包月策略。
4. 服务来源健康状态自动恢复。
5. 用户界面的模型用途树和能力说明。
6. Agent 多角色模型偏好管理。

### 9.3 P2

1. 根据历史效果自动优化选择策略。
2. 支持受控实验比较不同服务来源效果。
3. 根据用户长期偏好自动调整模型选择。
4. 更细粒度的成本与质量多目标优化。

## 10. 风险与约束

| 风险 | 影响 | 需求约束 |
|---|---|---|
| 服务来源能力差异大 | 同一任务在不同来源上的结果质量不一致。 | 必须清楚展示能力范围和选择原因。 |
| 自动切换过宽 | 用户指定用途后得到不符合预期的模型。 | 严格用途和高隐私策略必须允许用户锁定。 |
| 成本估算不准 | 可能超出用户预算。 | 必须记录估算与实际使用，并允许设置上限。 |
| 服务来源动态变化 | 模型下线或异常导致任务失败。 | 必须有健康状态、备用来源和明确错误说明。 |
| 隐私策略误判 | 敏感任务可能流向不可信来源。 | 可信级别必须由系统可信设置确认。 |
| 解释信息泄露 | 排障信息可能暴露敏感内容。 | 选择说明只能包含必要摘要，不包含原始输入和凭据。 |
