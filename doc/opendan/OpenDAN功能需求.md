# OpenDAN 功能需求文档

## 1. 模块定位

OpenDAN 是 BuckyOS 的 Agent Runtime，用于承载用户默认 Agent、用户安装的 Agent 以及由 Agent 派生出的子 Agent。OpenDAN 负责接收用户消息和系统事件，驱动 Agent 按会话持续工作，并为 Agent 提供记忆、技能、工具、工作空间、任务记录和运行观测能力。

OpenDAN 的目标不是提供单一聊天接口，而是提供一个可持续执行任务、可追踪、可管理、可扩展的 Agent 运行环境。

## 2. 目标用户

- 普通用户：通过默认 Agent 完成问答、任务规划、提醒、文件处理和个人事务协助。
- 高级用户：为 Agent 绑定工作空间、安装技能、配置行为模式，并查看任务执行过程。
- 系统管理界面：查询 Agent 状态、会话、Todo、Worklog 和子 Agent 信息。
- 其他系统服务：向 Agent 投递消息、事件或任务，并接收 Agent 的处理结果。
- Agent 开发者：定义 Agent 行为、技能、工具使用边界和任务执行流程。

## 3. 核心概念

### 3.1 Agent

Agent 是 OpenDAN 中承担任务执行的主体。每个 Agent 应具备身份、名称、描述、运行状态、会话集合、工作空间、记忆和可用工具。

系统应至少支持一个默认 Agent。默认 Agent 用于承接用户的大部分日常交互，并可根据任务需要创建或协作子 Agent。

### 3.2 Session

Session 是 Agent 与用户、系统事件或任务之间的持续交互上下文。一个 Session 应保存当前任务状态、输入历史、当前行为模式、工作空间绑定、已加载技能、Todo、Worklog 和必要的运行元数据。

Session 应支持等待输入、等待事件、准备执行、正在执行、结束、暂停和恢复等状态。

### 3.3 Behavior

Behavior 描述 Agent 在某类任务中的行为方式。不同 Behavior 可用于消息路由、任务规划、执行、自检、总结、等待外部事件等场景。

OpenDAN 应允许不同 Session 使用不同 Behavior，并允许 Agent 在任务过程中切换 Behavior。

### 3.4 Skill

Skill 是可被 Agent 加载的能力描述或提示词组件。Skill 应可作用于当前 Behavior，也可作用于整个 Session。

Skill 用于让 Agent 在特定任务中获得更明确的操作规范、领域知识或流程约束。

### 3.5 Workspace

Workspace 是 Agent 执行任务时使用的工作上下文。它可以是 OpenDAN 管理的内部工作空间，也可以是用户授权绑定的外部目录。

Workspace 应用于保存任务产物、临时文件、上下文资料、Todo 和 Worklog。

### 3.6 Todo

Todo 是 Agent 对任务拆解出的待办项。Todo 应支持记录标题、描述、状态、优先级、关联 Session、关联工作空间和执行结果。

Todo 用于让复杂任务可分解、可追踪、可恢复。

### 3.7 Worklog

Worklog 是 Agent 工作过程的结构化记录。它应记录输入、输出、工具调用、关键决策、步骤总结、任务状态变化和异常信息。

Worklog 用于让用户和系统理解 Agent 做了什么、为什么做、做到哪一步。

### 3.8 Memory

Memory 是 Agent 的长期记忆能力。它应支持保存用户偏好、长期事实、提醒事项、任务背景和跨会话复用的信息。

Memory 应可被 Agent 检索、更新和删除，并可用于主动触发自检或提醒类任务。

## 4. 功能需求

### 4.1 Agent 管理

OpenDAN 应支持管理 Agent 的基本信息和运行状态。

需求：

- 系统应支持查询当前可用的 Agent 列表。
- 系统应支持查询单个 Agent 的详细信息。
- 系统应支持区分主 Agent 和子 Agent。
- 系统应支持展示 Agent 的当前状态，例如空闲、运行中或不可用。
- 系统应支持展示 Agent 最近活跃时间。
- 系统应支持展示 Agent 所属工作空间信息。
- 系统应支持查询 Agent 的子 Agent 列表。

验收标准：

- 管理端可以列出所有主 Agent。
- 管理端可以选择是否包含子 Agent。
- 用户可以看到每个 Agent 的名称、类型、状态、工作空间和最近活动时间。

### 4.2 Agent 启动与默认 Agent

OpenDAN 应支持默认 Agent 的自动启动和运行。

需求：

- 系统激活后，应能为用户提供默认 Agent 能力。
- 默认 Agent 应能接收用户消息。
- 默认 Agent 应具备基础任务处理、对话、规划和工具使用能力。
- 默认 Agent 应能在缺少部分非核心依赖时保持基础可用。

验收标准：

- 用户完成 BuckyOS 基础初始化后，可以向默认 Agent 发送消息。
- 默认 Agent 可以回复用户并维护会话上下文。
- 当部分非关键依赖暂时不可用时，系统应给出可理解的降级表现。

### 4.3 消息接收与路由

OpenDAN 应支持 Agent 接收用户消息、群组消息、请求消息和系统事件。

需求：

- Agent 应能接收发给自己的用户消息。
- Agent 应能识别消息所属的会话。
- 当消息没有明确会话时，系统应能为其分配默认会话。
- Agent 应能接收系统事件，并将事件投递到匹配的 Session。
- Agent 应能在等待事件的 Session 中按事件触发继续执行。
- Agent 应避免重复处理已经消费的消息。

验收标准：

- 新消息可以唤醒对应 Session。
- 无会话消息可以进入默认会话。
- 等待事件的 Session 可以被匹配事件唤醒。
- 已处理消息不会被重复消费。

### 4.4 Session 生命周期管理

OpenDAN 应支持 Session 的完整生命周期管理。

需求：

- 系统应支持创建或获取 Session。
- Session 应支持等待用户输入。
- Session 应支持等待系统事件。
- Session 应支持进入准备执行状态。
- Session 应支持正在执行状态。
- Session 应支持结束状态。
- Session 应支持暂停和恢复。
- Session 状态应可持久保存。
- Session 应能在系统重启后恢复必要状态。

验收标准：

- 管理端可以列出某个 Agent 的 Session。
- 管理端可以查看某个 Session 的详情。
- 用户或管理端可以暂停 Session。
- 被暂停的 Session 不应继续自动执行。
- 用户或管理端可以恢复 Session。
- Session 恢复后可继续接收输入或执行任务。

### 4.5 Behavior 执行

OpenDAN 应支持通过 Behavior 驱动 Agent 执行任务。

需求：

- Agent 应支持为不同场景配置不同 Behavior。
- Session 应记录当前正在使用的 Behavior。
- Agent 应能根据输入选择或切换 Behavior。
- Behavior 应定义输入组织方式、输出要求、可用工具、可用技能和执行限制。
- Behavior 应支持多步执行。
- Behavior 应支持在执行失败时进入可控的后续处理流程。

验收标准：

- 默认对话消息可以进入消息路由 Behavior。
- 工作类任务可以进入任务规划或执行 Behavior。
- Behavior 输出应能被系统识别并推动 Session 状态变化。
- Behavior 使用的工具应受权限约束。

### 4.6 AI 推理能力

OpenDAN 应支持 Agent 调用 AI 模型完成推理。

需求：

- Agent 应能基于 Session 上下文构造模型输入。
- Agent 应能调用系统提供的 AI 模型服务。
- Agent 应能接收模型输出并解析为结构化行为结果。
- Agent 应能处理模型暂不可用、超时、输入过长等异常情况。
- Agent 应记录模型调用的必要追踪信息。

验收标准：

- Agent 可以基于用户消息生成回复。
- Agent 可以在复杂任务中生成计划、步骤、总结和工具调用请求。
- 当模型不可用时，用户应收到明确的失败或稍后重试提示。
- 管理端可以追踪一次任务涉及的模型调用信息。

### 4.7 工具调用

OpenDAN 应支持 Agent 调用受控工具完成具体操作。

需求：

- Agent 应能查询当前 Behavior 允许使用的工具。
- Agent 只能调用被授权的工具。
- 工具调用应返回结构化结果。
- 工具调用失败时，Agent 应能获得失败原因并决定后续动作。
- 工具调用过程应被记录到 Worklog。

基础工具能力应包括：

- 读取文件。
- 写入文件。
- 编辑文件。
- 执行受控命令。
- 创建工作空间。
- 绑定工作空间。
- 管理 Todo。
- 读写长期记忆。
- 查询 Session 状态。
- 加载和卸载 Skill。

验收标准：

- 未授权工具不能被 Behavior 调用。
- 工具执行结果可以进入下一轮 Agent 推理。
- 文件写入、命令执行、Todo 变更等关键操作应有可追踪记录。

### 4.8 Workspace 管理

OpenDAN 应支持 Agent 围绕工作空间执行任务。

需求：

- 系统应支持创建 Agent 工作空间。
- 系统应支持将 Session 绑定到工作空间。
- 系统应支持绑定外部工作空间。
- Agent 应能在当前工作空间中读取上下文资料。
- Agent 应能在当前工作空间中生成任务产物。
- 系统应支持列出 Agent 可访问的外部工作空间。
- 工作空间访问应有边界，避免越权访问未授权路径。

验收标准：

- 用户可以为一个任务创建工作空间。
- 用户可以将已有目录授权给 Agent 使用。
- Agent 的文件操作应限定在授权范围内。
- 管理端可以查看 Session 当前绑定的工作空间。

### 4.9 Todo 管理

OpenDAN 应支持 Agent 使用 Todo 管理复杂任务。

需求：

- Agent 应能创建 Todo。
- Agent 应能更新 Todo 状态。
- Agent 应能列出当前 Session 或工作空间相关 Todo。
- Todo 应能区分未完成、执行中、已完成、失败等状态。
- Todo 应支持关联 Agent、Session、工作空间和任务步骤。
- 管理端应能查询 Agent 的 Todo 列表。

验收标准：

- Agent 执行复杂任务时可以生成 Todo 列表。
- Todo 状态可以随任务推进而变化。
- 用户可以从管理端看到任务拆解和完成情况。

### 4.10 Worklog 管理

OpenDAN 应支持完整记录 Agent 工作过程。

需求：

- 系统应记录 Agent 接收到的重要输入。
- 系统应记录 Agent 的关键输出。
- 系统应记录工具调用计划和执行结果。
- 系统应记录每个任务步骤的总结。
- 系统应记录失败、重试和异常信息。
- Worklog 应支持按 Session、步骤、类型、状态和关键字查询。
- Worklog 应适合展示给用户和管理端。

验收标准：

- 用户可以查看某个工作 Session 的执行轨迹。
- 管理端可以按条件检索 Worklog。
- 文件变更和工具执行类操作应能在 Worklog 中找到对应记录。

### 4.11 Memory 管理

OpenDAN 应支持 Agent 长期记忆能力。

需求：

- Agent 应能读取长期记忆。
- Agent 应能写入长期记忆。
- Agent 应能删除不再需要的记忆。
- 记忆应支持摘要、类型和时间相关信息。
- Agent 应能在后续会话中使用已有记忆。
- 用户应能通过 Agent 修改或清理记忆。

验收标准：

- Agent 可以记住用户明确要求保存的信息。
- Agent 可以在后续任务中引用相关记忆。
- 用户可以要求 Agent 删除某条记忆。

### 4.12 定时自检与主动任务

OpenDAN 应支持 Agent 主动检查记忆或时间条件，并在需要时启动任务。

需求：

- Agent 应支持定时自检。
- 自检应能基于记忆中的提醒、时间点或周期性条件触发。
- 自检触发后，应进入独立 Session 或指定 Session 执行。
- 自检不应在任务正在运行时造成重复执行。
- 自检触发原因应可追踪。

验收标准：

- Agent 可以根据保存的提醒信息主动触发后续处理。
- 自检触发过程可以在 Worklog 或状态记录中被追踪。
- 自检不会频繁重复触发同一任务。

### 4.13 Skill 管理

OpenDAN 应支持按需加载和卸载 Skill。

需求：

- Agent 应能加载指定 Skill。
- Skill 应可加载到当前 Behavior。
- Skill 应可加载到整个 Session。
- Agent 应能卸载已加载 Skill。
- Session 应记录已加载 Skill。
- Skill 加载失败时，应给出明确原因。

验收标准：

- Agent 可以在任务中加载某个领域 Skill。
- 加载后的 Skill 会影响后续任务执行。
- 卸载 Skill 后，该 Skill 不应继续影响后续步骤。

### 4.14 子 Agent 管理

OpenDAN 应支持主 Agent 为复杂任务创建和管理子 Agent。

需求：

- 主 Agent 应能创建子 Agent。
- 子 Agent 应拥有独立身份、角色描述和工作目录。
- 子 Agent 应能被查询和展示。
- 子 Agent 应能与父 Agent 建立归属关系。
- 子 Agent 的工作过程应可独立追踪。

验收标准：

- 主 Agent 可以为复杂任务创建专门子 Agent。
- 管理端可以看到某个 Agent 下的子 Agent 列表。
- 子 Agent 的状态、工作空间和活动记录可以被查询。

### 4.15 管理接口

OpenDAN 应为控制面板、调试工具和测试系统提供稳定管理接口。

需求：

- 接口应支持列出 Agent。
- 接口应支持查询 Agent 详情。
- 接口应支持查询 Agent 工作空间摘要。
- 接口应支持列出 Worklog。
- 接口应支持列出 Todo。
- 接口应支持列出子 Agent。
- 接口应支持列出 Session。
- 接口应支持查询 Session 详情。
- 接口应支持暂停和恢复 Session。
- 接口返回应适合 UI 展示和自动化测试使用。

验收标准：

- 控制面板可以通过接口展示 Agent 运行状态。
- 自动化测试可以通过接口验证 Agent 任务执行结果。
- 接口在无数据时应返回空结果，而不是不可理解的错误。

### 4.16 可观测性

OpenDAN 应提供足够的运行可观测能力。

需求：

- 系统应记录 Agent 启动、加载、运行、失败等关键状态。
- 系统应记录 Session 状态变化。
- 系统应记录消息路由结果。
- 系统应记录模型调用结果。
- 系统应记录工具调用结果。
- 系统应记录 Todo 和 Worklog 变化。
- 系统应允许管理端查询关键运行状态。

验收标准：

- 当 Agent 没有回复时，开发者可以通过记录判断卡在哪个阶段。
- 当工具调用失败时，用户或开发者可以看到失败原因。
- 当模型服务不可用时，系统能明确呈现异常来源。

## 5. 非功能需求

### 5.1 可用性

- OpenDAN 应保持默认 Agent 的基础可用。
- 非核心依赖失败时，系统应尽量降级运行。
- Agent 执行失败时，应避免影响其他 Agent 或其他 Session。

### 5.2 持久化

- Agent 身份、配置、Session、Todo、Worklog、Memory 和 Workspace 绑定关系应可持久保存。
- 系统重启后，应尽量恢复未结束 Session 的必要上下文。

### 5.3 安全与权限

- Agent 只能使用被授权的工具。
- Agent 文件访问应限定在授权工作空间内。
- 外部工作空间必须经过显式绑定。
- 关键操作应留下可审计记录。

### 5.4 可扩展性

- 系统应支持新增 Behavior。
- 系统应支持新增 Skill。
- 系统应支持新增工具。
- 系统应支持新增 Agent 类型。
- 系统应支持后续扩展多 Agent 协作能力。

### 5.5 可测试性

- OpenDAN 应支持绕过外部聊天入口直接向 Agent 投递测试消息。
- OpenDAN 应支持查询任务运行结果和 Worklog。
- OpenDAN 应支持在测试环境中验证 Agent loop、工具调用、Todo、Worklog 和模型交互。

## 6. 典型用户场景

### 6.1 用户向默认 Agent 提问

用户向默认 Agent 发送一条消息。OpenDAN 接收消息，找到或创建对应 Session，选择合适 Behavior，调用 AI 模型生成回复，并将回复发送给用户。

验收结果：

- 用户收到回复。
- Session 中保留本次交互上下文。
- 必要的模型调用和执行过程可追踪。

### 6.2 用户要求 Agent 完成复杂任务

用户要求 Agent 完成一个多步骤任务。Agent 创建工作 Session，拆解 Todo，必要时创建工作空间，逐步执行任务，并在 Worklog 中记录过程。

验收结果：

- 用户可以看到任务拆解。
- Agent 可以持续推进任务。
- 用户可以查看任务执行记录和产物。

### 6.3 用户绑定外部目录让 Agent 工作

用户授权一个本地目录作为外部工作空间。Agent 在授权目录范围内读取资料、修改文件或生成结果。

验收结果：

- 目录绑定成功。
- Agent 文件操作不越过授权范围。
- 文件修改有记录。

### 6.4 Agent 根据记忆主动提醒用户

用户要求 Agent 记住一个未来提醒。OpenDAN 保存记忆，并在自检中发现触发条件后唤醒 Agent 执行提醒任务。

验收结果：

- 提醒信息被保存。
- 到达触发条件后 Agent 主动处理。
- 触发原因可以追踪。

### 6.5 管理端查看 Agent 运行情况

用户或开发者打开管理界面，查看 Agent 列表、Session、Todo、Worklog 和子 Agent。

验收结果：

- 管理端能展示 Agent 状态。
- 管理端能查看某个任务的执行过程。
- 管理端能暂停或恢复指定 Session。

## 7. 边界说明

- OpenDAN 不直接定义底层 AI 模型服务能力，而是使用系统提供的 AI 能力完成推理。
- OpenDAN 不直接承担系统级用户身份管理，而是使用 BuckyOS 身份体系。
- OpenDAN 不应绕过 BuckyOS 的权限、消息、任务和配置体系。
- OpenDAN 不保证每个 Agent 都具备所有工具，具体能力应由 Agent 配置和授权决定。
- OpenDAN 不应把所有 Agent 行为写死在运行时中，Agent 行为应可配置和扩展。
