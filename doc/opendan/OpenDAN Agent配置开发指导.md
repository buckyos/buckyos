# OpenDAN Agent 配置开发思路

> **本文目的**:为 OpenDAN 的"配置开发者"建立心智模型——拿到一个需求,能立刻定位"我要改的东西在哪个概念里"。
> **本文不是**:实现说明、API 文档、配置 DSL 规范。

---

## 一、立论:为什么是"配置开发",而不是 SDK

主流 Agent 框架把 Agent 当成一段需要编写的程序(ADK / SDK 路线)。OpenDAN 不这么做。

OpenDAN 主张:**Agent 是一种服务,服务的形态应该由配置描述,而不是由代码编写**。

理由有三:

1. **Agent 的复杂度集中在编排,而不是算法**。编排的事是"什么时候触发 LLM、用什么提示词、能调什么工具、推理结果怎么转下一步"——这些天然是**声明式**的,用代码写反而绕。

2. **配置可移植、代码不可移植**。OpenDAN 的 Agent 跑在容器里,容器在哪台机器、什么操作系统上,Agent 都该一样跑。纯文本配置是平台无关的,Windows / Linux / macOS 都能跑。

3. **安全边界由容器保证,配置层不需要关心**。从 OS 视角,一个 Agent 就是一个加载了特定配置的容器化普通应用,**配置怎么写也越不出容器**。这让配置开发者可以放心地只关心"业务",不关心安全。

这一立场决定了后面所有概念的形态:**每一个概念都对应配置文件里的一个区块,而不是代码里的一个类**。

---

## 二、心智锚点:用 Nginx 来理解 OpenDAN

Nginx 的成功不在性能,在于**它的配置文件就是它的架构图**——读 nginx.conf 就能看懂一个网关全部行为。OpenDAN 想给 Agent 配置开发者同样的体验。

对应关系如下(**这是全文最重要的一张表**):

| Nginx 概念 | OpenDAN 概念 | 职责 |
|---|---|---|
| Nginx 进程 | **Agent Runtime** | 接收外部事件,做总路由 |
| `server { }` | **Session** | 一类对话/任务的承载单元,有自己的状态 |
| `location { }` 内的处理逻辑 | **Behavior** | 在 Session 当前状态下,具体怎么处理 |
| `upstream` / `proxy_pass` 目标 | **工具集 / Skills** | Behavior 可调用的下游能力 |

一句话总结:
> **AgentRuntime 是 Gateway,Session 是 Server,处理逻辑由当前状态选择一个 Behavior,具体怎么处理写在 Behavior 配置里**。

## 三、三个核心概念的"心智位置"

### 3.1 Agent Runtime —— Gateway 层

**它做什么**:
- 声明 Agent 能从哪些通道接收 Message / Event(IM、HTTP、内部事件总线……都可以)
- 声明一个 Dispatcher,把事件按类型路由到对应的 Session
- 决定何时复用已有 Session、何时开新 Session

**心智位置**:这是 Agent 的"门面"。读这一段配置,就知道 Agent 长什么样、对外暴露什么。

**改动频率**:🟦 **极少改**(框架型)
> 一个 Agent 的"对外形态"基本一开始就定了,后续业务迭代很少动这里。

---

### 3.2 Session —— Server 层

**它做什么**:
- 声明一类对话/任务的**承载形态**
- 持有这一类交互的**状态**(对话历史、上下文、当前所在的 Behavior、环境变量等)
- 选择这一类 Session 的 **Loop 模式**(普通 Agent Loop / Behavior Loop)
- 指定这一类 Session 启动时的 **default Behavior**

**心智位置**:Session 是 Agent 对外暴露状态的最小语义单元。**用户视角的"一次对话""一个任务"**,在配置层就是一个 Session。

**Loop 模式决定了 Session 是不是会切换 Behavior**:

| Loop 模式 | Behavior 切换 | 等价心智 |
|---|---|---|
| **普通 Agent Loop**(UI 风格) | **不切换**——永远停留在 default Behavior 上 | "UI Session"是"只有一个固定 Behavior 的 Session"的别名 |
| **Behavior Loop**(Work 风格) | 由 Behavior 自己声明 next,**显式切换** | 一个 Session 在多个 Behavior 间走状态机 |

> **重要的概念统一**:
> OpenDAN 里**没有"两种 Session"这种二分**——只有 Session、Loop 模式、Behavior 三个概念。所谓"UI Session"只是**"选了普通 Agent Loop、所以永远固定在一个 Behavior 上"**的特例。
>
> 这个统一让心智模型变干净:**Session 是壳和状态,Behavior 是处理逻辑,Loop 模式只是决定壳里要不要切 Behavior**。

**对应的"一次推理何时停"语义**:

| Loop 模式 | 一次推理在何时停 | 典型场景 |
|---|---|---|
| 普通 Agent Loop | 模型自己决定(没有 function call 就停) | 对话式交互——"停"是天然事件(对方说完了) |
| Behavior Loop | **配置显式决定**(Behavior 状态机判定) | 后台任务——"停"必须事先由人定义(任务完成的判据) |

这就是为什么需要 Behavior Loop:任务场景里"停"不是天然事件,必须显式声明。

**改动频率**:🟨 **偶尔改**(半框架型)
> 加一类新交互场景(比如"加一个邮件处理通道"),才会动到这里。日常业务迭代不动这里。

---

### 3.3 Behavior —— 处理逻辑层

**它做什么**:Behavior 是"**围绕 Session 当前状态**编写的一段处理逻辑"。它要管两件事:

#### A. 常规处理:把状态渲染成一次 LLM 调用
- 基于 Session 当前状态,**渲染 system prompt 和上下文**
- 声明这次调用可用的**工具集 / Skills**
- 声明这次调用完成后,**下一步切到哪个 Behavior**(或回到调用方)

#### B. 异常处理:响应 LLMContext 的事件
一次 LLM 调用不是总能顺利跑完——它可能撞到下层 LLMContext 抛出的事件,典型如:
- **上下文满了**(`ContextLimitReached`)——要不要压缩?压缩用什么策略?压缩完接着跑还是切到另一个 Behavior?
- **provider 失败 / 超时**——降级到哪个 Behavior?
- **被外部 suspend / resume**——恢复时怎么读回 Session 状态?

这些事件**不是写在 Session 里的**——Session 只持有状态。**响应事件的逻辑写在 Behavior 里**,因为响应方式跟"当前在干什么"强相关:
> 一个"正在写代码"的 Behavior 和一个"正在回复用户"的 Behavior,对"上下文满了"的合理反应是不一样的。

**心智位置**:Behavior 是 Agent 真正"在干活的地方"。**业务逻辑写在哪里?写在 Behavior 里**。包括:
- 正常路径上的提示词、工具、下一步——写在 Behavior 里
- 异常路径上对 LLMContext 事件的响应——也写在 Behavior 里

**Behavior 之间怎么切**:**fork 是默认推荐**。
- Behavior A fork 到 Behavior B,B 执行完(无论成败)**整个分支消失,只把结果带回 A**
- 主干上下文保持干净,B 的中间过程不会污染 A

为什么默认 fork 而不是图模式跳转?
> 图模式下,任何一个 Behavior 可能从多个上游进入,它的提示词必须为"所有可能的上游状态"做适配——**提示词复杂度 ∝ 图的连通度**。这违反了"简单声明就能跑"的承诺。
>
> Fork 让每个 Behavior 只关心"我从干净的初始状态开始处理什么",这是配置开发能成立的前提。

**改动频率**:🟥 **经常改**(应用型)
> 这就是日常业务迭代的主战场——加一个新行为、调一段提示词、改一个工具集、改一种异常应对方式,都在这里发生。

## 四、概念变更频率全景:让开发者知道动哪里

这张表是开发者拿到需求后的**决策树**:

| 需求形态 | 改哪一层 | 频率 |
|---|---|---|
| "我要让 Agent 多支持一种接入方式(比如新增微信通道)" | Agent Runtime(Gateway) | 🟦 极少 |
| "我要加一种全新的交互形态(比如除了聊天还要支持工单流)" | 新增一类 Session | 🟨 偶尔 |
| "我要把某类对话从普通 Agent Loop 改成显式状态机" | 改 Session 的 Loop 模式 | 🟨 偶尔 |
| "我要改某个任务的处理方式 / 提示词 / 可用工具" | Behavior | 🟥 经常 |
| "我要加一种新能力 / 新工具" | 工具集层(Behavior 引用) | 🟥 经常 |
| "我要改某个 Behavior 在上下文满时怎么处理" | Behavior(LLMContext 事件响应段) | 🟥 经常 |
| "我要新增一个分支处理(fork 出去做某件事再回来)" | 新增一个 Behavior + 在源 Behavior 里加 fork 声明 | 🟥 经常 |

**关键判据**:
- 改的是**对外形态** → Gateway
- 改的是**状态语义 / Loop 模式** → Session
- 改的是**做事方式(正常路径或异常路径)** → Behavior

## 五、OpenDAN 的设计立场(贯穿全文的原则)

以下原则不是规则,而是**判据**——后续具体的 DSL 设计、运行时实现、扩展机制,都应该用这些原则来裁判。

### 5.1 显式大于隐式
"一次推理何时停"、"Behavior 何时切换"、"上下文何时压缩"——这些都必须在配置里**显式声明**,不依赖模型自觉,不依赖运行时猜测。

### 5.2 配置即架构
读一份 Agent 配置就能看懂这个 Agent 的全部行为。**没有藏在代码里的逻辑**。

### 5.3 树形优于图形
Behavior 之间的关系默认是**调用树**而非**状态图**。这让每个 Behavior 的提示词自由度最大、复杂度最小。

### 5.4 未引用即零成本
Behavior 可以被 Agent 提供、被用户运行时扩展。**没有被引用的 Behavior 不加载、不计费、不影响主路径**。这让"扩展"成为安全操作。

### 5.5 容器是硬边界
配置层不承担安全责任。这反过来允许配置层放飞——任何配置错误都不会让 Agent 越出容器能力边界。

### 5.6 提示词不该腐烂
当一个 Session 的 system prompt 开始膨胀(把所有能力都塞进去),那就是该**拆 Behavior** 的信号。Behavior 切换的本质是**一种更高层的路由策略**——按"现在该干什么"把提示词、工具、Skills 切片。

---

## 六、本文不规定什么(给实现留的自由度)

为了让指导文档保持稳定,以下事项**故意不规定**,留给后续设计:

- Gateway 支持的具体协议清单——任何能产生事件流的通道都可以
- Dispatcher 的具体 DSL 形态——但要求**声明式、可静态分析**
- Behavior 跳转结果回主干时的上下文序列化协议——但要求**可序列化、可回放**
- UI Session 的旁路事件点全集——但要求**事件点是可枚举的、非开放扩展的**(避免无限钩子导致的不可推理性)

这些是 **OpenDAN 的"留白"**,不是欠的债。
