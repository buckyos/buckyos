# OpenDAN AgentSession 的事件订阅设计

> 整理自语音记录。本文讨论 Agent Session 在"非自然推理结束"——即需要等待外部状态变化才能继续推进——这一场景下的几种实现模式，以及它们各自的语义边界。

## 1. 问题场景

当一个 Agent 把任务委派出去（无论委派给另一个 Agent 还是给人）之后，从 Session 的运行状态来看，它的推理实际上已经结束了：

- 推理已结束，但**任务并未完成**；
- Session 正在等待一个 **Task 对象的状态改变**；
- 这种等待本质上是一个**轮询状态**。

在分布式系统中，触发 Session 下一次工作的事件，逻辑上只有两类：

1. **侦测到目标 Task 对象的状态发生了改变**；
2. **Timeout**：因为是分布式系统，担心漏消息，所以即使没有状态变更通知，也需要按超时主动激活一次推理。

被激活后的这次推理，本质上要让系统提示词能够识别"当前处于等待中"这个状态。例如：Agent 发现某个 delegate 出去的任务已经过了 25 分钟，需要根据目标对象的当前状态决定下一步动作。可能的决策无非是：

- 继续等；
- 催一下（尤其是 delegate 给人的情况）；
- 换一个方案。

围绕这个场景，下面整理出三种实现模式。

---

## 2. 三种实现模式

### 模式一：长 Tool / PendingTool（隐式等待）

**核心机制**：Agent 在调用某个 Tool 时，Tool 内部直接触发了一个等待。Tool 通过 `PendingTool` 机制立即返回一个 `TaskID`，此时 ToolResult 还处于"工具悬停"状态。

**状态机语义**：
- 这一轮推理**不会最终结束**；
- 系统**精确地**知道这一轮推理依赖哪些 TaskID 的最终状态；
- 当这些 TaskID 全部进入最终状态后，通过 `Resume` 把最终结果塞回去，让推理恢复。

**特点**：
- 这是一种**精确的、显式的依赖筛选**——状态机上这一轮推理精确依赖某个特定状态的改变；
- 通常 Tool Call 是串行的，必须等这个状态回来才能进入下一环节；
- 从 LLM 视角看，这是**隐式逻辑**：LLM 自己并没有主动表达"我要等待某个东西改变"的意图，等待是被 Tool 的实现方式所决定的。

**适用与缺点**：
- 适合**短等待**；
- 不适合长等待。核心原因：等待发生在**一轮推理的内部**，意味着同一轮里**前面已经成功的 Tool Call 结果会被一直保留并参与最终推理**。如果其中包含读取类操作（读某个对象状态、读某个文件），25 分钟后这些读取结果其实**已经不可信了**——但状态机上它们仍然作为"已成功"的输入参与下一步推理，会得出基于过期数据的结论；
- 从状态机上看就是"一个 Tool 没返回"，控制权被压出来后，上层通过标准系统 API 订阅 Task 的最终状态，再 Resume 推回去。

### 模式二：显式事件订阅（主动通知 / 真实世界模式）

**核心机制**：从 LLM 的角度看，**这一轮推理已经完成了**。但它表达的意图是"我需要等待一个外部事件发生"。这是一种**意图状态机**：LLM 用自然语言表达了一个意图（"我等 xxx 通知我"），但**并未规定得到这个通知的具体通道**——通道由系统根据这个意图自行解释和绑定。一轮已结束，Session 何时再次进入推理，取决于显式订阅的事件被触发。

**实现思路**：提供一个通用的事件订阅函数。

- 因为我们把所有东西都做了路径化，所以 Session 可以指定"我要订阅某条路径指向的对象的某种事件"；
- 当该对象状态发生改变时——**关键点来了**——这个事件会被**转化为一条消息**，而不是直接把事件 JSON 塞进上下文。

**为什么要转成消息？**

之前我们做固定的 JSON 转换，直接把事件 JSON 塞进去，有时会导致**语义漂移**。作为 Agent 的编写者，你大致知道它会订阅哪些类型的事件，所以你完全可以提前把它**渲染成一条自然语言事件消息**。

从系统视角看，这条消息进来后相当于收到了一条 `user message`，只是它的**来源**是对象状态改变。这样系统就变成了一个**朴素的状态机**——统一通过 user message 来主动激活 Agent Session 的继续工作。

**类比**：把 LLM 想象成一个人，它不是通过软件回调"得知"事件，而是有人在它耳朵边说一句："某某摄像头看到了一个人走过。"它再根据自己的系统提示词逻辑、以及上一轮的状态，继续处理。

**与模式一的对比**：

| 维度 | 模式一（长 Tool） | 模式二（显式订阅） |
|---|---|---|
| LLM 意图 | 隐式（LLM 不知道自己在等） | 显式（LLM 主动表达"等待某事") |
| 状态机 | 一轮推理内部挂起 | 一轮推理已结束，等下次激活 |
| 适用 | 短等待 | 长等待、跨人/跨 Agent 协作 |

**典型场景**：等待某个人确认一件事——给他发了消息后等他回复；如果到时间没回复，下一次推理时根据提示词主动再推他一下。

### 模式三：环境感知 / 弱订阅（被动通知）

**核心机制**：这类变化的特点是——它**并不是 Agent Session 主动注册的事件**，**不足以单独激活 Session**；但当用户主动发来新消息时，这些环境变化对 Agent 处理这条用户消息相对来说比较重要。

**与模式二的根本区别——两种认知模型**：

| | 模式二 | 模式三 |
|---|---|---|
| Agent 是否知道自己在等这个 | **知道**（自己显式注册） | **不知道**（系统在用户消息到达时顺便提醒） |
| 类比人的认知 | "我在等小王的回复" | 走到楼下才注意到"天色已经晚了" |

模式三模拟的是人类思维方式中的**"浮现"**——你并没有刻意去监控周围环境，但当你要做某个决定时，相关的环境信息会自然地浮现到意识里。系统要做的，是对 **Session Topic 的最新状态做低成本探测**，从而判断哪些环境变量"应该浮现"。

> 命名说明：这个词不太好解释，没有专门的定义，可以叫"弱订阅"或"半订阅"，对应一种**被动的事件通知**。
> - 模式二是 Agent Session 主动订阅 → **主动通知**（事件 → user message）；
> - 模式三是用户消息触发时附带的环境快照 → **被动通知**（变化 → 注入到当前 user message）。

**前提：Session 持有一个动态 Topic**

模式三依赖一个前提——Agent Session 持有一个**当前 Topic**，这个 Topic 由旁路 LLM 持续维护更新（具体机制见 §3）。Topic 是模式三的中枢，它决定了"现在应该订阅哪些环境变量"。

**触发时机**：
- 事件发生时**不会**激活 Agent Session；
- 当用户发来一条新消息时，系统检查"上次用户消息以来"哪些被订阅的环境变量发生了变化；
- 将这些**变化的环境变量自动注入到这条 user message 的前面**。

**典型例子：用户说"帮我订明天早上飞东京的票"**

这一刻：
- 用户的**地理位置**变得非常关键（涉及时区、出发机场）；
- 如果用户经常出差，地理位置是会经常变化的。

过去的常见做法有两种，都不理想：

1. **硬编码进系统提示词**：适合不常变的状态，不适合地理位置这种动态状态；
2. **提供 `readObject` / `readEnv` 类函数**：会迫使系统提示词暴露**大量状态提示**，且因为不知道有没有变化，主上下文会被反复查询塞满。

**我们的旁路思路**：

与"历史记录压缩"对偶——
- 历史记录压缩：对**老信息**做旁路 LLM 处理；
- 环境变化感知：对**新信息**做旁路 LLM 处理。

具体流程：

1. 当用户两次消息之间的"环境变更点"达到某个阈值/条件，触发一次**旁路 LLM**；
2. 旁路 LLM 根据**当前聊天的 Topic**，为该 Session 做环境订阅；
3. 这些订阅的环境变量（基于路径标识）一旦发生改变，就有机会在**下一条 user message 到达时**，把"变化了什么"植入到这条用户消息的环境块中。

回到订机票的例子：当用户说"帮我订明天早上飞东京的票"，如果旁路系统发现用户的常用坐标是硅谷，而当前位置是上海，就会在这条 user message 里主动植入一条用户地理位置变化的环境信息。

## 3. 模式三的执行层总结

从执行层面，Session 持有一组与**当前 Topic 相关**的环境**弱订阅 / 半订阅**。其行为约束：

- **触发点**：每次用户有新的输入消息时；
- **检查**：系统当前弱订阅的状态相比上一次是否发生了改变，且本次"环境更新"事件被允许触发；
- **行为**：若满足条件，自动把改变的环境变量插入到这条用户输入消息的前面。

**触发条件的工程约束**：

环境变化注入的核心目的是辅助 Agent 决策，但代价是**消耗上下文**——背景信息改变得太频繁会污染主上下文，反而降低主推理质量。因此触发条件本质上是一个**上下文成本控制**问题，工程实现上需要至少两个可配置维度：

1. **变化幅度阈值**：环境变量改变多少才值得注入（例如地理位置移动了多少公里、时间过去多久才算"变了"）；
2. **变化间隔阈值**：两次注入之间至少要间隔多久，防止短时间内反复抖动。

> 具体阈值如何确定、不同环境变量类型是否共享同一组阈值，属于实现期待定项。

**实现细节：独立 user message vs 追加 block**

- 从**语义权重**上讲，理想的形式是**追加一个 block 到当前 user message 里**——因为这条环境信息是"附属于这条用户消息的背景"，独立成一条 user message 会让它在历史中显得权重过平，与真正的用户输入难以区分；
- 真正的需求是 UI 层和上下文构造层能够**识别这个 block 的特殊语义**，把它作为"环境上下文"而不是"用户说的话"来对待；
- 这套机制类似于过去在 user message 中插入背景信息（syncing 类）的技术。

**Topic 是怎么来的？**

由前面提到的旁路 LLM 决定——一个**快速的旁路 LLM**根据当前对话信息持续 update Session 的 Topic。这个 Topic 维护机制本质上是**对 Session 当前主题的低成本探测**，是对人类思维中"浮现"机制的模拟。

Topic 一旦变化，会连锁触发多种事情：

- 触发弱订阅的重新订阅（订阅哪些环境路径）；
- 触发 memory 的相关查找；
- ……

**延伸思考：memory 与环境感知的关系**

某种意义上讲，memory 也可以被视为一种环境的改变——如果把用户的记忆想象成脑袋里不同的对象，也可以用路径来表达。沿着这条思路推下去，memory 的改变与环境状态的改变在机制上有相似之处：都可以通过 Session Topic 的变化、由旁路 LLM 找到相关路径并完成订阅。

> 注：这是一个延伸观察，**并不急于在工程上把两者统一**。现阶段 memory 与环境感知按各自独立的机制实现即可。

## 4. 三种模式对比

| 维度 | 模式一：长 Tool | 模式二：显式订阅 | 模式三：环境弱订阅 |
|---|---|---|---|
| **本轮推理是否结束** | 未结束（挂起在 Tool 内部） | 已结束 | 已结束 |
| **激活下次推理的来源** | Task 状态改变 / Timeout | 事件 → 转 user message | 用户主动消息（事件附带） |
| **LLM 视角的意图** | 隐式 | 显式（明确表达等待） | 隐式（事件被动附带） |
| **订阅注册者** | 系统（由 Tool 调用决定） | Agent Session 主动 | 旁路 LLM 根据 Topic 自动 |
| **适用等待时长** | 短 | 长 | 不直接处理等待 |
| **事件载体** | ToolResult Resume | 一条新的 user message | user message 中的 env block |
| **典型场景** | 同步 RPC 风格的延迟调用 | 等待某人确认、跨 Agent 协作 | 地理位置、时间、用户偏好等环境上下文 |

**三种模式是并存的，不是三选一**。在一个完整的 Agent Session 里：

- 模式一处理同步短任务（一次工具读取、一次内部调用）；
- 模式二处理长时间协作（等人回复、等下游 Agent 完成）；
- 模式三处理环境上下文（地理位置、时区、相关 memory 浮现）。

三者各自覆盖不同语义维度，工程上需要在同一个 Session 状态机里同时实现并协调。

## 5. 运行时语义：并发与事件队列

前面四节描述的都是"事件如何激活 Session"的**静态语义**，但真实运行时还有一个绕不开的问题：

> **如果 Session 正在运行（推理中），此时又来了一个事件，怎么办？**

这是三种模式都会遇到的共有问题，尤其是模式二——显式订阅本来就期待事件随时到达。下面给出运行时约束。

### 5.1 单 Active Run 约束

**同一个 Session 在任意时刻只能有一个 active run**（一次正在进行的推理）。

理由：
- LLM 推理本身是一个**有状态的过程**，并发激活同一个 Session 会导致状态机分叉，结果难以合并；
- 上下文构造、Tool Call 序列、Resume 检查点等机制都假设串行执行。

### 5.2 Pending Event Queue

当 Session 处于 active run 时新事件到达，事件进入该 Session 的 **pending event queue**，**不**打断当前推理。

队列在以下时刻被消费：
- 当前 run 自然结束（推理完成、进入等待态）后；
- 系统判断需要激活 Session 时，从队列中取出事件，构造下一次激活的输入。

> 即使 Session 处于"等待态"（模式二订阅完成后），事件也是先进队列、再在系统统一调度下触发激活，而不是事件回调直接驱动 LLM 入口。

### 5.3 事件压缩：Coalesce 与 Override

队列中的事件**不是简单 append**，需要根据语义做合并。这里区分两种不同的合并语义：

**Coalesce（同路径去重）**：
- **定义**：同一个路径标识的对象，先后产生了多次状态变更事件，只保留**最新一次**的状态快照；
- **典型例子**：地理位置在 5 分钟内连续上报了 10 次，只需要最后那次；订阅的某个对象属性被频繁修改，只需要最终值；
- **本质**：这些事件**语义上不冲突**，只是中间态可以丢弃。

**Override（语义覆盖）**：
- **定义**：后一个事件让前一个事件**在语义上失效**，前者直接作废；
- **典型例子**：先收到"任务进行中（25%）"，再收到"任务已完成"——前者完全无意义；先收到"用户已确认"，再收到"用户取消"——前者已被推翻；
- **本质**：这些事件**有终态关系**，旧事件不只是"过时"，而是"被新事件否定"。

实现上，Coalesce 通常按"同 path + 同事件类型"自动做；Override 则需要事件类型本身声明终态关系（哪些状态是终态、哪些状态覆盖哪些）。

### 5.4 Event Batch Message

当 Session 从队列消费多个事件时，**不应**为每个事件单独激活一次 LLM——这会让 LLM 被一连串小事件反复打断、缺乏整体判断。

正确做法：**将队列中的事件批量构造为一条 event batch message**，然后激活一次 LLM 推理。

这条 batch message 在语义上仍然是"系统告诉 Agent 发生了哪些事"，但允许 LLM 在一次推理里**整体地**看到所有相关变化，从而做出更合理的决策（例如发现"任务已完成 + 用户已取消"同时存在时，优先处理取消逻辑）。

### 5.5 模式间的差异

三种模式在运行时队列上的处理略有差异：

| | 队列内容 | 消费时机 | 备注 |
|---|---|---|---|
| **模式一** | Resume 所需的 TaskID 终态 | 所等 TaskID 全部到达终态时 | 无关事件被旁路忽略或转给模式二/三 |
| **模式二** | 显式订阅的事件 | 至少一个订阅事件到达，或 Timeout | 多事件需 batch；可触发主动催办逻辑 |
| **模式三** | 环境变化快照（按路径） | 用户发新消息时 | 不单独激活，仅作为 user message 的附加 block |

模式一关心的是"我等的那个 TaskID 来了没"，其他事件不归它管；模式二是真正意义上的"事件驱动队列消费者"；模式三的队列其实是**快照差量**，消费时机由用户消息触发，而不是事件本身触发。

## 6. 模式选择反例

为了让设计边界更稳，下面给出三种模式各自的**典型误用场景**和后果。

### 反例一：用模式一做长协作

**错误用法**：用 PendingTool 实现"等待用户审批某个文件"——把审批做成一个长 Tool Call，等审批完成才返回 ToolResult。

**后果**：
- 一旦审批跨越数小时甚至数天，同一轮推理中**前面已经成功的读取类 Tool Call**（例如"读取该文件的当前内容"）会变得严重过期，但状态机仍然把它们当作"已成功的输入"参与最终推理，导致基于陈旧数据做决策；
- 推理上下文需要长时间保活，工程上不可持续。

**正确做法**：改用模式二。Agent 发出审批请求（这是一个独立的、立即返回的 Tool Call），然后显式订阅"审批结果"事件，本轮推理自然结束；事件到达时重新激活，此时所有需要的状态都是新读的。

### 反例二：用模式二做高频环境感知

**错误用法**：让 Agent 显式订阅"用户地理位置变化"事件，每次位置变化就激活一次推理。

**后果**：
- 用户走在路上、坐在车上、出差飞行时，位置高频变化，Session 被反复激活，**频繁打断真正的任务推理**；
- 每次激活都消耗一次完整的 LLM 调用成本，绝大多数激活的结论都是"目前没什么要做的"——纯粹浪费；
- LLM 上下文里堆满了无意义的"位置变化"消息，污染主推理质量。

**正确做法**：改用模式三。地理位置作为**环境弱订阅**注册，不单独激活 Session；只在用户真的发来一条消息（例如"帮我订机票"）时，才把"当前位置已变化到上海"注入到这条 user message 的环境 block 里。

### 反例三：用模式三替代显式等待

**错误用法**：Agent 把任务 delegate 给另一个 Agent 后，**不**显式订阅任务完成事件，而是依赖模式三——希望"任务完成"这件事作为环境变化，在用户下次说话时被自动注入。

**后果**：
- 如果用户长时间不说话，Agent **永远不会知道任务已经完成**，更不会做后续动作（催办、通知用户、链式触发下一步）；
- 即使用户最终说话了，"任务完成"以环境信息形式出现，**语义权重过低**，Agent 可能根本不把它当成需要响应的事件；
- 主动性彻底丧失——Agent 退化成"只在被叫醒时才看一眼世界"的被动系统。

**正确做法**：改用模式二。任务 delegate 同时**显式订阅完成事件**，事件到达时主动激活推理，由 Agent 自行决定下一步动作。

### 反例的共同教训

三个反例指向同一组判断准则：

1. **看等待时长**：短（秒级）选模式一，长（分钟以上）选模式二；
2. **看是否需要主动响应**：需要主动做后续动作选模式二；只是辅助理解上下文选模式三；
3. **看事件频率与必要性**：高频但单条意义低，选模式三批量浮现；低频但每条都重要，选模式二逐条响应。

## 7. 设计要点速览

1. **统一通过 user message 激活**：模式二将事件渲染成自然语言消息，避免直接塞 JSON 导致的语义漂移；系统状态机由此变得朴素而一致。
2. **显式优于隐式**：模式二相比模式一，把"等待"从 Tool 的实现细节提升为 LLM 可识别的显式意图，更适合跨人/跨 Agent 的长协作。
3. **环境感知走旁路**：模式三与历史记录压缩对偶——老信息走旁路压缩，新信息走旁路环境感知——避免主上下文被状态查询污染。
4. **Topic 是连锁中枢**：旁路 LLM 维护的 Session Topic 是 memory 召回、弱订阅注册等机制的统一入口。
5. **路径化是基础设施**：模式二的事件订阅和模式三的环境弱订阅，本质上都是**对路径标识的对象状态的订阅**——这是统一抽象的基础。memory 是否纳入同一抽象，作为延伸方向保留，不急于在工程上统一。
6. **单 Active Run + Pending Queue + Event Batch**：运行时并发由系统调度保证——Session 串行执行、事件进队列、Coalesce 与 Override 合并、批量构造 batch message 一次激活，避免被事件流反复打断。
