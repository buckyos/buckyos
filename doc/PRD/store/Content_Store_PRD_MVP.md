# Content Store PRD（MVP / 桌面端流程跑通版）

基于语音记录整理，用于指导产品、设计与 Figma 等 AI 设计工具出图

## 文档信息
- **文档版本**：V1.0
- **产品阶段**：MVP（桌面端优先）
- **文档目的**：把 Content Store 的浏览、收藏、安装、安装引导、安装进度与“我的内容”流程完整跑通
- **信息来源**：用户语音记录整理

## 1. 产品定义与目标

Content Store 是一个面向数字内容分发与安装的内容商店。MVP 阶段的核心不是做最终视觉形态，而是在桌面端先跑通“浏览 → 查看详情 → 收藏/安装 → 安装引导 → 进入我的内容 → 查看安装进度/结果”的完整闭环。

当前版本只要求桌面端可用，界面可以偏工具化、结构化，不追求品牌化与最终视觉精修。未来会快速转向移动优先设计，因此本版需要在信息架构与交互上保留可扩展性。

- MVP 首要目标：跑通安装流程，而不是追求最终 UI 完整度。
- 当前只做桌面端主流程；移动端是下一阶段，不在本次设计交付范围。
- 产品定位是开放式数字内容商店，而不是封闭商店；安装引导能力未来会以 SDK / 协议能力开放给第三方商店使用。
- 内容详情在桌面端使用浮层/模态层承载，未来移动端再切换到业内常见的独立详情页逻辑。

## 2. 设计原则（供 UI / Figma AI 使用）

- 先把流程跑通：优先保证信息层级清晰、CTA 明确、状态变化可见。
- 桌面优先：建议以 1440px 宽度作为主画板基线设计，兼容 1280px。
- 中性工具风：简洁、理性、留白充足，适配未来嵌入第三方场景的开放生态感。
- 状态统一：同一个内容在商店、详情浮层、我的内容、安装队列中的状态必须实时同步。
- 模块化：同一套卡片骨架适配多种内容类型，不同类型仅在元信息区与操作区差异化。
- 可扩展：一级 Tab、左侧分类、详情模块、安装引导选项都必须支持未来扩容。

> 建议设计语言：浅色底、深色文字、轻边框、少量强调色。不要在 MVP 阶段做重品牌化、强插画、复杂动效。

## 3. 范围与非范围

### 3.1 本期范围（In Scope）

- 一级导航：商店、我的内容（本文统一命名；录音中也出现过“我的收藏 / 我的库”表述）。
- 左侧分类 + 右侧内容卡片的双栏浏览结构。
- Header 内当前 Tab 范围内的搜索。
- 内容详情浮层（桌面端）。
- 收藏动作。
- 安装动作与支付前置占位检查。
- 安装引导 Popup Window。
- 安装完成后自动跳转到“我的内容”。
- 我的内容页顶部安装队列/进度展示。
- 我的内容页右上角通过 package meta URL 手动添加内容。
- 多内容类型的统一展示骨架与基础状态体系。

按独立页面分
- 商店页 /store_index
- 内容详情页 /content?content_id=xxx
- app安装引导 /app_install?method=install_app&url=...&params...


### 3.2 明确不做（Out of Scope）

- 完整支付流程与真实购买结算。
- 移动端最终交互形态与适配。
- AI 自动翻译简介。
- 评分、评论、排行榜、推荐算法。
- 复杂社交能力（分享、关注开发者、社区讨论）。
- 完整的卸载、升级、回滚、批量安装管理。
- 丰富媒体资产展示（截图轮播、视频预览等），MVP 可不做。

## 4. 信息架构（Information Architecture）

MVP 的一级结构以 Header 横向 Tab 切换为核心，当前只保留两个一级入口：商店、我的内容。未来会在 Header 继续扩展更多大类，但“我的内容”应始终保留且建议位于最后一个一级 Tab。

| 层级 | 名称 | 说明 |
| --- | --- | --- |
| 一级 Tab | 商店 | 浏览、搜索并进入内容详情；支持收藏与发起安装 |
| 一级 Tab | 我的内容 | 承载已收藏、手动添加、安装中、已安装、运行中的内容 |
| 左侧二级分类 | 内容分类导航 | 从上到下排列，点击后切换右侧内容区域 |
| 右侧主区域 | 内容卡片区 | 按当前分类展示 Item Card 列表/网格 |
| 全局辅助 | 搜索 | 位于 Header 右侧，仅搜索当前 Tab 范围内的内容 |

> 命名建议：为避免“我的收藏 / 我的库 / 我的内容”混用造成理解偏差，产品与设计稿统一使用“我的内容”。其中“收藏”作为一种状态或动作，而不是一级 Tab 名。

## 5. 内容分类体系

录音中对内容分类存在递进补充，综合后建议在 MVP 中采用“统一骨架 + 可扩展分类”的方案。为保证设计可落地，本文将分类拆为当前必须支持的类型与保留扩展槽位。

| 一级内容类型 | 二级说明 | MVP 处理建议 | 备注 |
| --- | --- | --- | --- |
| 应用 | 移植类应用（容器化迁移的传统应用） | 必须支持 | 示例：WordPress、n8n、Home Assistant |
| 应用 | 原生应用（基于平台原生 SDK 开发） | 必须支持 | 属于重点鼓励生态方向 |
| Agent | 可安装的 Agent / Sub-Agent | 建议支持 | 主 Agent 与 Sub-Agent 关系后续可细化，MVP 先完成安装入口 |
| Skills | Agent 能力扩展 | 预留类目 | 如开发资源不足，可先作为 Agent 下子类展示 |
| Models | 本地模型 | 建议支持并做前置条件提示 | 仅在本地 AI 能力满足时可安装 |

- Agent 类内容依赖系统至少配置一个可用的大语言模型 Provider。
- Models 类内容依赖本地 AI 能力与足够算力（如 GPU）。
- 即使系统当前能力不足，也建议允许用户浏览相关内容，但在详情页与安装按钮处明确前置条件。

## 6. 未来扩展方向（本期只预留，不设计细节）

- Media / Videos：视频、音频、图片、文件等数字媒体内容。
- Games：游戏内容。
- Books：电子书或其他阅读内容。
- 更广义目标：成为覆盖多种数字内容类型的开放式数字内容商店。

> 未来新增的大类主要通过 Header 顶部横向 Tab 扩展；本次 PRD 只要求当前结构在视觉和信息架构上可扩展。

## 7. 用户任务流

### 7.1 主流程 A：从商店安装内容

1. 进入“商店”Tab。
2. 在左侧选择分类，或使用 Header 搜索当前 Tab 内容。
3. 点击任一内容卡片，打开内容详情浮层。
4. 在详情浮层阅读简介、开发商/发行商信息、版本记录等。
5. 点击“安装”。
6. 系统先显示一个约 2 秒的资格检查占位流程（例如“正在检查是否有资格购买/安装…”）。
7. 检查通过后，弹出安装引导 Popup Window。
8. 用户按步骤完成概览确认与必要配置。
9. 点击最终确认开始安装，Popup 自动关闭。
10. 系统自动切换到“我的内容”Tab。
11. 在“我的内容”顶部安装队列与对应卡片中查看安装进度。
12. 安装完成后，内容状态变为已安装；进入详情可看到当前状态与运行状态。

### 7.2 主流程 B：先收藏，后续再安装

1. 用户在商店中进入内容详情。
2. 点击“收藏”。
3. 内容进入“我的内容”，状态为“已收藏”。
4. 用户后续在“我的内容”中再次进入详情并发起安装。

### 7.3 主流程 C：手动添加商店外内容

1. 用户进入“我的内容”。
2. 点击右上角大号“＋ 添加内容”。
3. 在弹窗中输入 package meta URL。
4. 系统解析成功后，在“我的内容”中生成一张新内容卡片。
5. 用户点击卡片进入详情，再触发后续安装流程。

## 8. 全局框架与页面结构

### 8.1 全局 Header

- 左侧：品牌 / 产品名（可极简）。
- 中部：一级 Tab —— 商店 / 我的内容。
- 右侧：当前 Tab 范围内搜索框。
- 搜索占位文案随 Tab 变化，例如“搜索商店内容”“搜索我的内容”。

### 8.2 左侧分类栏

- 纵向排列，从上到下展示分类。
- 建议包含：全部、应用、Agent、Skills、Models；如应用下需细分，可展开显示“移植类应用 / 原生应用”。
- 选中态明确，支持 hover。

### 8.3 右侧内容区

- 使用卡片网格为主，支持不同内容类型复用统一卡片骨架。
- 桌面端建议 3~4 列响应式卡片布局。
- 当内容较少时，优先保证卡片宽度与信息可读性，不必强行铺满。

## 9. 页面一：商店（Store）

商店页的目标是帮助用户按分类浏览内容，并通过卡片进入详情浮层。MVP 中商店首页不需要做复杂推荐，只需完成可理解、可搜索、可进入详情的内容浏览体验。

| 模块 | 必备内容 | 交互要求 |
| --- | --- | --- |
| Header | 一级 Tab、搜索 | 搜索只作用于商店内容 |
| 左侧分类 | 全部 / 应用 / Agent / Skills / Models | 点击分类刷新右侧内容 |
| 内容卡片 | 图标、名称、类型标签、发行方、短描述、状态提示 | 点击卡片打开详情浮层 |
| 空状态 | 暂无内容 / 无搜索结果 | 提供返回上一级分类或清空搜索的引导 |

- 商店页不强制在卡片上放“安装”按钮，MVP 可统一采用“点卡片进详情，再操作安装/收藏”的方式。
- 若设计需要弱提示，可在卡片上用状态标签标识“未安装 / 已收藏 / 已安装”。

## 10. 页面二：内容详情浮层（Desktop Modal / Overlay）

桌面端点击任何内容卡片后，默认以浮层而不是独立页面展示详情。这样可以更快跑通流程，也便于未来替换为移动端独立详情页。

| 区域 | 内容要求 | 说明 |
| --- | --- | --- |
| 顶部信息区 | 图标、名称、内容类型、发行方/开发商 | 信息一眼可识别 |
| 主 CTA | 安装 | 视觉上最显著 |
| 次 CTA | 收藏 | 用于暂不安装，先加入我的内容 |
| 简介模块 | 多语言描述 + fallback 逻辑 | 优先显示系统当前语言；无则 fallback 到默认语言 |
| 开发商/发行商模块 | 名称、简介、可信标识（可选） | MVP 不需要复杂品牌页 |
| 版本记录模块 | 历史版本 / 发布时间 / 更新说明 | 以列表或时间线形式展示 |
| 状态模块 | 当前是否已收藏、安装中、已安装、运行中等 | 与全局状态保持同步 |

- MVP 不做自动翻译；只做语言匹配与 fallback。
- 不同内容类型可在详情中插入差异化模块，但外层骨架保持一致。
- 若前置条件不满足（如未配置 LLM Provider、GPU 不足），仍可展示详情，但安装按钮需置灰或点击后提示前置条件。

## 11. 安装前占位检查（支付/资格检查 Placeholder）

由于支付流程尚未完成，MVP 在用户点击“安装”后，需要增加一个固定的过渡步骤，用于向早期用户表达：内容不一定全部免费，系统会先检查其是否有资格购买或安装。

- 触发时机：用户在详情浮层点击“安装”后立即触发。
- 表现形式：轻量浮层、Toast 或中间态弹层，持续约 1.5~2 秒。
- 默认结果：当前 MVP 一律通过，随后进入安装引导 Popup。
- 推荐文案：正在检查是否有资格购买 / 安装……

## 12. 页面三：安装引导 Popup Window（SDK 能力）

安装引导不是商店页内部的普通弹窗，而是一个可被第三方商店调用的独立 Popup Window。它需要尽量简洁、通用，并能适配不同 UI 体系。其定位更接近开放安装协议 / SDK 的统一安装引导界面。

> Popup 完成最终确认后即可关闭，不需要在 Popup 内展示长时间下载进度；下载/安装进度回到“我的内容”统一查看。

### 12.1 设计目标

- 轻量、简洁、中性，适配不同宿主产品。
- 清晰表达当前要安装的内容对象。
- 清晰展示必要配置项与开发者推荐配置。
- 完成配置后能立即进入安装准备阶段并关闭窗口。

### 12.2 推荐步骤

| 步骤 | 名称 | 内容 | 主要按钮 |
| --- | --- | --- | --- |
| Step 1 | 安装概览 | 展示应用/内容的名称、图标、简介、来源、关键提示 | 下一步 |
| Step 2 | 配置选项 | 展示必要选项、开发者推荐选项、全部可配置项（重点是目录挂载、权限、公开/登录访问等） | 上一步 / 下一步 |
| Step 3 | 确认并开始安装 | 展示最终摘要，用户确认后开始安装 | 返回 / 开始安装 |

### 12.3 配置项设计要求

- MVP 重点围绕目录挂载、访问权限、是否公开访问、是否需要登录等配置。
- 配置分为：必填项、推荐项、可选高级项。
- 开发者可给出推荐配置，界面上需要明显标识“推荐”。
- 即使未来配置项扩展，界面结构也应支持长列表与分组。

## 13. 页面四：我的内容（My Content）

“我的内容”是内容的归集页，用于承接收藏、手动添加、安装中、已安装、运行中等状态。它既是用户资产入口，也是安装过程的主状态承接页。

| 模块 | 内容要求 | 交互要求 |
| --- | --- | --- |
| Header | 一级 Tab、搜索 | 搜索只作用于我的内容 |
| 右上主操作 | ＋ 添加内容 | 打开“通过 URL 添加内容”弹窗 |
| 左侧分类 | 与商店相同或相近的内容类型分类 | 便于统一认知 |
| 顶部安装队列 | 展示正在安装/处理的内容队列 | 可显示 1 个或多个任务 |
| 内容卡片区 | 显示用户已收藏、已添加、安装中、已安装的内容 | 卡片状态实时变化 |

- 顶部安装队列可设计为固定条、吸顶模块或轻量浮层，但必须在进入“我的内容”后立即可见。
- 同一个内容的状态在顶部队列、卡片、详情浮层中必须一致。
- 内容安装成功后仍保留在“我的内容”中，不返回商店态。

## 14. 页面五：通过 URL 添加内容

MVP 需要保留一条商店外手动安装入口，以支持用户添加商店里暂时没有收录的内容。

| 元素 | 说明 |
| --- | --- |
| 入口位置 | “我的内容”右上角大号“＋ 添加内容”按钮 |
| 弹窗标题 | 添加内容 / 从 URL 添加 |
| 输入项 | package meta URL（单行输入） |
| 辅助说明 | 提示用户填写可解析的 package metadata 地址 |
| 主按钮 | 添加 |
| 成功结果 | 在“我的内容”生成一张新内容卡片，状态为未安装或已收藏 |
| 失败结果 | 提示 URL 无效、解析失败或内容已存在 |

- MVP 只暴露 URL 输入即可；录音中提到系统支持多种方法，其他方式留待后续版本扩展。
- 若添加的是系统已存在内容，应优先提示“该内容已存在于我的内容”。

## 15. 内容卡片设计规范

### 15.1 商店卡片（Store Card）

- 必须字段：图标、名称、类型标签、发行方/开发商、1~2 行短描述。
- 推荐字段：状态标签（未安装 / 已收藏 / 已安装）、前置条件标签（需要 AI Provider / 需要本地算力）。
- 交互：点击整张卡片进入详情浮层。

### 15.2 我的内容卡片（My Content Card）

- 必须字段：图标、名称、类型标签、当前状态。
- 安装中时必须展示：进度条 + 当前阶段文案。
- 已安装后建议展示：版本号、最近状态、运行状态。
- 不同类型可在底部信息区差异化，例如 Agent 显示 Provider 状态，Model 显示大小/算力要求。

### 15.3 统一卡片骨架

- 顶部：图标 + 名称 + 类型标签。
- 中部：摘要描述 / 类型关键信息。
- 底部：状态区（安装状态、进度、运行状态或前置条件）。
- MVP 优先保持骨架统一，避免为每个类型设计完全不同的卡片。

## 16. 状态体系（Global State Model）

| 状态 | 出现位置 | 说明 | 可见表现 |
| --- | --- | --- | --- |
| 未安装 | 商店、详情、我的内容（手动添加后可选） | 尚未收藏/安装 | 默认按钮为“安装” |
| 已收藏 | 商店、详情、我的内容 | 用户已加入自己的内容库但未安装 | 状态标签“已收藏” |
| 资格检查中 | 详情中间态 | 点击安装后的临时占位状态 | Spinner + 检查文案 |
| 配置中 | 安装 Popup | 用户在完成安装配置 | 步骤条高亮 |
| 安装中 | 我的内容、详情、队列 | 下载/安装正在执行 | 进度条 + 阶段文案 |
| 安装成功 / 已安装 | 商店、详情、我的内容 | 安装完成，可进一步查看运行状态 | 成功标签 / 对勾 |
| 运行中 | 详情、我的内容 | 内容已启动运行 | 运行态标签 |
| 安装失败 | 详情、我的内容、队列 | 安装过程中失败 | 错误标签 + 重试入口（如 MVP 来不及可先只展示错误） |

> 状态必须全局同步。示例：若某内容在“我的内容”中已进入安装中，则商店详情浮层再次打开时也应显示安装中。

## 17. 搜索、筛选与排序

- 搜索框位于 Header 右侧，并且仅搜索当前 Tab 内的数据集。
- 商店搜索范围：名称、短描述、开发商/发行方、标签。
- 我的内容搜索范围：名称、短描述、状态、类型。
- MVP 以搜索与左侧分类为主，不强制加入复杂排序器。
- 若需要排序，建议仅保留“默认 / 最近更新 / 最近添加”三种简单模式。

## 18. 多语言与文案策略

- 内容简介字段支持多语言存储。
- 展示时优先匹配系统当前语言。
- 若当前语言缺失，则 fallback 到内容的默认语言。
- MVP 不做 AI 自动翻译。
- 若连默认语言内容也缺失，则显示统一空状态文案，例如“暂无简介”。

## 19. 前置条件与能力约束

- Agent 内容至少要求系统存在一个可用的大语言模型 Provider。
- Models 内容要求系统具备本地 AI 能力，且在需要时满足本地算力条件。
- 前置条件不足时，建议策略为“可浏览详情、不可直接安装”，并明确展示阻塞原因。
- 前置条件提示可以出现在卡片标签、详情提示条、安装按钮禁用说明中。

## 21. 关键交互与边界情况

- 重复收藏：若内容已在“我的内容”中，点击收藏应提示“已在我的内容中”。
- 重复添加 URL：若内容已存在，应阻止重复创建卡片。
- 前置条件不足：安装按钮置灰或点击后说明原因，不应静默失败。
- 安装中再次打开详情：应看到安装状态与进度，而不是重新显示默认“安装”。
- 安装成功后：商店中的同一内容也应显示已安装状态。
- 搜索无结果：应提供清空搜索的恢复操作。
- 分类为空：显示空状态而不是空白页。

## 24. 设计补位假设（为出图而做的明确化）

- 一级 Tab 最终名称统一为“商店 / 我的内容”。
