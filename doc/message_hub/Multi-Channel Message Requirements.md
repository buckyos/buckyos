# 多渠道消息中心需求

## 1. 背景

Message Center 是 BuckyOS 的统一消息域。它需要用一套一致的产品模型，接收、展示、路由和投递来自不同来源的消息。

消息来源包括 Telegram、Control Panel 中的 Message Hub 网页体验、Agent 与用户之间的消息、系统服务消息，以及未来可能接入的第三方应用或外部通信平台。

目标不是为每个渠道分别建设一套消息系统。每个渠道只负责适配自己的账号模型、消息格式和传输行为。Message Center 负责提供统一的身份、联系人、会话、权限、投递和通知模型。

## 2. 目标

Message Center 必须提供：

- 跨渠道统一的消息身份。
- 统一的收件箱、发件箱、请求箱和群聊视图。
- 面向外部平台账号和 BuckyOS DID 的统一联系人管理。
- 消息接收和投递前的统一访问控制。
- 出站消息的统一投递规划。
- 面向用户、应用和 Agent 的统一实时变更通知。
- 面向未来渠道的清晰扩展模型。

## 3. 非目标

Message Center 不需要把每个外部平台的全部细节都提升为一等产品概念。

Message Center 不需要让浏览器客户端直接访问原始服务边界。面向浏览器的应用应通过可信后端包装层访问，由后端解析当前用户范围并执行产品层权限检查。

Message Center 不需要让 Agent 理解 Telegram、网页聊天、邮件或其他渠道。Agent 应只处理来自自身消息视图的标准化消息。

## 4. 核心概念

### 4.1 统一身份

系统内部的每个消息参与方都应表示为 BuckyOS DID。

外部账号，例如 Telegram 用户或群组，应映射为联系人。一个联系人可以拥有一个或多个渠道绑定，例如 Telegram、网页、邮件或未来应用自定义账号。

系统应记录足够的绑定信息，用于后续选择合适的渠道回复该联系人。

### 4.2 统一消息

所有渠道的入站消息进入 Message Center 前，都应转换为系统标准消息。

标准化消息应表达：

- 发送方。
- 接收方或接收方列表。
- 消息类型，例如聊天、事件、群消息或通知。
- 文本内容。
- 必要时的附件引用。
- 可用时的会话或线程上下文。
- 用于诊断和路由的来源渠道信息。

渠道元数据可以保留，但业务逻辑应主要依赖统一消息模型。

### 4.3 统一消息视图

同一条消息可以根据不同参与方出现在不同视图中：

- 发送方发件箱。
- 接收方收件箱。
- 群聊收件箱。
- Agent 收件箱。
- 需要用户确认的请求箱。
- 面向具体渠道投递的发送队列。

这些视图应允许每个 owner 独立管理状态，例如未读、阅读中、已读、等待发送、已发送、发送失败、归档或删除。

每个消息视图记录应能承载足够的产品层状态，用于支持列表展示、会话归类、投递进度、错误恢复和用户侧状态管理。消息本体应保持稳定，阅读状态、投递状态、归档、删除、重试和会话归类等可变状态应属于具体视图。

### 4.4 幂等与可恢复性

Message Center 必须支持幂等写入语义。同一来源事件、同一发送请求或同一消息对象被重复提交时，不应产生重复的用户可见消息或重复的投递任务，除非调用方明确表达需要创建新的业务消息。

幂等语义应同时覆盖：

- 入站渠道重复上报。
- 出站发送请求重复提交。
- 渠道投递任务被重复扫描或重试。
- 服务重启后重新加载未完成任务。

系统应保留足够状态，使订阅方、Agent、网页应用或渠道适配器在错过实时通知后，能够通过重新查询消息视图恢复进度。

### 4.5 消息查询与记录加载

Message Center 除了队列式消费能力，还必须提供面向产品界面的查询能力。

查询能力应支持：

- 按 owner 和消息视图列出历史记录。
- 按时间分页加载。
- 按状态过滤，例如未读、阅读中、等待发送或发送失败。
- 预览消息视图而不改变状态。
- 获取单条消息视图记录。
- 在权限允许时加载消息本体。

查询能力应允许调用方按需选择是否携带完整消息内容。列表场景可以只返回轻量记录，详情或处理场景再加载消息本体。

### 4.6 会话归类

Message Center 应支持将消息记录归类到用户或 Agent 可理解的会话中。

会话归类应满足：

- 能使用标准消息中的线程、主题或相关上下文生成会话标识。
- 能在消息创建后由可信后端或 Agent 更新会话归属。
- 同一条消息在不同 owner 的视图中可以拥有独立会话归类。
- 会话归类变化应触发消息视图变更通知。

会话归类是产品层状态，不应改变消息本体的身份。

## 5. 入站消息需求

当某个渠道收到外部消息时，渠道适配器必须将其转换为标准化系统消息，并提交给 Message Center。

Message Center 必须：

- 解析或创建发送方联系人身份。
- 解析目标身份或群组身份。
- 保留必要来源上下文，例如平台、外部账号 ID、聊天 ID、消息 ID 和附件引用。
- 在投递前应用访问策略。
- 将被接受的消息投递到正确的收件箱或群聊收件箱。
- 当策略要求用户确认时，将低信任消息路由到请求箱。
- 拒收或丢弃被联系人策略屏蔽的消息。
- 在消息记录创建或更新后发布消息视图变更通知。

入站提交必须支持来源幂等。渠道适配器应提供足够的来源标识，使 Message Center 能识别重复事件，并把重复提交处理为同一次业务投递。

Telegram 是一个具体入站渠道。未来渠道应遵循同一产品契约。

## 6. 出站消息需求

用户、Agent 和系统服务应通过 Message Center 使用统一的接收方与内容模型发送消息。

发送方不应必须知道最终传输渠道，除非它主动指定优先渠道。

Message Center 必须：

- 保存发送方的发件箱视图。
- 解析接收方的首选或可用渠道绑定。
- 为一个或多个渠道构造投递计划。
- 创建面向渠道的投递任务。
- 跟踪投递状态、重试状态、外部消息 ID 和失败详情。
- 在状态变化时发布消息视图变更通知。

如果没有可用渠道绑定，Message Center 应返回明确失败原因，或将消息置于可由用户动作恢复的状态。

出站发送必须支持发送请求幂等。用户、Agent 或系统服务重复提交同一次发送意图时，不应产生重复的发件箱记录或重复的外部平台投递。

渠道投递任务应支持可恢复重试。投递失败时，系统应区分可重试失败和最终失败，记录失败原因，并在需要用户干预时让该状态能被产品界面发现。

## 7. Web Message Hub 需求

Web Message Hub 应作为 Message Center 之上的受控产品入口。

网页应用必须：

- 根据已认证会话解析当前用户的消息范围。
- 列出当前用户可见的联系人。
- 列出用户收件箱和发件箱中的消息。
- 以已认证用户身份发送文本消息。
- 接收消息级实时更新。
- 按会话查看消息历史。
- 打开单条消息或消息记录详情。
- 更新用户视图中的阅读状态、会话归类、归档或删除状态。

网页应用不得把原始 owner 范围或 contact-manager 范围选择权暴露给浏览器。范围解析应属于可信后端包装层。

实时更新应是消息级变更，不要求提供 AI token 级流式输出。

网页应用使用的后端包装层必须负责把登录用户映射到可访问的消息范围，并阻止浏览器直接指定其它 owner 的消息视图或联系人范围。

## 8. Telegram 需求

Telegram 集成应作为一种渠道适配器。

它必须：

- 在能力允许的情况下接收 Telegram 私聊、群聊或频道消息。
- 将 Telegram 用户和聊天映射为系统联系人或群组身份。
- 将 Telegram 消息转换为标准化系统消息。
- 保留用于路由、展示、诊断和幂等处理的 Telegram 元数据。
- 尽可能将附件保存为引用。
- 将标准化消息提交给 Message Center。
- 当选中的投递路径使用 Telegram 时，将系统出站消息发送回 Telegram。
- 将投递成功或失败结果回报给 Message Center。

Telegram 特有细节不应泄漏到 Agent 行为或通用网页聊天行为中，除非作为可选展示或诊断元数据。

Telegram 渠道应支持独立的运行配置，包括是否启用、是否接收入站、是否允许出站、账号绑定和网关运行方式。配置变化应能在服务运行中被重新加载，并应同步影响后续入站接收和出站投递。

Telegram 出站投递在缺少发送账号、目标地址或必要上下文时，必须返回明确失败原因，而不是静默丢弃。

## 9. Agent 与 AI 需求

Agent 应通过 Message Center 的消息视图消费消息，而不是直接依赖某个渠道集成。

Agent 必须：

- 观察自己的收件箱、群聊收件箱和请求箱。
- 在收到变更通知后拉取标准化消息。
- 将消息路由到合适的 Agent 会话。
- 将已处理消息标记为已读或其他已处理状态。
- 通过自身行为和 AI Runtime 生成回复。
- 将回复提交回 Message Center 进行投递。
- 在需要时读取单条消息本体或消息记录详情。
- 将消息关联到自身会话，并在处理后更新会话归类或处理状态。

Agent 不应需要知道原始消息来自 Telegram、网页聊天还是未来应用。除非 Agent 主动请求优先渠道，否则回复路由应交给 Message Center 决定。

## 10. 实时通知需求

Message Center 必须在消息记录创建或更新时发布变更通知。

订阅方可以包括：

- Web Message Hub。
- OpenDAN Agent。
- 系统服务。
- 未来应用。

通知应表达某个消息视图发生变化，并提供足够信息让订阅方加载更新后的记录。通知不需要携带完整消息内容。

订阅方应将通知视为提示，并能通过重新加载消息视图恢复状态。

通知内容应至少能表达：

- 发生变化的 owner 范围。
- 发生变化的消息视图类型。
- 发生变化的消息记录。
- 当前状态或更新时间。
- 变化类型，例如创建、状态更新、投递结果更新或会话归类变化。

通知发布失败不应阻断消息写入。订阅方必须能通过查询能力补偿遗漏通知。

## 11. 联系人与权限需求

Message Center 必须使用联系人信息同时决定入站接收和出站投递。

联系人模型应支持：

- 手动联系人。
- 从外部账号自动推断出的联系人。
- 单个联系人拥有多个渠道绑定。
- 首选绑定选择。
- 好友、陌生人、临时授权和屏蔽等访问等级。
- 绑定到具体上下文的临时授权。
- 导入、更新、合并和屏蔽操作。

入站消息进入普通收件箱前，必须根据目标 owner 的联系人策略进行检查。

出站消息必须使用联系人绑定选择投递路径。

联系人范围必须支持按 owner 隔离。系统范围联系人和具体用户范围联系人可以并存，访问策略和投递绑定应按当前消息范围解析。

Message Center 应能从 Zone 内用户配置同步系统内部联系人。已激活的 Zone 用户应能作为联系人出现在需要通信的用户范围内，并带有可用于权限判断和投递规划的基础身份信息。该同步应是可重复执行的，不应破坏用户后续维护的联系人信息。

## 12. 群消息需求

Message Center 应支持将群组类身份作为消息参与方。

对于群消息，系统应：

- 维护群级收件箱。
- 将原始消息保留为一条逻辑消息。
- 根据需要创建 reader 视图或阅读状态。
- 支持基于群订阅者或群成员的投递。
- 避免让每个渠道各自发明一套群语义。

外部平台群聊应尽可能映射为系统群组身份。

### 12.1 Self-host Group 需求

Message Center 应支持由当前 Zone 托管的可消息化群实体。Self-host group 不只是联系人标签或 UI 分类，而是拥有独立 DID、成员关系、群消息视图和访问策略的消息参与方。

Self-host group 必须支持：

- 创建群实体并维护群资料。
- 读取和更新群公开信息。
- 邀请成员、提交成员证明、申请加入、审批加入、拒绝加入、移除成员和更新成员角色。
- 维护群成员状态和成员角色。
- 支持 subgroup 或子集合，用于表达群内结构。
- 支持递归成员展开，并能限制展开深度、避免循环和保留展开快照。
- 支持面向投递、权限检查、公开展示和归属表达的不同展开目的。
- 支持群访问检查，用于判断某个 DID 是否可读取、发送或管理群相关资源。

群成员可以是普通 DID，也可以是另一个群 DID。系统处理群 DID 时应默认把它视为一个完整实体，只在投递、权限判断或展示需要时展开成员。

### 12.2 群策略需求

Self-host group 应支持独立的集合策略和归属策略。

集合策略应表达：

- 是否允许嵌套群。
- 不同场景下是否允许展开成员。
- 成员可见性和公开展示边界。
- 加入、邀请和成员确认规则。

归属策略应表达群 DID 在作者、协作主体、收益归属主体或权限主体等场景中的使用规则。

策略变化应被记录为群状态变化，并能被管理界面和审计流程观察到。

## 13. 持久化与存储需求

Message Center 必须使用持久化存储保存消息视图、联系人、群状态、群成员、群事件和投递状态。服务重启不应导致消息历史、未完成投递、联系人策略或群管理状态丢失。

持久化能力应满足：

- 支持按 owner 隔离消息视图和联系人数据。
- 支持按时间、状态、消息类型和发送方查询消息视图。
- 支持跨服务重启恢复等待投递、正在处理或失败待重试的任务。
- 支持保存消息本体引用，并在需要时加载消息本体。
- 支持保存群成员、群事件、群策略和群展开结果。
- 支持未来替换或扩展底层数据库，而不改变产品层消息模型。

## 14. 服务运行与配置需求

Message Center 应支持运行期配置管理。配置变化应能影响渠道启用状态、渠道账号绑定、入站和出站能力、网关连接方式以及联系人同步范围。

服务运行需求包括：

- 启动时加载当前配置并初始化启用的渠道。
- 支持运行中重新加载配置。
- 配置重新加载时应停止不再启用的渠道实例，并启动或更新新的渠道实例。
- 配置错误应返回明确原因，并尽量避免破坏已有可用渠道。
- 后台投递任务应持续扫描可发送队列，并根据投递结果更新消息视图。
- 后台投递流程的失败不应导致 Message Center 主服务不可用。

## 15. 未来渠道扩展需求

新渠道应能通过提供以下能力接入：

- 将外部账号映射为联系人身份。
- 将入站消息转换为标准化消息。
- 实现出站投递。
- 回报投递结果。
- 可选的附件导入和导出。
- 可选的实时或轮询行为。
- 可选的运行期配置加载和账号绑定管理。

新增渠道不应要求改变核心消息视图模型。

潜在未来渠道包括：

- Email。
- Slack。
- Webhook。
- 移动端推送或 App 内聊天。
- 第三方 Agent 平台。
- 其他 BuckyOS 应用。

## 16. 验收标准

多渠道消息设计在满足以下条件时可视为可接受：

- Telegram 消息、网页聊天消息和未来应用消息都能进入同一套统一消息模型。
- 用户能通过 Message Hub 查看混合渠道历史。
- Agent 能在没有渠道专用逻辑的情况下处理消息。
- 回复能通过接收方可用的渠道绑定投递。
- 被屏蔽或低信任发送方在进入普通收件箱前被策略处理。
- 消息变化能通过统一通知被观察到。
- 新渠道能作为适配器加入，而不需要重新设计核心消息域。
- 重复入站事件和重复出站请求不会产生重复用户可见消息或重复外部投递。
- 服务重启后仍能恢复消息历史、联系人策略、群状态和未完成投递任务。
- Web Message Hub 和 Agent 能按会话加载消息、读取详情并更新各自视图状态。
- Self-host group 能作为独立 DID 消息实体被创建、管理、投递和权限检查。
- 运行期配置变化能被服务重新加载，并正确影响渠道实例和联系人同步。
