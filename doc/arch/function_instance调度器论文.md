# 基于 Function Instance 的分布式调度系统：

## 一种以不可变状态、表达式求值与全局记忆化为核心的理论框架

### 摘要

随着 Agent-Human-Loop、云原生工作流以及大模型驱动自动化系统的发展，传统以命令式任务（OPTask）为基本原语的分布式调度架构，越来越难以同时满足可预测性、可审计性、可治理性和跨工作流复用能力。本文提出一种以 **Function Instance（函数实例）** 为调度原语的分布式系统理论框架：将复杂工作流问题还原为表达式树求值问题，将可变系统状态改写为版本化不可变状态序列，并通过 Named Object 与全局 Memoization 实现跨任务、跨工作流乃至跨系统边界的结果复用。

本文的核心观点是：**工作流引擎不应侵入调度器语义，调度器也不应理解业务层工作流结构；二者的正确分工，是由编排层在用户态管理依赖，由调度层在内核态或基础设施层只负责函数求值分配。** 在这一前提下，TODO 不再被看作独立系统，而应视为 TaskMgr 面向人类的一种视图；Step 不再被视为“要执行的一段过程”，而是一个具有输入/输出 Schema、资源约束、副作用边界与终止性约束的受限函数。基于该思路，本文进一步论证：约束不是能力削弱，而是信息增益；不可变状态不是表示负担，而是可推理性与治理能力的来源；全局缓存不应附着于调度器，而应下沉为基础设施层的命名对象系统。

本文最终给出一个四层架构：**人机交互层 / TaskMgr 视图层 / 编排层 / 调度与 Named Object 基础设施层**，并说明其在容错恢复、审计治理、类型安全、缓存复用与人类介入语义上的统一性。该框架并非对传统工作流系统的局部优化，而是一种从“过程调度”向“函数求值分配”转变的系统范式。

**关键词**：分布式调度；Function Instance；工作流引擎；不可变状态；Memoization；TaskMgr；Agent-Human-Loop；内容寻址计算

## 1. 引言

分布式系统中的“调度”长期被理解为一个资源分配问题：在什么时间、将什么任务、放到哪台机器上执行。然而，当工作流系统、自动化 Agent 系统以及人机协作型任务系统逐步融合后，这一定义开始显得不足。原因在于，系统需要管理的不再只是“执行”，而是：

1. 任务的来源与上下文；
2. 任务之间的依赖与因果关系；
3. 执行过程中的人类确认、回退与审阅；
4. 状态跨设备汇聚后的统一观测；
5. 历史可追溯、结果可复用、失败可恢复；
6. 系统层面对错误操作的事前阻断，而不仅是事后追责。

传统架构通常试图在调度器内不断注入扩展能力，以支持数据局部性、步骤亲和性、业务优先级乃至特定工作流语义。但实践反复表明：**调度器越聪明，往往越脆弱；越可扩展，往往越不可预测。** 这是因为调度器承担的是一个高度全局化、强一致性敏感且时延极其敏感的核心职责。任何额外语义都可能破坏其快速、确定、可解释的基本特征。

因此，本文提出一个相反方向的设计原则：**不升级调度器，而是降级调度原语。**
与其让调度器理解越来越复杂的业务，不如让提交给调度器的对象变得更小、更纯、更可推理。由此，本文将调度器的基本原语从传统的 OPTask，替换为 Function Instance；将工作流由“流程图上的步骤推进”重述为“表达式树上的函数求值”；将节点状态从“当前环境”的隐含背景改写为“带版本号的命名状态值”；将缓存去重从调度层移交给 Named Object 层；将人类视为一种特殊的求值条件而非系统外部例外。

该理论框架的价值不只是工程实现层面的“更优雅”，而在于它给出了一个可统一解释工作流、调度、缓存、审计、治理和人机协作的系统语言。

## 2. 研究问题与核心命题

本文关心的不是“如何做一个功能更多的工作流引擎”，而是更基础的问题：

### 2.1 研究问题

**RQ1：** 在不修改调度器核心语义的前提下，工作流系统能否获得足够强的表达能力与优化空间？
**RQ2：** 分布式执行中的状态、一致性、缓存与重试，是否可以通过更受限的原语设计自然统一？
**RQ3：** 人类介入、审计治理与系统可验证性，能否从“外挂能力”变成系统结构内生性质？

### 2.2 核心命题

本文提出以下命题：

**命题一：** 调度器不应理解工作流，只应理解函数实例。
**命题二：** 工作流引擎应位于用户态，负责依赖管理与投递时序，而非资源分配。
**命题三：** 可变状态必须被版本化为不可变对象，否则调度语义无法获得可推理性。
**命题四：** 结果缓存必须以 Named Object 的形式作为基础设施层能力存在，而非调度器的附属优化。
**命题五：** 人类介入、本地脚本执行、副作用处理，都可以被纳入统一的函数式求值框架，只要其边界被显式标注。

这些命题共同构成了本文系统设计的理论基础。

## 3. TODO、Task 与 TaskMgr：从独立系统到统一视图

### 3.1 TODO 系统的局限

传统 TODO 系统隐含一个强假设：**人是任务执行的主体。**
但在 Agent-Human-Loop 场景下，执行主体往往是 Agent、服务或调度系统，人类的作用则更接近于审查者、确认者和仲裁者。这导致经典 TODO 模型与实际运行模型之间存在结构性错位。

在真实系统中，待办事项往往不是从“空白表单录入”开始的，而是在沟通、事件、消息、告警、工作流推进中自然浮现。用户真正关心的不是某一个脱离上下文的“待办条目”，而是一个任务运行的全貌：它为何产生、目前运行到哪里、谁在等待谁、下一步可能发生什么、自己需要做什么决策。

### 3.2 Task 作为统一任务语义

因此，更合适的抽象不是 TODO，而是 **Task**。
Task 不是一个勾选框，而是一个可被观察、可被追踪、可被交互的运行实体。对于复杂自动化系统，一个 Task 往往对应一个完整的 Workflow Run，或者至少是其可独立观测的一段子运行。

### 3.3 TaskMgr 作为统一状态管理器

在这一语义下，TODO 只是 TaskMgr 的某种投影视图。TaskMgr 的职责不是执行任务，而是：

* 聚合跨设备、跨组件、跨步骤的状态；
* 为不同角色提供差异化视图；
* 提供统一 taskid 寻址能力；
* 让 UI 可以订阅任务状态演进；
* 让人类操作能够被正确投递回编排系统。

于是，同一个底层运行过程可以投影成多种视图：

* 面向 Agent：当前待执行的 Step；
* 面向人类：`waiting_human` 的任务集合；
* 面向运维：失败、重试、资源占用；
* 面向审计：不可篡改的历史轨迹。

### 3.4 理论意义

这一步看似只是产品视角调整，实际上改变了整个系统边界：
**任务管理从“记录待办事项”转向“统一表达运行状态”。**
也正因此，TaskMgr 不再是工作流引擎的附属组件，而是人机协作系统中的状态观察与操作中枢。

## 4. 调度器与工作流引擎的边界：为何“调度器不改”是根本原则

### 4.1 调度器的本质职责

调度器在分布式系统中的根本职责，可以概括为一句话：
**对可执行对象进行快速、确定、可预测的分配。**

这一定义中最重要的不是“分配”，而是“快速、确定、可预测”。调度器一旦吸收过多业务语义，就会面临三个问题：

1. **全局状态爆炸**：需要理解过多上下文；
2. **扩展点冲突**：不同策略相互干扰；
3. **一致性代价失控**：更多信息意味着更重协调。

### 4.2 为什么调度器不适合做工作流优化

很多系统设计会倾向于让工作流引擎“下沉”到调度器，例如：

* 为特定步骤做亲和性调度；
* 识别跨步骤的数据局部性；
* 在调度器中内置 DAG 感知；
* 让调度器决定分支策略或重试策略。

这些能力看似提高全局优化，实则将工作流的高层语义耦合进调度内核。其结果通常是：

* 调度器实现复杂化；
* 故障归因困难；
* 不同类型任务的公平性下降；
* 后续扩展越来越依赖特例。

### 4.3 正确分工：编排在用户态，调度在基础设施层

本文坚持以下边界：

* **工作流引擎（编排组件）**：负责依赖关系、步骤推进、分支、重试、人类交互；
* **调度器**：负责单个函数实例的求值分配；
* **Named Object**：负责已求值结果的存储与重用；
* **TaskMgr**：负责状态聚合与视图呈现。

这样做的好处在于，每一层都只处理本层最稳定的问题，避免跨层语义污染。尤其关键的是：
**工作流引擎成为纯粹的用户态消费者，它不拥有特殊权限，也不要求调度器为其改变调度语义。**

这意味着整个系统的可维护性来自边界清晰，而不是能力集中。

## 5. 从 OPTask 到 Function Instance：调度原语的理论转向

### 5.1 OPTask 的问题

传统 OPTask 的典型形式是：“在某节点上执行某段逻辑”。
例如：在 `node3` 上运行 `deploy.sh`。

这种表示方式具有几个根本缺陷：

1. **隐式状态依赖**：脚本依赖机器当前状态，但当前状态未显式进入参数；
2. **副作用不可见**：脚本改了什么很难抽象；
3. **失败模式不透明**：是环境问题、网络问题、脚本逻辑问题还是权限问题，难以统一表达；
4. **不可预测性强**：运行时长、资源消耗、结果结构都难以静态推断；
5. **版本管理困难**：同名脚本随时可能被替换。

在这种语义下，调度器看到的是一个不可分析的黑盒命令，因此天然丧失优化能力。

### 5.2 Function Instance 的定义

为解决上述问题，本文引入 **Function Instance** 作为新的调度原语。
Function Instance 可以形式化表示为：

[
FI = (funid,\ params,\ meta)
]

其中：

* `funid`：函数身份标识，通常由函数内容或注册版本确定；
* `params`：输入参数集合，必须是可寻址、可序列化、可验证的值或引用；
* `meta`：包括资源需求、幂等性、副作用类型、超时、重试约束等元信息。

直观地说，调度器不再接收“命令”，而接收“求值请求”。

### 5.3 原语降级的意义

这被称为“原语降级”，因为它不是给调度器更强对象，而是给它更小、更纯的对象。
例如：

* 过去：`在 node3 上跑 deploy.sh`
* 现在：`exec(hash("deploy.sh"), node_state("node3", 3701))`

这一区别是根本性的。后者把“脚本版本”和“节点状态版本”都显式化了。它不再依赖一个模糊的“当前环境”，而是依赖两个确定对象。因此，这个求值请求具备：

* 可审计性；
* 可缓存性；
* 可重复性；
* 可比较性；
* 可拒绝性（若类型不匹配则不能执行）。

### 5.4 约束产生信息

Step 若具备以下特征：

* 显式输入 Schema；
* 显式输出 Schema；
* 显式资源需求；
* 显式副作用标记；
* 无隐式共享状态；
* 保证终止；

那么调度器和编排器将获得比 OPTask 多得多的系统信息。这些信息不是附加元数据，而是优化与治理的前提。例如：

* 幂等任务可以安全重试；
* 输入确定的任务可以全局缓存；
* 资源声明使容量规划更准确；
* 终止性保证使调度系统无需担心逻辑死循环吞噬资源。

因此，**受限不是牺牲，而是将原本隐藏在运行时的不确定性，提前暴露为系统可利用的信息。**

## 6. 可变状态的消解：版本化不可变状态模型

### 6.1 可变状态为何破坏调度语义

传统系统中，同一条命令在不同时间执行可能得到完全不同结果，因为它依赖的是“某节点当前状态”。然而“当前状态”本身是一个随时间变化、难以冻结、难以复用、难以引用的概念。

一旦输入不是固定值，系统就难以判断：

* 两次执行是否等价；
* 一个结果是否可以复用；
* 一个重试是否安全；
* 并发操作是否冲突；
* 历史是否可重放。

### 6.2 版本化状态的定义

本文将状态表示为：

[
node_state(node_id,\ version)
]

即某节点在某版本上的逻辑状态。该状态不是节点的物理全量快照，而是一个**具有确定 Schema 的逻辑状态对象**。其关键性质包括：

1. 一旦生成，不再变化；
2. 可被引用、哈希、缓存和审计；
3. 只能通过前一状态和一个函数实例推导出下一状态。

状态转移可表示为：

[
node_state(node3, 3702) = exec(hash("deploy.sh"), node_state(node3, 3701))
]

这样，系统内部不再存在“可变对象”，只存在状态序列。

### 6.3 理论收益

这一改写带来四项关键收益：

**第一，冲突检测结构化。**
若两个操作都依赖 `node_state(node3, 3701)`，则它们在依赖图中天然显现竞争关系，而不需要依赖隐式锁或人工约束。

**第二，历史可重放。**
任何状态都由前态与函数调用推导而来，因此历史本身就是一个可解释的状态生成链。

**第三，缓存更可靠。**
当状态版本进入参数时，“相同输入得到相同输出”的条件得以恢复。

**第四，治理能力增强。**
脚本只对特定版本的状态 Schema 有效，一旦 Schema 演进导致参数类型不匹配，系统会自然拒绝执行旧脚本。这使过时逻辑暴露为类型错误，而不是线上事故。

## 7. 工作流的本质：从流程推进到表达式求值

### 7.1 DSL 与表达式树的等价性

工作流 DSL 通常用节点和边来描述流程，但从计算本质看，它与表达式树是等价的。可以做如下对应：

| 工作流 DSL 概念 | 表达式求值概念       |
| ---------- | ------------- |
| Step       | 函数调用节点        |
| Edge A → B | B 的参数依赖 A 的结果 |
| Reference  | 对已有结果的字段读取    |
| branch     | 模式匹配或条件选择     |
| parallel   | 多参数独立求值       |
| for_each   | 对集合的映射        |

因此，一个工作流不必被看成“控制流图”，更可以被看成“待求值的结构化表达式”。

### 7.2 求值视角的优势

当工作流被改写成表达式树，很多传统工作流问题会自动变得简单：

* **依赖关系**不再是调度器额外维护的元数据，而是表达式结构本身；
* **可缓存性**来自子表达式的可重用；
* **局部失败恢复**可以从失败子表达式重新开始；
* **部分求值**允许系统只为用户真正关心的最终值展开计算。

### 7.3 Pull 与 Push：两种求值驱动模式

在表达式视角下，系统可以采用两种基本求值方式：

#### （1）Read-driven / Pull / Lazy Evaluation

从最终结果出发向后追溯，只在真正需要时求值。
优点：

* 自然节省资源；
* 与缓存高度契合；
* 适合大规模 `for_each` 场景；
* 不必一口气提交所有子任务。

#### （2）Write-driven / Push / Eager Evaluation

从输入变化出发向前推进，尽快触发后继步骤。
优点：

* 实时性更高；
* 更适合事件驱动；
* 前端交互延迟更低。

本文认为，这两者并不冲突。真正关键的是：
**无论采用哪一种推进策略，底层对象仍然应是可独立求值的 Function Instance。**

### 7.4 人类介入的求值语义

在传统系统中，人类审批常被视为工作流中的特殊节点。但在本文框架下，它可以被解释为一种特殊 thunk：
它不是“不能算”的节点，而是“其 force 条件由人类决策满足”的节点。

例如：

[
human_confirm(x)
]

其输入 `x` 可以已就绪，但其结果只有在用户做出 approve / reject / revise 决策后才可产生。这意味着人类介入并非框架例外，而是统一求值语义的一部分。

## 8. 副作用、幂等性与受控不纯性

### 8.1 纯函数并不足以覆盖真实世界

任何真实系统最终都要触达外部世界：发送消息、部署服务、写数据库、调用 API、变更权限。这些操作都带有副作用，不可能被完全当作数学纯函数处理。

因此，系统设计的目标不应是“消灭副作用”，而是**将副作用显式化、边界化、可审计化。**

### 8.2 幂等性作为调度协议的一部分

Function Instance 需要显式声明：

* `idempotent: true`
  表示同样输入重复执行，不会产生额外外部影响。此类任务适合重试、投机执行和全局缓存。

* `idempotent: false`
  表示多次执行可能造成多次副作用，系统需限制重复 force，并进行收据管理。

### 8.3 side_effect_receipt：副作用收据

为统一处理副作用，本文引入 **side_effect_receipt** 概念。
它表示：某个不纯操作已经被执行，并返回一份可验证、可存档、可引用的收据。调度器或编排器不必直接“相信外部世界”，而是将副作用的完成结果表示为新的命名对象。

如此一来，即使任务需要重试，系统也可先检查收据是否存在，从而避免重复调用外部接口。这使“至少一次”交付环境中的幂等保证有了更强的系统支撑。

### 8.4 理论意义

这实际上是将副作用重新纳入求值体系。
副作用不是对函数模型的破坏，而是通过“显式标记 + 收据对象化”的方式被重新约束。因此，整个系统仍可以在较大范围内维持函数式推理能力。

---

## 9. Named Object 与全局 Memoization：缓存为何必须下沉为基础设施

### 9.1 缓存不应属于调度器

许多系统把去重、结果缓存、重复提交优化交给调度器实现。但这会造成两个问题：

1. 调度器需要理解“相同任务”的判定逻辑；
2. 缓存只在调度器队列或短生命周期内生效，不能跨工作流复用。

因此，本文认为缓存应从调度层剥离，下沉为独立的命名对象系统。

### 9.2 命名规则

给定：

[
name = funid(hash(params))
]

则其结果对象拥有一个全局唯一名称。调用者在提交 Function Instance 之前，先检查该 name 是否已有结果：

* 若已存在：直接复用；
* 若不存在：才投递给调度器。

于是，调度器永远只看到真正需要求值的对象，而不用承担缓存判定职责。

### 9.3 跨工作流复用

全局 Memoization 的真正价值在于跨工作流：

* 工作流 A 计算过的 `embedding(doc_x)`；
* 工作流 B 需要相同结果时，直接复用；
* 无需再次排队、调度、执行。

这使系统从“单次运行优化”跃迁到“全局知识积累”。本质上，Named Object 层承载的不是简单缓存，而是**面向内容寻址的结果知识库**。

### 9.4 与表达式求值的统一

一旦子表达式结果可被命名、缓存和复用，整个工作流就不再是一次性过程，而是一组可持久化、可组合、可共享的已知值与待求值值。这为系统带来更高阶的优化可能，例如：

* 子表达式公共子项消除；
* 增量计算；
* 中间结果再利用；
* 跨版本比较；
* 运行历史回放。

## 10. 四层架构：完整系统的分工模型

为了让上述理论真正落地，本文提出四层分工架构。

### 10.1 第一层：人类 / UI 交互层

该层负责展示和操作，不承担求值与调度职责。
人类可能看到不同界面：

* TODO 视图；
* 看板视图；
* 对话式协作视图；
* 审计与时间线视图。

它们都是对 TaskMgr 状态投影的不同渲染。

### 10.2 第二层：TaskMgr 状态汇聚层

TaskMgr 的职责是“让人看到算到哪了”。它聚合来自编排组件、调度器、执行节点乃至外部交互事件的状态，并通过统一 taskid 或 runid 建立跨组件一致观察。

TaskMgr 不决定算什么，也不决定在哪算；它只负责：

* 生命周期状态汇聚；
* 统一观测与订阅；
* 对人类操作的接收与转发；
* 审计轨迹沉淀。

### 10.3 第三层：编排组件（工作流引擎）

编排层持有 DSL 或表达式树，并负责：

* 依赖关系管理；
* 何时提交哪个 Function Instance；
* 分支选择；
* 重试策略；
* 人类交互节点管理；
* 与 Named Object 的预查和结果监听。

这意味着编排层是“什么时候该算什么”的唯一拥有者。

### 10.4 第四层：调度器 + Named Object 基础设施层

这层可进一步分为两部分：

#### 调度器

负责：

* 从队列取 Function Instance；
* 检查参数是否就绪；
* 选择合适节点；
* 执行并返回结果；
* 报告执行进度与错误分类。

#### Named Object

负责：

* 结果对象存储；
* 全局寻址；
* memoization；
* 引用解析；
* 供编排器与调度器查询。

### 10.5 架构边界的重要性

该四层设计的关键不在于“功能划分完整”，而在于：

* 每一层都拥有单一、稳定、长期不变的职责；
* 上层不侵入下层决策；
* 下层不理解上层业务语义；
* 系统优化通过对象表达能力提升，而非跨层特例堆积来实现。

## 11. 形式化接口与执行协议

为了将该理论框架进一步工程化，可以给出一个简化的接口模型。

### 11.1 Function Instance 结构

[
FI = {
funid,\ params,\ guards,\ idempotent,\ timeout,\ retry_policy,\ effect_type
}
]

其中：

* `guards`：资源约束与执行前提；
* `idempotent`：幂等声明；
* `effect_type`：纯函数 / 外部副作用 / 状态迁移型函数；
* `retry_policy`：编排层可参考的失败处理偏好。

### 11.2 调度器执行循环

调度器核心逻辑可以简化为：

```text
loop:
    fi = dequeue()
    if params_ready(fi):
        node = select_node(fi.guards)
        result = execute(node, fi)
        named_object.put(name(fi), result)
        taskmgr.report(fi, result.status)
    else:
        reject(fi)
```

这里的 `reject` 并不意味着失败，而意味着“参数未就绪，不应由调度器等待”。等待依赖满足属于编排层职责。

### 11.3 错误分类协议

为避免调度器与编排器职责混淆，调度器应返回结构化错误，例如：

* `node_failure`
* `execution_error`
* `timeout`
* `resource_unavailable`
* `param_not_ready`

由编排层依据 Step 的语义与策略决定是否重试、换节点、回退、升级给人类或终止整个 Run。

## 12. 治理、安全与审计：从事后追责到事前阻断

### 12.1 Hash 不是版本号，而是准入机制

脚本或函数通过 hash 注册后，系统只承认被正式命名的逻辑对象。未注册的脚本在系统语义中不存在，因而无法被合法调度。

这意味着治理能力前移：

* 不是脚本跑坏后再问“谁执行的”；
* 而是在系统层面先问“这个对象是否被允许存在”。

### 12.2 类型系统作为治理机制

当 `node_state` 的 Schema 演进时，旧脚本若参数类型不再匹配，就无法构造合法 Function Instance。
这是一种极其重要的治理思想：
**让错误操作无法表达，而不是让错误操作事后被发现。**

### 12.3 不可篡改的审计轨迹

由于每次执行都可表示为：

[
funid(params) \rightarrow result
]

系统天然保留了一条清晰的、可验证的执行轨迹。相比传统日志，这种轨迹具有更强的可比较性与可重放性。它不是文本叙述，而是结构化的因果关系。

### 12.4 治理哲学

本文的治理观可以概括为一句话：
**好的系统不是靠流程要求人不犯错，而是让错误无法成为合法表达。**

这比培训、规范、审计流程更强，因为它将组织治理内嵌为系统结构本身。

---

## 13. 与传统工作流与调度系统的理论差异

### 13.1 相比 DAG 工作流系统

传统 DAG 系统强调静态拓扑与节点推进，本文框架强调表达式求值与命名结果对象。前者更像流程控制系统，后者更像带副作用约束的函数求值系统。

### 13.2 相比命令式调度系统

命令式调度系统接收的是“做什么”的过程描述；本文系统接收的是“求什么值”的函数描述。二者在优化、缓存、治理上的空间截然不同。

### 13.3 相比仅做 Durable Execution 的系统

持久化执行系统擅长通过历史回放恢复内存状态，但仍然强依赖过程执行历史。本文框架则更强调结果对象、状态版本与表达式结构本身的可重建性，因此恢复更依赖对象图而非完整指令回放。

### 13.4 理论上的核心差异

归根到底，差异在于系统把“执行”视为什么：

* 传统系统：时间中的步骤推进；
* 本文系统：空间中的对象求值关系。

前者重在流程连续性，后者重在结果可寻址性与状态可推理性。

## 14. 风险、限制与工程挑战

本文框架并非没有代价，其主要挑战包括：

### 14.1 迁移成本高

从 OPTask 到 Function Instance 不是简单封装，而是一次语义清洗。大量旧任务若依赖隐式状态或脆弱脚本，将难以直接迁移。

### 14.2 对类型与 Schema 纪律要求更高

系统越依赖显式 Schema，越要求接口定义、版本演进、对象命名严格统一。组织若缺乏这类工程纪律，落地会困难。

### 14.3 纯函数边界之外仍需处理现实不确定性

外部系统可能不可靠，收据可能延迟返回，逻辑状态与物理现实可能出现偏差。因此，不可变状态模型是一种强抽象，但仍需依赖正确的边界建模。

### 14.4 编排层压力集中

当依赖管理、重试、人类交互都位于编排层时，编排器本身需要具备高可用、可持久化和无状态接管能力。

### 14.5 并非所有场景都值得采用完整模型

对于简单、短生命周期、一次性流程，该框架可能显得过重。它更适合需要跨系统、跨团队、跨时间维度维护一致性与治理性的复杂自动化系统。

---

## 15. 讨论：这一框架为何适合 Agent-Human-Loop 时代

随着 Agent 系统的发展，任务执行主体越来越多地由软件代理承担。人类不再直接执行每一步，而是：

* 提供目标；
* 审核关键决策；
* 在不确定处介入；
* 在异常时接管。

这使系统天然需要同时支持：

* 自动化执行；
* 人工介入；
* 中间状态可观察；
* 历史结果可复用；
* 错误操作可阻断；
* 多角色协同视图。

传统 TODO 系统、脚本调度器、流程引擎、日志审计系统各自解决一部分问题，却难以共享统一语义。而本文提出的框架之所以具有时代适配性，恰恰是因为它提供了一种共同语言：

* 对 Agent 而言，是函数求值任务；
* 对人类而言，是 Task 视图与等待确认点；
* 对调度器而言，是资源约束下的对象计算；
* 对治理者而言，是类型安全与可追溯执行；
* 对基础设施而言，是命名对象与全局缓存。

---

## 16. 结论

本文提出了一种以 **Function Instance、版本化不可变状态、表达式求值和全局 Memoization** 为基础的分布式系统理论框架。其核心不是在传统调度器上堆叠更多工作流特性，而是通过改变调度原语与系统边界，使原本难以统一的问题获得同一套解释：

* TODO 被还原为 TaskMgr 的视图；
* 工作流被还原为表达式树求值；
* 节点状态被还原为版本化不可变对象；
* 缓存被下沉为 Named Object 基础设施；
* 人类介入被还原为特殊求值条件；
* 审计与治理被内嵌为对象命名、类型匹配与执行收据。

这一框架所追求的，并不是“更强的调度器”，而是**更小、更纯、更可推理的调度核心，以及更清晰的系统分工**。在这种结构中，复杂性没有消失，但被重新安置到了正确的位置：
依赖关系属于编排层，状态汇聚属于 TaskMgr，缓存属于 Named Object，执行分配属于调度器，而治理能力则来自整个系统对合法表达形式的严格约束。

因此，这一设计不应被理解为某种具体产品实现，而应被视为一种系统理论主张：
**分布式自动化系统的下一步，不是让调度器学会更多业务，而是让业务以更受约束、更可命名、更可求值的方式进入调度器。**

## 参考性附录：可直接作为论文结尾的“设计原则总结”

1. **TODO 是视图，不是系统。**
   人类看到的是任务的投影，系统管理的是任务运行实体。

2. **调度器不改。**
   调度器的价值在于快速、确定、可预测，业务语义不应侵入其核心。

3. **原语降级优于能力堆叠。**
   将 OPTask 降级为 Function Instance，带来更好的推理性与治理性。

4. **约束产生信息，信息产生优化。**
   Schema、幂等性、资源声明、终止性约束，为系统提供优化前提。

5. **可变状态必须版本化。**
   不可变状态是缓存、审计、冲突检测与历史重放的共同基础。

6. **缓存应下沉为基础设施。**
   全局 Memoization 应位于 Named Object 层，而非调度器内部。

7. **工作流是用户态编排，不是调度器内核扩展。**
   工作流系统通过标准协议消费调度能力，而非改写调度规则。

8. **DSL 与表达式树等价。**
   对外保留可读 DSL，对内转换为适合机器求值的依赖表达。

9. **副作用必须显式管理。**
   通过幂等声明与 side_effect_receipt 将不纯性交还给系统边界。

10. **治理优先于补救。**
    最强的治理不是追责，而是让错误操作根本无法构造。
