## 一、调度器的核心本质

调度器本质上是一个状态转换引擎。给定系统的当前状态，再给定一个目标状态，调度器的核心工作就是输出一组 action——告诉系统：执行这组动作，就能从当前状态演进到目标状态。

其中，当前状态应该由调度器根据系统实际运行情况在现场收集得到；而目标状态的定义，则是使用者的核心职责。

## 二、如何定义目标状态

### 2.1 目标状态是产品性需求

目标状态的定义本身是一项需要设计的工作。大部分目标状态的定义其实是拓扑无关的。比如最常见的例子：我希望系统里能可靠地运行一个服务，至少有两个副本。或者更简单地说：我希望在系统里可靠地运行一个服务，副本数量由调度器来决定。

这里面有一个关键洞察：目标状态描述得越模糊、越直达本质目标，调度器可操作的空间就越大。目标状态的定义很多时候本质上是产品性需求。

### 2.2 服务的核心指标

在日常实际目标中，我们关心一个服务的几个核心指标：首先是可靠性指标，然后是可用性指标，再往上是性能指标。从提升性能的角度出发，我们希望尽可能减少系统中的冗余操作。

举个例子：系统里有两个服务 A 和 B，A 总是要消费 B。如果 A 和 B 分配在不同节点上，就需要走网络通信；如果能分配到同一个节点上，跨机器通信就变成了本机进程间通信，大幅减少网络损耗。

### 2.3 需求之间的优先级与矛盾

实际中存在很多这样的定义，它们之间有一组优先级关系，而且有些需求是互斥的。对此，调度器有两种思路：

第一种思路是引入"需求编译"的过程。用户提出不太精确的定义时，通过一个编译器将其翻译为调度器可以精确执行的目标状态描述，然后让用户 review。因为一组需求里确实存在本质矛盾的情况，在每个调度 loop 或每次目标状态改变时都去解决这些矛盾并不理想，最好还是有一个用户确认的过程。

### 2.4 AI 驱动的自然语言到目标状态的转换

现在有大语言模型了，这件事变得很方便。用户用自然语言描述一个目标状态，系统根据规则拆解出一组用户也能看得懂的精确目标状态描述——比如根据系统规模推断出的副本数、A 和 B 之间的亲和关系或反亲和关系等。

这个过程至少要保证生成的目标状态是不矛盾的、可以按优先级执行的。站在调度器过滤器算法的角度来讲，至少要把用户用自然语言描述的目标状态变成一组清晰的、有优先级的过滤条件。用户 confirm 之后，再提交给执行层的调度器，这样调度器的分层也会更加清楚。

如何找到 AI 驱动的自然语言到调度器目标状态描述之间的边界，这是一个非常有意思的问题，可能需要通过一些实际的例子来思考。

## 三、无状态服务

分布式系统里，我们实际开发中认为大部分应用开发都是无状态组件。无状态组件意味着它只需要在 manifest 里声明自己依赖哪些服务就好了：我是一个服务，我依赖系统的 RDB，依赖消息队列 MessageQ，依赖事件总线 EventBus，依赖其他应用服务——它只需要描述依赖关系就行，因为自己没有状态。

这也是分布式开发里的一个基本分层逻辑：对于不同水平的开发者，我们让 80% 的工程师只写这种无状态服务。这种服务本质上很多时候甚至可以退化成一个函数——在哪里跑都可以，甚至不需要持续运行的服务进程。它可以进一步退化成类似 QF（Query Function）的模式：一个请求过来，一个函数被拉起来，求值完成后把 response 扔回去就行了。

这种最简单的服务甚至不需要在系统里持续运行。系统某一时刻如果只有几十个不同类型的请求，那系统里也只跑几十个不同类型的服务实例就好了，大部分服务都处于休眠状态。

## 四、有状态服务的挑战

### 4.1 依赖关系是一切困难的根源

无状态好解决，大家都爱研究。但真正难的是有状态服务。有状态服务的痛苦很多时候在于状态之间的关系——我认为一切的困难还是从依赖关系出发的。

### 4.2 循环风暴

对于分布式系统来讲，过去经常碰到一种叫做"循环风暴"的问题。比如写放大风暴：A 访问 B，B 访问 C，系统一开始这样没问题。但系统运行时间久了之后，后来维护 C 的人已经不记得之前的设计了，在 C 里又去访问了 A，就会在分布式系统里产生一些永久存在的回环流量。

由于这种回环不出任何错误，也没有可以观测到的迹象，所以对系统的无声损耗会长期发生，而且非常难定位。运行一段时间后，如果没有及时发现，某一天链路被放大之后，可能导致系统直接崩坏——所有人都困惑"最近没有任何修改，为什么突然上线后系统全部过载了？"这种跨服务、跨机器的潜在循环死循环，是分布式系统的隐形杀手。

### 4.3 依赖关系的强制定义

所以对调度算法来讲，定义依赖关系特别重要。因为在定义依赖关系时，依赖图不可能成环——成环就意味着潜在的循环一定会被检测出来。引入调度系统之后，从边界上就阻止了服务开发者的随意修改：只要他要部署，就必须定义依赖关系，而定义依赖关系就能把成环问题暴露出来。

## 五、有状态服务的设计模式

### 5.1 以 KV 存储为例的基本模式

以 RDB 为例，它是最复杂的组件之一，内部也会做分层。比如现在常见的做法，最底层是一个 Key-Value 存储。KV 服务是有状态的，它的核心就是需要定义存储器。

分配存储器时，第一个原则是存储器绝不能分配到同一个节点上——大家之间的距离要尽可能远。在大型系统里，最好不在同一个机架上，万一机架交换机坏了，能影响到 KV 系统的节点要尽可能少。所以存储桶之间是互相反亲和的，要尽可能使大家的平均距离更大。

第二步是请求选择器：来了一个 KV 查询请求，通过什么算法找到目标 V？从实现的角度，我们一般希望选择器逻辑在客户端，因为除了浏览器改不了，系统里大部分客户端都是自己开发的。客户端逻辑甚至可以做多版本选择——系统的桶分布可能在从版本一演化到版本二的迁移状态中，客户端先用最新版本的桶分布查一遍，查不到再用上一个版本的桶分布查。这建立在 KV 只读性的基础上。

客户端需要通过调度器构造的 system config 里的桶分布拓扑图，dump 到自己的缓存中，查询时查缓存即可。

### 5.2 模式总结：不依赖别的服务的有状态服务

这种模式就是一个不依赖其他服务的有状态服务。它定义了自己存储状态容器的一些特性，调度器根据系统当前所需的容器副本数，分配到具体节点上，然后把分配结果作为一个升级版本写入配置——老的分配结果永远不会删除，新旧版本都可以作为客户端使用的基本数据。

对应用开发者来说，主要工作就是理解这套机制后，去开发状态桶和客户端选择器——让客户端能够连上正确的有状态组件。

## 六、有状态服务之间的依赖问题

### 6.1 依赖 KV 的有状态索引器

当第一个 KV 有状态服务已经上线后，开始开发第二套有状态组件——比如一个分布式索引器，它依赖底层的 KV。系统数据可能非常大，索引器在构建索引时需要扫描 KV 里的原始数据，在 KV 修改时需要刷新索引。

这时就会产生亲和与反亲和的问题。一方面，作为有状态组件，它在某些情况下希望反亲和（保障可靠性，避免崩溃后数据全没了）。另一方面，从索引器本身的性能角度，它又希望能够与特定特征的 Key 在一起。比如很多 Key 是路径格式的，索引器关心某个路径模式下的所有数据，那它应该在哪个节点上运行？

### 6.2 对 KV 分布算法的挑战

如果这是一个组织的核心业务，可能会提出：之前 KV 基于 Key 哈希的副本分布算法不合理——数据散得太开了，一个路径模式下的所有文件被分散到系统各处，索引器放哪儿都一样远。能不能为了核心业务，在 KV 算法里写个特殊逻辑，把某个模式下的所有数据都塞到一个机柜上？这样就不用走二层交换，在同一个机柜里速度更快，也不会导致整个机房的处理风暴。

传统做法在这时候就开始人肉运维了——通过手动调整把业务做完。但如果我们希望有一个长期稳定工作的调度器，那么调度器必须能够理解这种现实中的复杂问题。

## 七、金融与电商系统的特殊复杂性

### 7.1 大文件系统的特点

我过去的背景是做大吞吐、大 IO 文件系统之类的东西。大文件系统的特点是对写延迟要求一般不太高，但数据吞吐量特别大，尤其是读的比例远远超过写。这个背景让我对系统的数据分布比较敏感，特别愿意做多副本，读的副本会特别多。

### 7.2 电商与支付系统的挑战

另一个很大的分支是电子商务和支付系统。它们对可靠性的需求非常高——支付流程的每一步都涉及到钱，很多还涉及到与外部系统（如银行接口）的交互。这时需要做全程有状态的分布式事务。

过去的思路是直接用数据库来做，但业务量大了之后（特别是电商），就需要一个非数据库的分布式事务系统，能够跟踪一个订单从头到尾的全流程。

### 7.3 分布式事务的核心难点

难点从"有没有货"开始：用户下单成功后，SKU 库存就被锁定了；后面还有支付环节，支付本身也有异常。用户需要不断看到反馈，确认交付承诺是正确的、有保障的。最惨的情况是用户付完钱后被告知东西卖完了——这是不可接受的。

这里面有很多矛盾的需求：有时候需要数据可靠（多副本），有时候又特别害怕读到副本数据（需要实时数据）。这对系统的状态分布提出了另一个不同视角的挑战。

### 7.4 分布式锁与事务的实践

通常场景的抽象是分布式锁：通过锁 ID 可以获取当前状态，有一个业务引擎持有锁并管理所有 step，业务引擎超时后释放所有锁并执行回滚操作。由于是分布式回滚，还存在回滚时系统宕机等问题。

实践中一般不做太复杂，最多做三层：第一层是事务——在事务视图里看到的状态都是确定的，资源已经被 lock，库存已经被减掉；第二层是提交——如果提交失败可以做二次提交；第三层是回滚——超时后回滚，通常处理一层的回滚失败（rollback fallback）。这已经够复杂了。rollback fallback 里还可能再失败，但通常不再处理——这种超低概率事件只能认倒霉。

这是另一个方向的复杂性，也是调度器在设计时需要考虑的现实约束。
