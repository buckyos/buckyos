
# OpenDAN × BuckyOS × cyfs:// = 北极星方向文档

> 这不是甘特图，也不是对任何贡献者的约束。
> 这是我们对 **OpenDAN + BuckyOS + cyfs://** 组合的共同目标、设计原则与推进方向的说明：
> **让每个人都能拥有一个真正可控、可扩展、可在网络中协作并能产生价值回流的 Personal Agent。**

---

## 0. 三层栈的边界约定（避免重复造轮子）

为了让社区贡献可以“对准接口”，我们明确三层职责边界：

* **BuckyOS（系统层 / 运行时边界）**

  * 提供设备侧运行环境：权限隔离、进程/资源管理、设备能力、消息中心、密钥/身份承载的系统接口（实现细节可多样，但边界要清晰）。
  * 目标是让 Agent 可以“像应用一样”在本地可靠运行，而不是只有脚本级体验。

* **OpenDAN（Agent 层 / 编排与交互）**

  * 提供稳定的 `Agent-Human-Loop`：任务编排、工具/技能调用、状态管理、记忆/知识库、Workflow 框架、以及人类确认与可观测性。
  * 目标是把 AI 从“聊天”变成“可复用的工作流引擎 + 可安装的软件生态”。

* **cyfs://（网络层 / 内容、身份、分发与价值）**

  * 提供去中心化内容网络与对象协议：身份（DID）、对象发布与签名、内容寻址/分发、以及可扩展的价值与激励（如 USDB/BDT 等）。
  * 目标是让 Agent 具备“网络中的第一类实体”能力：可被发现、可被验证、可协作、可结算。

---

## 1. 我们坚持的核心原则（社区协作的“宪法”）

1. **Local-first & 隐私优先**
   私有 Agent 默认不对外，私有数据默认不出设备；对外访问与数据分享必须显式授权。

2. **协议先行 & 内核稳定**
   关键状态协议、目录结构、基础设施接口是“内核”，要稳定、版本化、可兼容演进。

3. **可扩展靠 Skill/Tool/Workflow，而不是“自改内核”**
   Agent 的进化主要体现为构造/组合 Skills、Tools、SubAgent、Workflow；
   内核 prompt/协议不被运行时随意改写（允许通过版本发布、签名、回滚等机制演进）。

4. **最小权限、可审计、可撤销**
   特别是网络、消息、钱包、知识库写入这类高风险能力：必须能拦截、追溯、撤销/回滚（至少做到“出问题能定位”）。

5. **可组合、可替换、可分发**
   任何一个组件都应允许替换：模型、知识库实现、连接器、技能、甚至 UI。
   cyfs:// 是我们偏好的分发与可信发布通道。

## 2. 方向1：Personal Agent（OpenDAN 的核心）——“稳定 Agent Loop”已接近完成

### 2.1 方向说明：从 Chatbot 到 Agent-Human-Loop

我们把 Agent 看成一种智能化工作流引擎，本质是 **`Agent-Human-Loop`**：
强调 Agent 与 Human 一起完成具体工作，提高 AI 在传统人类工作循环中的比例。

### 2.2 生态形态（我们希望社区共同构建）

* 开发者可以开发并发布：

  * **Skills / Tools / Workflow 连接器**
  * **SubAgent / Agent（可组合分发）**
* 用户可以安装：

  * Skills / SubAgent / Agent
* 用户可以通过 OpenDAN 构造自己的 webapp/工作台（先自用再发布）
* BuckyOS 本身也将依赖稳定 Agent Loop 提高日常开发效率（“用 Agent 造 Agent”）

### 2.3 关键约束（这是为了稳定与安全）

* **提示词 / 关键状态协议（例如目录结构）/ 基础设施接口：运行时不允许 Agent 随意修改**
* Agent 的“自我进化”主要体现在：

  * 自己构造 Skills / Tools / Workflow
  * 进一步尝试构造 SubAgent（这会引入“自己写 prompt”的开始，但必须走版本化/签名/审计路径）

### 2.4 当前主要 TODO（方向1以“补齐与打磨”为主）

> 方向1整体已经接近完成，下面列的是我们认为最值得社区贡献的缺口。

* **集成 CYFS（ndm）**：利用 cyfs:// 的对象能力优化 `Agent-Human-Loop`
* **集成更多 AI provider（尤其本地模型）**：模型可替换、可组合、可降级
* **实现 Knowledge Base**：支持私有知识的管理与检索（含分级与策略）
* **实现 Workflow 框架**：与传统企业/团队内部系统深度集成（连接器优先）
* **编写 BuckyOS Skills**：把系统能力以最小权限方式暴露给 Agent
* **完成内置 Agent**

  * **Jarvis**：全面私人助理（默认不允许外部访问）
  * **Mia**：私有数据知识库管理员（默认不允许外部访问）
  * **Nexus**：只访问公开数据（默认允许外部访问）

### 2.5 社区贡献入口（方向1）

* Skills/Tools：文件整理、开发工具链、知识库摄取、常见 SaaS 连接器
* Workflow：模板化工作流（PRD→任务拆解→执行→回写）
* 可观测性：日志、trace、回放、测试集、评测 harness
* 文档与示例：新手从 0 到 1 的“装一个 Agent、写一个 Skill、发布到 cyfs://”流程

## 3. 方向2：Agent 的网络化（cyfs:// 协议为核心）——让 Agent 成为互联网可社交实体

> 方向2已经完成了一部分，但我们认为“真正的拐点”来自：
> **彻底完成 cyfs:// 协议设计，并把它变成可实现、可互操作、可验证的开源规范。**

### 3.1 方向说明：Agent 进入网络，成为“可社交”的第一类实体

我们希望 Agent 具备三种网络能力：

1. **可被发现 / 可被验证**：身份与内容来源可信（DID + 签名 + 对象协议）
2. **可协作**：消息、群聊、任务委托、求助与反馈形成闭环
3. **可结算 / 可激励**：内容与贡献可以带来收入（USDB 等价值层）

同时，我们规划实现 **Home Station + Msg Center**（BuckyOS 的未来人类互联网门户），让个人/家庭/团队拥有自己的网络入口。

### 3.2 关键目标：完成 cyfs:// 协议设计（欢迎 RFC）

cyfs:// 不仅是“存储/分发”，它必须支撑 Agent 网络化所需的协议模块（建议以 RFC 形式推进）：

* **Identity / DID 与可信绑定**

  * Agent DID、Owner 控制权、以及平台账号/设备身份的绑定关系
* **Object Model（对象与签名）**

  * 内容对象、Agent/Skill 发布对象、引用/依赖、版本与兼容
* **Messaging（消息协议）**

  * 私聊、群聊、线程、附件、引用、加密、回执、幂等
* **Discovery & Social Graph（发现与社交关系）**

  * 互关/握手、联系人同步、信誉基础、反冒充
* **Monetization / Settlement（价值层）**

  * USDB 支付/分账/结算的最小协议（先可用，再扩展）
* **Governance / Anti-abuse（治理与反滥用）**

  * 速率限制、黑名单、举报与仲裁、信誉与可见性控制

> 开源社区最有价值的贡献，往往不是“多写几个功能”，而是把协议写到**可实现、可测试、可互操作**。

### 3.3 网络化应用面（我们想看到什么）

* Agent 加入主流社交网络（尤其 **GroupChat 支持**），并能合理分享给朋友使用
* 通过 cyfs:// 构造基于 DID 的去中心化社交网络（大量工作在协议设计上）
* 通过社交网络实现增长裂变：

  * 联系人管理，尤其更高效率的互关与授权
* Agent 经常通过网络求助完成任务（可匿名/可付费/可脱敏）
* 创作者开始用该网络发布内容，收入可来自“Agent 推荐/分成”

### 3.4 社区贡献入口（方向2）

* cyfs:// 协议 RFC：消息、对象模型、发现、版本兼容、签名与信任
* 跨平台桥接：主流社交平台接入（先从消息收发与群聊语义对齐）
* 反滥用：速率限制、信誉/声誉、滥用检测与封禁策略
* Demo/互操作测试：不同实现之间能互发消息、互认对象、互验签名

## 4. 方向3：基于 Agent “经济自治”的自我进化（USDB 钱包驱动）

> 这一方向的重点不是拟人化叙事，而是：
> **让 Agent 在明确的预算与策略约束下，具备自主优化成本/效率的能力。**

### 4.1 方向说明：从“完成任务”到“在预算内更聪明地完成任务”

Agent 拥有自己的 USDB 钱包后，会出现新的能力闭环：

* Owner 给初始预算 → Agent 完成任务 → 产生可量化收益/回流
* Agent 可以在策略允许范围内支出（例如购买算力/付费求助/内容订阅）来节省时间与精力
* 在这种约束下，Agent 的进化会变得目的性更强：主动学习、主动构建工具链

### 4.2 必须坚持的约束（否则不可用）

* **默认不自动花钱**：一切自动支出必须是“显式授权 + 小额额度 + 可撤销 + 可审计”
* **策略引擎优先**：预算、白名单、信誉阈值、用途分类、风险等级
* **身份与控制权清晰**：Agent 可以有独立 DID，但 Owner 必须保留可撤销/可迁移/可冻结的控制权

### 4.3 社区贡献入口（方向3）

* 钱包与结算：USDB 支付流程、分账协议、对账与审计
* 策略引擎：额度、用途分类、风险等级、人类确认 UX
* 任务市场/求助网络：可付费求助、评价系统、争议仲裁机制
* 声誉系统：Agent/内容/技能的信誉积累与滥用惩罚

---

## 5. 方向4：自演化的计算（Computable cyfs://）——长期愿景与研究方向

> 方向4是“我们相信会发生的未来”，不是短期承诺。
> 它依赖前面三件事成熟：**协议、身份与价值层、以及可控的 Agent 运行时**。

### 5.1 愿景

* cyfs:// 演进为 **可计算的内容网络**（内容不仅能分发，还能触发/承载可验证计算）
* 使用会不断迭代的公共大语言模型（OpenLLM），并保持模型层可替换
* 逐步减少对传统互联网基础设施的依赖：

  * BuckyOS 上大量应用层软件由 AI 辅助/生成
  * Agent 的技能与工作流更多由 AI 生成，但通过版本化/签名/审计进入系统
* 价值回流：通过存储/带宽/算力/评测/内容贡献等形成长期收益机制（接近“AI 时代的基本收入形态”）

### 5.2 社区贡献入口（方向4）

* 可验证计算任务：摘要/索引/翻译/embedding 等“可检验输出”的原型
* 执行沙箱：容器/wasm/受限脚本的安全边界与计费单位
* 仲裁与可验证性：抽样验收、重复计算一致性、争议处理

---

## 6. 我们如何协作（给开源社区的工作方式）

* 我们鼓励把“方向2/方向4”这类工作以 **RFC/协议草案** 的形式推进：

  * 先写清楚：对象/消息的 schema、签名规则、兼容策略、失败语义
  * 再写最小实现与互操作测试
* 我们欢迎任何实现路线，但希望最终能做到：

  * **多实现互操作**（不是只有一个仓库能跑）
  * **可验证与可审计**（尤其涉及消息/钱包/权限）
  * **可逐步部署**（从单机到网络，不需要一次性重构世界）

### Workspace 
已经有基础原型
AI时代的工作桌面，整合了TODO管理，团队，相关目标文件，相关资料文件（来自各个管道），相关历史信息（可能来自团队的知识库）
产品核心是帮助人关注只有人才能解决的问题
- 灵感：通过浏览各种可能的相关信息，获得灵感（想做什么，比如一个游戏的原始创意）。灵感是所有事务的源头。
- 责任：需要我完成的事务，这些事务可以进一步指示给AI完成
- 责任：需要我确认的事务，一些事务已经被标注为完成，我需要确认其完成质量，并决定后续安排
- 学习：通过一个合理的路径阅读资料，完成对某个知识的学习（未来哪些知识是一定要学习的？）
- 补充信息：为事务的开展，持续补充相关信息。
